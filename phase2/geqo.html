那个叫 Planner 的家伙在想什么？

遗传查询算法优化器 (GEQO, Genetic Query Optimizer)

在前一章中，我们详细解析了 PostgreSQL 的骄傲——自底向上的动态规划（Dynamic Programming, DP）机制。DP 能在极度复杂的关系里寻找出理论上的绝对最优执行序列。

但天下没有免费的午餐，即便是应用了 join_collapse_limit 等启发式参数进行搜索边界裁剪，DP 寻找最优路径的算法复杂度依旧是随基表数量呈 $O(3^N)$ 乃至更糟的指数级膨胀的。 当你的一个极其庞大的数仓或分析型报表视图涉及到数十个表（默认参数 geqo_threshold = 12 甚至更多）互相 Join 时，如果坚持让数据库去穷举动态规划排列组合，优化器规划一条 SQL 的时间可能会比直接全表扫一遍执行出来的总时间还长成百上千倍。

这就是 GEQO (Genetic Query Optimizer) 挺身而出的时刻。

PostgreSQL 在 src/backend/optimizer/geqo/ 目录下植入了一个完备的基因进化算法器。它的目的并不是给你最完美的解答，而是在你可以接受的几十毫秒时间内，抛弃遍历思想，直接给你一个**“虽然可能不是全局最顶尖，但绝对足够好”的近似优解**。

1. 概念隐喻：SQL 优化如何转变为旅行商问题 (TSP)

生物演化与遗传算法的核心在于将现实问题编码成“染色体结构”。

在 GEQO 视角下，寻找多个表的连接排序，实质上相当于组合优化领域最著名的 NP-Hard 难题——旅行商问题 (Traveling Salesman Problem, TSP)。

  • 城市 (City) = 基础表（Base Relation）。
  • 路线 (Tour) = 表之间拼合的连接顺序（Join Order），比如这是一条路径：$A \to C \to B \to D$。
  • 路费 (Path Cost) = 当前路线上跑出来的连接计划的总代价值（Total Cost），也就是执行器要付出多少资源来跑这条排好序的执行计划。

由于 GEQO 的本意是简化复杂度，目前 PG 源码里限制了 GEQO 只在 Left-deep 树空间内进行染色体编码搜索（这也是 TSP 问题的一个自然映射方式——从一个点出发一条路走到底），放弃了复杂的 Bushy 多路聚合基因组建模。

2. GEQO 的整体运转循环

遗传算法是一个基于达尔文“优胜劣汰、适者生存”理论的高度迭代模型。它的全局调用入口在 geqo_main() 这个函数里。

整个 GEQO 宇宙的创世与演进包含了以下五个神圣阶段：

2.1 种群初始化 (Initialization)

上帝说要有光，程序里调用了 random_init_pool()。 优化器会从所有可能的表排列组合中，直接按白噪音一般的随机排列，生成初始的若干个路线配置组合（由于基表数量决定了基因变异的跨度，此时的初始种群数量 Pool Size 会通过内置公式基于 N 自适应地扩张。N 越大，池子越大，默认最多会产生数百个个体）。

2.2 适应度评估 (Fitness Evaluation)

这是一个“照妖镜”环节，入口在 geqo_eval()。 对于 Pool 里的每一个个体（排好序的一串表名集合，例如 $A, B, C, D$），GEQO 不会自己凭空算盘，而是非常讨巧地去白嫖传统优化器的代价计算引擎！ 它会按这个顺序去强行构建一个完整的 Join Tree，交给 cost_qual_eval 等底座函数计算一遍，拿到诸如需要 3450.25 的硬性数字指标，然后把它映射为个体在这个种群中生存下去的适应度 (Fitness)。Fitness 越低越好（因为 Cost 越低越快）。

然后，所有染色体由于它们不同的“强壮度”会被打分，并调用 gimme_pool_size() 在数组里面按优劣直接排好队（Ranking）。最上面的一定是最强壮的，最底下的随时要被淘汰。

2.3 选择 (Selection) 权重的马太效应

遗传的法则是：越是强者，留下子嗣的概率越高。 当系统进入繁殖轮次时，会随机挑选两个个体作为父母 (Parents)。如何保证强者恒强但又保留穷人的翻盘几率？ 在源文件 geqo_selection.c 中，PG 采用了一种叫做 基于排名的线性选择法 (Linear Ranking Selection)。排名越靠前（Fitness 好的代价值低的个体），抽中作为父母的概率基数越高，但底下的哪怕是最次的依然有一丝微弱概率被点兵中选。

2.4 交叉/受孕 (Crossover / Breeding)

最核能的部分来了，由于 TSP 序列必须是表名的无重复混洗（不能出现 $A,B,A,C$ 这种畸形路线），PG 在几十年前选用了一种学术界公认最为精密巧妙的乱交算法：边缘重组交叉算子 (Edge Recombination Crossover, ERX)。 调用在 geqo_erx.c 中的 gimme_edge_table() 及其核心调度里。

这个奇妙算法能极大地保留父母代“连接对”的基因。例如，爸爸有 A->C，妈妈也有 C->A，既然 C 一直缠着 A 不放，那说明虽然你们俩路径别的不管，但 A 相邻着连接 C 必定是个极其有缘的优良基因！在产下全新子代（Child）时，子代码会主动向周边有交集的游离基因靠拢并生成出 $A \to C$。这是打破常规遗传算法断点交叉的最强防线，非常贴合 Joint 树“关联近邻更好”的特征。

2.5 变异 (Mutation) 为了防止内卷

如果在好几百个代的推演里，大家只生娃，整个群体的基因很容易陷入局部最优解 (Local Minima)（大家都变成了一种相近的排列，怎么交叉还是生出那个德性的路线），从而彻底停止了对更优路径的探索。 PG 设计了 geqo_mutation() 函数，给每一个新产下的个体植入一点点叛逆的随机数。它会以微小的概念发生染色体调换，比如让里面的第 3 个表和第 5 个表突然对调一下顺序。这就强行引入了野生的新特性，使得搜索图能够跳出局部坑洼。

3. 收敛与局限性：双刃剑的利与弊

经过由配置参数 geqo_effort（从 1 到 10 级）控制的上百代种群疯狂繁衍以后，系统强行叫停。整个上帝视角的演化场里，站在 Pool 存活最顶端的排在第 0 顺位的那个最强个体，就会被强行抽取出来。 PG 就拿着这个通过概率、玄学以及漫长时间线压缩计算得到的最后序列，拍板定夺：“我们下一跳就按照这个 Join 计划去运行吧！”

副作用与建议：每次看 Explain 计划都在跳舞？

如果你真的在超过 12 张表上尝试复杂的 BI 报表级 JOIN，你会面临一个非常神奇且让你 DB 图谱工程师血压升高的现象：

既然遗传算法本身是带有伪随机特性起手的，并且随机算子始终交织在繁衍树中，那么同一个复杂的查询，在第一次跑出和第二次跑出的时候，哪怕整个世界的数据都不曾变动，它最终选出来的解释计划（Explain Plan）竟然很有可能是大相径庭的！ 这也就意味着查询响应时间会出现偶尔长偶尔短的神奇心跳图。

为了解决调试麻烦，如果你确实希望在复杂图景里获得极其稳定的执行规划树：

  1. 可以考虑去修改当前会话里的 geqo_seed（遗传随机种子），把它定死为一个小数，使得同一个版本内的搜索具备唯一重演性。
  2. 如果不差几百毫秒准备时间或者用的是极高性能规划服务器，甚至可以通过设置 SET geqo = off 强行禁用掉遗传算法引擎，或者极度调大 geqo_threshold 强逼底层执行标准的 DP 爆破扫描。

在这个庞大犹如钟表机械般细致刻度的规划器深渊里，无论动态规划如同地毯式轰炸般的精准，抑或是在极端条件下遗传算法的退缩防御妥协，都展示出了 PostgreSQL 对数学之美在应用工程中调和最强悍的一面。
