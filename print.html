优化器概览

本章基于 PostgreSQL 源码中的 src/backend/optimizer/README，简要介绍查询优化器（Query Optimizer）的整体工作流程和核心函数调用链。

优化器的主要任务是接收查询树（Query Tree），并生成最优的执行计划（Plan）。

核心调用流程

PostgreSQL 优化器的入口函数位于 src/backend/optimizer/plan/planner.c。

整个规划过程大致可以分为以下几个层级：

graph TD
    A[planner] --> B[standard_planner]
    B --> C[subquery_planner]
    C --> D[grouping_planner]
    D --> E[query_planner]
    E --> F[make_one_rel]
    F --> G[make_rel_from_joinlist]
    D --> H[create_plan]

1. planner()

这是优化器的全局入口点。它的主要职责非常简单：

  • 如果配置了钩子函数（planner_hook），则调用钩子。
  • 否则，调用默认的优化器实现 standard_planner()。

2. standard_planner()

这是标准的优化策略入口。它的工作包括：

  • 配置设置：设置由于 CURSOR 或其他上下文导致的相关标志。
  • 预处理：调用 subquery_planner 来处理顶层查询。
  • 计划清理：生成的计划树可能包含一些不需要的节点，在这里进行最终清理。

3. subquery_planner()

这个函数负责处理一个特定的查询层级（Query Level）。对于包含子查询的 SQL，这个函数会被递归调用。

  • 重写与预处理：处理 ANY/ALL 子链接，提升子查询（Pull-up sublinks/subqueries）。
  • 调用下层：一旦查询树被扁平化或重写，它会调用 grouping_planner 来进行实际的规划。

4. grouping_planner()

这个函数是“全能选手”，负责处理：

  • 非 Join 部分：处理 GROUP BY、HAVING、WINDOW functions、DISTINCT 和 ORDER BY。
  • 它首先调用 query_planner 来计算最基本的“扫描/连接”（Scan/Join）路径。
  • 然后，它在 query_planner 返回的路径基础上，添加聚合（Aggregation）、排序（Sorting）和分组节点。

5. query_planner()

这是核心中的核心，专注于 Scan/Join 优化。

  • 基本关系处理：为查询中的每个表（Base Relation）生成扫描路径（Seq Scan, Index Scan 等）。
  • 连接搜索：调用 make_one_rel() 来搜索最优的连接顺序。它使用动态规划（Dynamic Programming）或遗传算法（GEQO）来组合这些表，直到所有表都被连接成一个最终的关系。

主要阶段总结

根据 standard_planner 的执行逻辑，我们可以将优化过程概括为以下步骤：

  1. 预处理 (Preprocessing):

    • 重写子查询，简化表达式（eval_const_expressions）。
    • 消除冗余的外连接（Outer Join Elimination）。
    • 分发约束条件（Qualifiers）。

  2. 扫描/连接规划 (Scan/Join Planning):

    • 为基表生成访问路径。
    • 计算两表连接、三表连接…直到生成包含所有表的连接路径。
    • 这部分主要由 query_planner 和 make_one_rel 完成。

  3. 后置处理 (Post-processing):

    • 在连接结果之上，添加 Grouping、Aggregation、Windowing 和 Sorting 节点。
    • 将最优的 Path 树转换为最终的 Plan 树（create_plan）。

查询树的预处理

在 PostgreSQL 优化器（Planner）接收到解析和重写后的查询树（Query Tree）以后，第一步并非直接开始路径搜索，而是通过 subquery_planner() 函数进行一系列复杂的预处理。预处理的核心目标是消除查询中的冗余结构、展开嵌套，并对表达式进行规范化和简化，从而为后续的 grouping_planner() 和 query_planner() 减轻负担，提供尽可能扁平化和易于优化的结构。

以下内容基于 PostgreSQL 18.1 源码（src/backend/optimizer/plan/planner.c 中的 subquery_planner）详细解析查询树预处理的各个核心阶段。

1. CTE 与底层结构展开

在真正进入表达式级别的预处理之前，优化器必须先把从属的查询块或语法结构给“铺平”。

  • 处理 WITH 语句 (CTE): 调用 SS_process_ctes。如果 CTE 可以被内联（Inline），则将其展开为 RTE_SUBQUERY；否则构建出 InitPlan 级别的 SubPlan，以便在主查询执行前计算完毕。
  • 补全空 FROM 子句: 通过 replace_empty_jointree。如果查询没有 FROM（如 SELECT 1），优化器会人为挂载一个 Dummy 的 RTE_RESULT 节点，统一后续的 JoinTree 处理逻辑。
  • SubLink 的显式提升 (ANY/EXISTS): 调用 pull_up_sublinks。将 WHERE 或者 JOIN/ON 里面涉及 ANY / EXISTS 的相关子查询（SubLinks）尽早转化为半连接（Semi-Join）或反连接（Anti-Join）。
  • 子查询扁平化 (Subquery Pull-up): 调用 pull_up_subqueries。如果 FROM 子句中直接写了普通的简单子查询，系统会尝试将其合并到当前查询层级，消除不必要的嵌套。
  • UNION ALL 的扁平化: 遇到简单的 UNION ALL，调用 flatten_simple_union_all 将其重写为并列的追加关系 appendrel，避免复杂的集合操作开销。

2. RTE (RangeTableEntry) 预处理

RangeTable 维护了查询涉及的所有表、视图、函数。

  • 函数 RTE 内联与常量化: preprocess_function_rtes 针对 FROM 子句中的函数进行计算。如果是简单的常数化函数，直接得出结果；如果有机会，则像展开视图一样将函数展开（Inline）。
  • 虚拟生成列展开: expand_virtual_generated_columns 会扫描所有的表，将查询内涉及到 Generated Columns 的 Var 引用直接替换为其背后的计算表达式，方便一并在后续步骤被折叠。

3. 表达式预处理阶段 (preprocess_expression)

这是预处理代码块中占篇幅最大、也最为细腻的一环。系统不仅要处理 WHERE 里的条件，还要将 TargetList、HAVING、LIMIT 及 ON CONFLICT 等各处的表达式利用 preprocess_expression() 统统洗一遍。核心动作包括：

3.1 连接别名展开 (Join Alias Variables Flattening)

通过 flatten_join_alias_vars，优化器将由于 JOIN ... USING 或自然连接引发的连接别名变量，替换为来自底层基表的原始列（Var 引用）。这确保后续的优化只需要面对最底层的真实列名。

3.2 常量折叠 (Constant Folding)

调用核心函数 eval_const_expressions。 不仅会将 1 + 1 化简为 2，还能对各种内部函数的常量输入进行预计算（前提是函数为 Immutable）。重要的是，它会压平嵌套的 AND/OR（将 AND (A, AND(B, C)) 变成 AND(A, B, C)）以保持逻辑树平坦。

3.3 条件标准化 (Canonicalize Qual)

利用 canonicalize_qual 函数，如果表达式是一个布尔条件（如 WHERE 里的项），它会尝试做一定的范式转化（比如尽力将其拉平或者排查明显的互斥矛盾）。

3.4 剩余 SubLink 转换与外层变量绑定

  • 经过 pull_up_sublinks 还没解决的杂项子链接（如出现在 SELECT 目标列里的标量子查询），在此被包装成 SubPlan。
  • SS_replace_correlation_vars 会找出对上层查询列的引用，并使用 Param（参数）进行绑定，使得父子查询产生实质的数据关联参数。
  • 最后对于 WHERE / HAVING 的顶层条件，调用 make_ands_implicit 退化为 Implicit-AND 的 List 结构存放，不再采用显式的 BoolExpr。

4. 后期合并与结构修正

经过表达式洗刷后，subquery_planner 在交付给 grouping_planner 之前会做最后的清理。

  • HAVING 下推为 WHERE: 这是经典的优化手段。系统会扫描 HAVING 子句，寻找那些不包含聚合函数、不包含易变函数（Volatile Function）并且不需要等到空分组集产生结果的过滤条件（例如 HAVING col = 1）。这些被认为是退化的聚合后过滤，系统会将其克隆或移动到 WHERE 列表中，从而在进入 Aggregate 算子之前尽早淘汰无关数据。
  • 外连接消除 (Outer Join Elimination): 调用 reduce_outer_joins。因为上一步各种 WHERE 表达式已经被处理，这一步系统会检查如果存在 WHERE 强约束要求某空侧列非空（比如 LEFT JOIN B ON ... WHERE B.id > 10），此时原本的 LEFT JOIN 已经失去保留 null 侧的意义，该外连接会被安全地降级成 INNER JOIN。
  • 剔除无用的 Result RTE: 调用 remove_useless_result_rtes。前面补充的或中间产生的不再起实质作用的 Dummy 关联在此被扫地出门。

至此，一个极其干净、扁平化且逻辑清晰的等价 Query Tree，才会正式交给 grouping_planner。

子查询提升 (Subquery Pull-up)

子查询提升（Subquery Pull-up）是 PostgreSQL 优化器中最为核心的逻辑重写优化机制之一。它发生在规划器的预处理早期阶段。其根本目标是**“打破查询边界，扁平化查询树”**，使得父子查询能够融合在一起，从而大幅度扩展 Join Order（连接顺序）的搜索空间，并且减少执行层的嵌套开销。

1. 背景与目标：为何要提升？

如果不进行提升，嵌套在 FROM 或者是 WHERE 子句中的子查询，在执行树上就会变成单独的 SubqueryScan 节点或是 SubPlan 操作（InitPlan/SubPlan）。 这限制了优化器的能力：

  • 黑盒隔离：父查询的表无法与子查询内部的表混合进行 Join Order 规划。
  • 运行时开销：执行引擎在运算时需要频繁穿越执行计划的不同层级进行上下文切换或独立求解。

因此，PostgreSQL 规划器在 src/backend/optimizer/prep/prepjointree.c 中，专门设计了两大核心路径来拆解各类“提升”。

2. SubLink 提升：pull_up_sublinks

SubLink 通常指的是出现在 WHERE 或者 JOIN/ON 过滤条件中的子查询结构。 优化器在 subquery_planner 一开始就会调用 pull_up_sublinks()。

其核心递归函数 pull_up_sublinks_qual_recurse 会遍历条件，并根据 SubLink 的类型采取不同的提升策略。这依赖于几个关键转换函数：

2.1 ANY SubLink (convert_ANY_sublink_to_join)

当遇到类似 WHERE x.id = ANY (SELECT id FROM y)（即 IN (subquery)）时：

  • 优化器会将其从 WHERE 条件中剥离出来，转化成与父查询的一张表的 JOIN_SEMI (半连接)。
  • 对于常数数组 IN (1, 2, 3)，也会在此阶段通过 convert_VALUES_to_ANY 做尝试折叠化简。

2.2 EXISTS SubLink (convert_EXISTS_sublink_to_join)

当遇到 WHERE EXISTS (SELECT 1 FROM y WHERE x.id = y.id) 时：

  • 本质与 ANY 类似，这会被转化为一个 JOIN_SEMI (半连接)。
  • 但是它进一步处理了带有 NOT 修饰的情况：如果是 WHERE NOT EXISTS ( ... )，在 PG 中会被直接转化为 JOIN_ANTI (反连接)。这就允许优化器使用 Hash Anti-Join 等高效物理算子直接过滤数据。

3. Subquery 提升：pull_up_subqueries

区别于 SubLink，一般的 Subquery 主要指代的是出现在 FROM 子句里面的表源定义（如 SELECT * FROM a JOIN (SELECT * FROM b) c ON ...）。 当处理完 SubLink 之后，planner 接着调用 pull_up_subqueries()，尝试将 FROM 下拉取出来的子查询给融会贯通。

3.1 简单子查询 (is_simple_subquery / pull_up_simple_subquery)

怎样才叫一个“简单（Simple）”的子查询？

  • 它不能包含 GROUP BY、HAVING。
  • 不能包含 LIMIT / OFFSET。
  • 不能包含集合操作（UNION、INTERSECT、EXCEPT）。
  • 不能包含顶层聚合函数或是窗口函数。

如果是简单子查询，pull_up_simple_subquery() 将大显身手：

  1. 它将子查询的 FromExpr 的儿子全部并入到当前层级的 FromExpr 树里。
  2. 剥除该子查询外层包装。
  3. 利用 pullup_replace_vars 构建或替换那些需要保持上下文独立性的变量引用（如果在 Grouping Set 里，有时还要为其包装一层 PlaceHolderVar）。

3.2 简单 UNION ALL 的扁平化 (pull_up_simple_union_all)

对于 FROM (SELECT x FROM a UNION ALL SELECT x FROM b) 这样的集合操作： 如果没有任何排序 ORDER BY、没有 LIMIT，并且是 UNION ALL （不用去重），优化器会调用 is_simple_union_all。 它会把这颗 UNION ALL 子树彻底压平，转化为一系列的追加关系 (AppendRelInfo)。这就将集合操作融化成底层扫描层级的简单 Append，而不用执行繁重的 Subquery 拼接。

3.3 LATERAL 子查询的修补

在 PostgreSQL 支持了 LATERAL 之后，提取时还涉及 LATERAL 跨层依赖参数。 如果发生 pull_up_subquery 且原本有 LATERAL 引用，代码会通过 IncrementVarSublevelsUp 等复杂的方法调整 Vars 的层级深度（Sublevel），并借由 CombineRangeTables 重组合并底层的 RangeTable 中记录被依赖的状态。这一机制对关联子查询能够被最终压平至关重要。

外连接消除与无用连接移除 (Outer Join Elimination & Join Removal)

在经过了表达式级别的折叠与扁平化之后，PostgreSQL 优化器会尝试通过约束分析，尽可能精简查询树中的连接节点（Join Nodes）。精简的连接树不但执行效率高，同时也给了后续的动态规划/遗传算法更多排列组合的选择。

这一步主要包含了两个阶段的优化：

  1. 外连接降级/消除 (reduce_outer_joins)
  2. 无用连接移除 (remove_useless_joins)

1. 外连接降级的原理与意义 (reduce_outer_joins)

源码位置：src/backend/optimizer/prep/prepjointree.c 调用时机：它是由 subquery_planner 函数中调用的。

外连接（Outer Join）通常会强制优化器遵循特定的连接顺序，因为 Nullable 一侧必须在关联后才能保留。reduce_outer_joins 的目的就是检查是否存在某些强有力的 WHERE 条件，使得原本的 Outer Join 可以退化为 Inner Join 甚至是 Anti Join。

1.1 两遍扫描机制 (Pass 1 & Pass 2)

为了减少不必要的递归，reduce_outer_joins 使用两阶段处理：

  • Pass 1 (reduce_outer_joins_pass1): 收集查询树底层的 Relids（涉及了哪些基础表），并快速标记哪些子树中真正包含外连接（contains_outer）。如果没有外连接，直接跳过整个复杂的分析。
  • Pass 2 (reduce_outer_joins_pass2): 进入实际的检查和转换逻辑。

1.2 Outer 到 Inner 的降级

其核心依靠提取过滤条件中的严格约束 (Strict Quals)。 利用 find_nonnullable_rels 扫描 WHERE 子句。如果 WHERE 强制要求 Nullable Side 的某个引用列不能为空（比如 LEFT JOIN B ON ... WHERE B.id > 10，当 B 补充 NULL 时，NULL > 10 为 NULL/False 被淘汰），那么：

  • LEFT JOIN 可以安全地转换为 INNER JOIN。
  • RIGHT JOIN 可以安全地转换为 LEFT JOIN 然后内部翻转，或者直接转为 INNER JOIN。
  • FULL JOIN 可以根据两边约束的情况降级为 LEFT、RIGHT 或是 INNER JOIN。

1.3 Outer 到 Anti 的转化

外连接不仅可以被降阶为内连接，有时还可以变成反连接。 利用 find_forced_null_vars 检查 WHERE 是否有显式要求变元为 NULL 的约束（例如 LEFT JOIN B ON ... WHERE B.id IS NULL）。 如果是这样，那么这个 Join 明确就是用来找“匹配不上的行”，代码中会将该 JOIN_LEFT 转换为 JOIN_ANTI。Anti Join 在执行器阶段具有更早熔断的高效特性。

2. 无用连接移除 (remove_useless_joins)

源码位置：src/backend/optimizer/plan/analyzejoins.c 调用时机：它发生得晚一些，是在 grouping_planner -> query_planner 的起始阶段，即准备构建底层路径（Scan / Join Paths）时进行。

有时候我们连接了一张表，只是为了取个数据，但后来发现（由于视图展开、ORM 生成的过度复杂的 SQL 等原因）这张表输出的列根本没有任何用处，那我们能直接移除这个 JOIN 吗？ 如果这是一个 LEFT JOIN，并且满足某些严格的等价条件，就能做到。

2.1 移除的苛刻条件 (join_is_removable)

要移除一个左连接的右表，必须同时满足以下苛刻条件：

  1. 没有列输出：Nullable Side（被连接的右表）的任何字段（包括 PlaceholderVars) 都没有被上层的 TargetList、HAVING、ORDER BY 引用（检查 attr_needed 集合）。
  2. 不改变左表的基数 (Cardinality)：这个最为关键。不能因为移除右表，导致原本左表匹配出多行的结果减少，或者原本膨胀的行数变少。因此右表的关联必须是最多匹配一行。
    • 优化器会调用 rel_is_distinct_for。
    • 根据连接条件，判断右表在这些条件限制下是否呈现唯一性 (Unique)。
    • 这通常依赖于数据库中建立的唯一索引 (Unique Index) 或 主外键约束 (Foreign Key)，且连接符必须是可合并连接的等值运算符。

2.2 全局状态清理 (remove_leftjoinrel_from_query)

如果这个无用连接成功被认定为可以移除，这绝不仅仅是在查询树里删掉一个节点这么简单。 因为此时规划器已经构建了诸如 RestrictInfo、EquivalenceClass（等价类）甚至是各种 PlaceHolderVar。规划器必须倒查并清理由于移除了该表（及其 Relid，甚至其特有的 Outer Join Relid ojrelid）而受到关联的所有辅助结构，确保后续基于 DP（动态规划）产生的子集连接搜索不会引用到幽灵节点。

单表扫描大军的集结 (Access Paths)

在 PostgreSQL 查询优化的漫长征途中，一旦查询树（Query Tree）熬过了逻辑重写、子查询提升、表达式扁平化以及外联结斩除等一系列严酷的预处理洗礼，规划器（Planner）便会正式拔剑，跨入整个数据库引擎最核心、也最见真章的战场——基于代价的优化阶段（Cost-Based Optimization, CBO）。

代价优化的第一落子，也是构筑整棵物理执行大树的绝对基石，便是要为查询里牵涉到的每一个基础门派（Base Relation，即基表），推演出所有能走的通的单表访问路径（Single Relation Access Paths）。在这个环节下，不管是笨重的全表顺序扫描、花样百出的各类索引奇兵，还是火力全开的并行扫荡模式，都会被事无巨细地拉到明面上排兵布阵，只待后续的打分淘汰与拼表大检阅。

本章节将结合 PostgreSQL 18.1 的滚烫源码（主战区在 src/backend/optimizer/path/allpaths.c 和 indxpath.c），带你深度潜入这套单表扫描路径的生成罗盘与暗含的数据兵法。

1. 大管家出面：make_one_rel 的统筹

生成基础路径的浩大工程，全权由核心大管家函数 make_one_rel()（盘驻于 allpaths.c）一手统筹。顾名思义，它最终的野望，是把查询里散落四方的所有表关系（Relations）像揉面团一样“揉合”成一个终极整体（One Rel），以此代表最终呼之欲出的查询结果。

不过在此之前，它必须先给手底下每一个单独的基表发放武器，武装到牙齿。

1.1 两段式的谨慎拆解

在 make_one_rel 的营房里，对单个基表的锤炼被极其考究地硬拆成了两大道工序：

  1. 摸清家底 (set_base_rel_sizes)：这步纯干算盘活儿，先摸透估算出这表会吐出多少行（Rows）、身段有多宽（Width）等一系列统计学底牌。
  2. 铺设战道 (set_base_rel_pathlists)：手握上面算出的体量底牌后，真刀真枪地开始为其幻化出由磁盘吸出数据的各种物理道路（Path）。

为什么要搞得这么神经兮兮非得切分两阶段？ 这是因为后边那种大杀器——并行查询，以及极其贼溜的参数化路径，根本离不开对表体量的精准把脉！ 举个典型的例子，要是某个表单薄得很，几页纸就写完了，优化器在估算时就会毫不留情地给它按下一个拒绝开多线程的红印（Cancel consider_parallel）。如果边估算边筑路，很容易导致前面的路修歪了，错失掉获得全局精准判断或是横攀比对的千载良机。

1.2 RelOptInfo：情报的集大成容器

在踏入筑路阶段时，每张基表、哪怕是合并后的中间态或是子查询，都会被极为隆重地封装进一个块头巨大且贯穿始终的情报中心结构体——RelOptInfo（Relation Optimization Info，定义于 nodes/pathnodes.h 这个大本营）。

在这造路一战中，RelOptInfo 有几个命脉般的心腹字段：

  • rows / reltarget: 死死捏着估算输出的基表行数和列宽指标，这俩直接咬死了随后所有扫描代价的喉咙。
  • baserestrictinfo: 也就是大家最为眼熟的 WHERE 过滤条件。在逻辑优化阶段，它们早就被肢解并隐式绑定成了带有 AND 血缘的约束限制表（RestrictInfo 的 List）。
  • pathlist: 这是一个命脉链表（List）。在这个档口，由 set_base_rel_pathlists 各种法术疯狂衍生出来的五花八门的扫描路径（Path）小队，都会乖乖在这里挂单报道。
  • cheapest_startup_path / cheapest_total_path: 当所有的跑法都被丢进沙盘试跑过后，优化器会极其功利地从中提拔出两位天选之子——拔脚开跑代价最小的和全盘跑完总折损最低的，将它俩当作“状元”被供奉缓存于此，以便日后随叫随到，快速入局。

2. 扫街派：顺序全表扫描 (Sequential Scan) 与群狼并行

顺序全表扫描（SeqScan），可以说是任何一棵基表在这个世上活下去的“最低兜底生命线”。就算你没建半个索引、就算代价高破天际，数据库也能硬着头皮，像推土机一样从硬盘上逐页刮尽全表，死抠出正确答案。

在 allpaths.c 下属的 set_plain_rel_pathlist 行省里：

2.1 兜底路线：create_seqscan_path

优化器绝不废话，一上来就会塞入一条朴实无华的 SeqScan 路径：通过底层的 add_path(rel, create_seqscan_path(root, rel, required_outer, 0)) 拍板钉钉。 顺序扫描的物理直觉就是顺着堆表（Heap Table）的底层房客分布，从第一号房（Block 0）老老实实一直敲门敲到最后一位。

  • 代价勒索要素：这个推土机的油耗其实很好算——主要就看这表占了多少个地盘（rel->pages），这锁死了连续猛读的 I/O 车马费；其次看表里到底住了多少口人（rel->rows / CPU filtering），这决定了 CPU 小人每条拉出来检查对账的心力损耗。

2.2 群狼撕咬：并行化扫描 create_plain_partial_paths

自从 PostgreSQL 点亮了查询并行化这棵科技树后，慢吞吞的 SeqScan 迎来了一位极具野性与暴力美学的孪生兄弟——部分顺序扫描（Partial Sequential Scan）。 紧跟着推土机的脚印，优化器立马会凑上去盘算：

if (rel->consider_parallel && required_outer == NULL)
    create_plain_partial_paths(root, rel);

一钻进 create_plain_partial_paths 的窝里，系统立刻会吹哨喊来算账先生 compute_parallel_worker()。这先生会根据当前这张表的腰围体量，套用一系列晦涩的数学对数阈值，去精打细算出到底值不值得叫几个闲散的后台劳动力（Background Workers）过来一并上阵开撕。 一旦拍板了打工人口数，就会调用 add_partial_path 分裂出一条挂着 parallel_workers 将军衔的分支战线。 需要特别点破的是：这里叫 partial (部分/残缺)，倒不是说大家各扫自家门前雪，而是这帮 worker 在底层共用一个抢块计数器，谁手快谁抢那个数据块，最后战果必须得汇总上交到最顶层的那个大太监 Gather 节点手里统一发落。

3. 刺客信条：索引扫描 (Index Paths) 深度解密

如果只会大声喧哗地全盘推平，那显然不够高明。对于绝大多数的 OLTP 事务秒级响应，以及那种只打算在茫茫人海里揪出极个别目标（所谓的高选择性 High Selectivity）的任务，拔出 B-Tree 这种能让耗时呈断崖式跳水暴跌的索引匕首，才是正途。

紧随其后，set_plain_rel_pathlist 便会推开 indxpath.c 里的刺客大堂——create_index_paths() 的暗门。这个函数的算计极深，它会如鬼魅般去尝试穷尽麾下所有的单柄短剑（单索引），甚至还有各种古怪的双剑合璧（Bitmap Scan 位图组合技）刺杀术。

3.1 经典重现 Index Scan：传统的单锋回马枪

当它通过向导 build_index_paths 探到查询 WHERE 里塞的暗号（比方说 id = 5）能够正好和系统账簿上记录的某个剑客 IndexOptInfo（索引特长）对上眼时，就会干脆利落地产出一条标准的 Index Scan Path。

命中与反噬： 普通的索引扫就算再准，哪怕它在聚簇树的枝繁叶茂间电光火石般锁死了 id = 5 的下落，并一把揪出了它的命门 TID（Tuple ID），它依旧逃脱不了一个极其狼狈的命运——它被迫还得使出一招回马枪（Heap Fetch 回表）。它得顺着 TID 原路杀回堆表（Heap）的数据块里，把那条血淋淋的记录全须全尾地生拽上来。因为只有这样，它才能去接着核对那些没上索引的附属条件（比如 name = 'pg') 到底还不合规矩。

3.2 刺客的巅峰体验 Index Only Scan：VM 辅佐下的无影手

但这毕竟太跌份了，于是 build_index_paths 还会暗自推演一招极其优美的变种武学。 如果在很多情况下，你的 SQL 写得十分干净利落，你只要取的首级部位，正好全在那把索引的剑鞘上刻着呢（比如只写了 SELECT id FROM table WHERE id = 5）。在这种如有神助的覆盖索引（Covering Index）梦幻情境下，刺客其实压根就不需要，也绝对不应该再回头去去翻那种又脏又沉的堆表黑块（Heap Page）。

致命破绽与 VM 面纱（Visibility Map）： 然而理想很丰满，PG 的并发世界很骨感。PG 死磕着那套经典的多版本并发控制（MVCC），而且极其偏执地把死活状态版本号跟数据本体绑在一起。这意味着，就算你只用揪出索引键，你也没法光凭索引拍胸脯保证这行记录如今在这个时空快照里**是不是已经是一具腐臭的死尸（Dead Tuple）**了，或者是不是还处于某个隐秘在途事务的庇护下对你不可见——因为所有的生死簿印记（xmin、xmax）全藏在堆表地底下的 Header 标头里，索引条目上光溜溜的什么提示都没有！ 为了能练成这套不用回表的无影手 Index Only Scan 盖世神功，PG 给索引扫配备了一张通灵卷轴——VM（Visibility Map，可见性监控网）。 在 create_index_paths 权衡利弊时，会赶紧偷瞟一眼目标表所在的那个内存块在 VM 里的明灯位：只有当该块上记录着所有人都是清白干净的 all-visible（意味着这个块最近没有陷入在跑其他肮脏事务的纷争泥潭，绝对可以放行），优化器底气才硬，敢于放手免去那恶心的 Heap Fetch 回源操作。要不然，哪怕你要的渣子全在索引上，优化器也得乖乖加上回表耗损的巨额账单比重，甚至直接将其打回冷宫退化成平庸普通的 Index Scan。

3.3 魔武双修 Bitmap Index/Heap Scan (位图连招扫描)

假使哪天碰上了不按套路出牌的主儿，你的 WHERE 里给出的不是一刀切的标靶，而是撒网式的范围扫，或者甚至祭出了 OR 将好几个不同的字段索引用法强行糅杂在一起呢（类似 WHERE age = 20 OR salary > 8000）？

这时候不管派哪个单一的 IndexScan 上场，它都会因为要在不同的索引老巢间疲于奔命而带来灾难性、令人崩溃的随机 I/O 操作狂潮 （Random I/O）。于是优化器一拍脑门，搞出了一套及其妖孽的二面绞杀战法——位图扫描大阵（Bitmap Scan）。 在 indxpath.c 的阵眼深处，全权交由 generate_bitmap_or_paths 来主理发落。

  • 上半场: 潜龙在渊 Bitmap Index Scan：先让各路刺客按兵不动，绝对不许碰真表数据一根毫毛。这帮家伙仅仅是钻进那些不同的索引树洞里各自疯狂刺探。一旦对上了号，就在自个儿拉起的一个内存虚拟大方阵里（通常是按物理 Page 级别划分的长布阵），把这行对应的站位标识符（Bit 标志位）点亮成嚣张的满血全变位 1。
  • 妙就妙在这里，如果这两个门派的活儿是 AND 交集或 OR 并集的关系，因为弄回来的全都是干干净净的纯纯内存比特黑白点阵，在内存大军中原地进行**按位与、按位或操作（BitmapAnd / BitmapOr）**那速度直叫一个快如闪电，合并过程丝滑得不可理喻。
  • 下半场: 图穷匕见 Bitmap Heap Scan：等到终于揉出最后一张硕大无比的手绘通缉 Bitmap 全图后，再全副武装拿着这张终极名录杀向真表据点。因为这次是严格顺着物理块的序列号地址，一路平推过去只敲亮着 1 的门。这一手直接将可怕的随机四散冲击，生生化骨绵掌般转化为极其优雅且无害的准顺序连续读取 (Sequential IO) 降维打击！从而极其体面地把原本成千上万碎玻璃片般的磁盘寻道消耗抹杀殆尽。这是一门极其蕴藏大智慧的镇派神功。

唯一的代价与硬伤：既然 Bitmap Heap Scan 是生拼硬凑靠物理排座位找过去的，那它吐出来的 Record 人潮当然全无当初那种“依山傍水天然的键值序列美感”。如果你老人家在上头还蛮横地强加了一手 ORDER BY age，那它将面对再也调借不出现成有序气场的尴尬境界，被逼出原形，迫使上层主管含泪自掏腰包，强行祭出那死慢死慢的内存 Sort 节点重新排队。

4. 千丝万缕的伏笔：参数化路径 (Parameterized Paths) 的精妙借力

刚才说的那堆寻常手段，多半只有闭门造车看看自家 baserestrictinfo 的时候才顶用。可你不要忘了，PostgreSQL 在下盘做战略推演的时候，眼光向来毒辣且超前，它老早就已经在做大名鼎鼎、且极其恶心的**嵌套循环大乱斗（Nestloop Join）**的最坏打算了！

你试想这么一场修罗场：大表 A 跟小表 B 狭路相逢，硬来了一句 A JOIN B ON A.val = B.id。 在这场厮杀里，要是小表 B 命苦被当成天天被翻牌子的那个被驱动内表，如果优化器只给它配了一把干干瘪瘪不带配件的扫描破枪，那不管老大 A 丢什么砖头过来，可怜的 B 全得从头扫到尾复读机一遍，代价简直要撕裂苍穹！但如果我能在前面留个后门插槽，把 A.val 作为一个**传递在途流转的活动变量引子（Parameter）**硬塞到 B 表的肚子里，等 B 在底层掏枪（调用索引）之前就能做到拔得头筹前置过滤呢？

  • 在 PG 的黑话里头，这就叫神鬼莫测的参数化借力路径 (Parameterized Path)。
  • 在这个层面上，PG 优化器不惜血本引入了特制机关组件：ParamPathInfo（参数化信息引信）。在操刀 create_index_paths 斩荆披棘的时候，优化器绝不满足只顾生造出针对 B 表自身狭隘小圈子过滤的死路径；它还会“贼溜溜”地探出头去扒一眼总司令部的连接缓冲区账目，看一眼 B 迟早要跟外人媾和勾搭上时，能不能巧借东风，顺手牵羊用别人嘴里说出的那些等值交换条件，彻底激活 B 自带那把原本生着锈的牛逼索引屠龙刀！
  • 但凡有一丝丝机会，PG 就会连夜打造一套夹带了所谓 lateral_relids（暗语：该刺客必须要依靠哪些外界供养方势力撑腰）烙印的高阶变异索引奇袭扫荡方案。这种路径如果是脱离开外层单挑，它的单独执行存活可行性差到极点甚至是 0！但这玩意一旦融入外围庞大的 NestLoop 巨型战争绞肉机中，它绝对会摇身一变，成为以极低损耗力挽千军之战局的绝杀神兵！而现如今大火的 LATERAL 关键字旁支查询，正是借由这一神坛基座，才得以在表象大发异彩、耀武扬威的。

5. 最残忍的杀戮角斗局：Pruning 剪枝与 add_path 门神

历经了上头这一番狂轰滥炸式的头脑风暴，别以为只有惊世大查询才开枝散叶，这哪怕就是遇上了最普普通通的一只底层阿猫阿狗表，优化器都会为其凭空捏造出几十上百条奇拉八怪的奇特跑法路线 Path（有纯靠脚走的 SeqScan, 骑马的特指索引X, 坐轿的特指索引Y, 双人抬轿 Bitmap 组合花枪，还有群殴并行…）。

要是一直这么任由它们子子孙孙无穷衍育下去，还没等最后真正两两大联表，光是那指数级膨炸出来的怪异连接组合数量（Path Explosion），就足以把机器的内存条活活吸干爆裂！所以，高高在上的主优化器必须时不时地举起屠刀清理门户，这就是血腥的——路径剪枝大逃杀（Path Pruning）。

那些被妄想出的不管好的坏的成百上千条扫描捷径，绝不会平白无故就被拉到名册 RelOptInfo->pathlist 队伍里头滥竽充数混饭吃。所有角色统统必须先排着队，去会一会那唯一掌控着生杀大权的镇派老门神：极其无情的过滤审判函数 add_path(RelOptInfo *parent_rel, Path *new_path)。

add_path 躯体里深埋着一条类似于帕累托最优（Pareto Optimality）那般极其现实主义的丛林淘汰碾压法则。 当某个怀揣着美梦的新生代 Path 来敲门想求录用挂单入册时，门神会拎着它的领子，把它跟 pathlist 门下那些早就占好席位的老骨头们展开一对一绝不留情的贴身死斗考核：

  1. 赤裸裸的价码拼杀（Cost Fudge 模糊裁决）：门神会对比它俩起步开跑的消耗代价（startup_cost，也就是为了先吐出头一行能拿来应付限流 Limit 的启动预热花销）和跑到头的死磕总消耗（total_cost）。如果新来的选手两腿沉重，这俩盘条数值都输得死死地高于某个早就稳坐名录上的老混混。那不管怎么看你这新兵蛋子也就全是缺点一无是处了，门神当场将其拉走人道毁灭。（当然，在这个修罗场里 PG 非常细心地引入了极为微观大约在 1e-10 这等毫末级别的模糊缓冲乘数 fuzz，就是为了防着碰上了因为破烂浮点数对齐舍入那种让人无语的误差而被判冤死的情况发生）。
  2. 身带免死金牌的高等爵位排序特征 (Pathkeys) ：这是 PG Planner 非常精妙、甚至是惊艳全场的一个反转后门大绝杀！万一出现了这么个局：新人初来乍到，新造的路子总消耗巨大看面相简直必死无疑。但是老路子弄出来的结果却是一盘毫无规律的粗暴散沙乱炖。而此新人路线（因为碰巧走了按键值生长的老牌 B-Tree 名门望族索引），这打娘胎里带出来的结果恰好完美迎合且对路了上一层长官急需的 ORDER BY 天生排序序列（自带光芒万丈的 pathkeys 特质）！ 这下门神可绝不敢单凭那一手账面上的烂开销就擅作主张把它给踢出局了！因为这新人相当于自带了军团极需的高阶战略免检福利，它为老东家提早绕开了跑到上头还要极其恶劣地外挂开辟巨无霸独立排序内存（那些恐怖的明文显式大 Sort 死结关卡）而变相立下战功，扭转乾坤！此时它非但不用死，更将被作为高贵的特型种子选手与老路径平起平坐被全数并存破格录用保留入军团名册里。

等最终肃清淘汰干净了那些里外各个层面（既拼不过出场穷尽血量、又拿不出半点特殊高阶花招特质收益）全部都一无是处的杂鱼淘汰品以后，留在那个历经千锤百炼战火洗过的神圣 rel->pathlist 名录金牌榜里的，就只是一批高度提纯过的、压缩了最恐怖底盘潜力与精血的至高扫描路径结晶战尊们了。它们都在各自摩拳擦掌蓄势待发，只等在随后那波诡云谲连环拼表的 Join 洗礼大战中掀起属于自己的最终腥风血雨！

多表连接次序与等价类推导 (Join Ordering & Equivalence Classes)

当查询从单表跃升为多表连接（Multiple Joins）时，优化器迎来了它最耗算力的终极考验：决定表的连接顺序（Join Order）以及连接方式（Join Methods）。

在理论中，N个表的联接顺序组合数呈阶乘爆炸。为了寻找最优解，只要参与联接的表数量低于预设的 geqo_threshold 阈值（默认 12），规划器就会坚定地踏入自底向上的动态规划（Bottom-up Dynamic Programming）深水区。

本章节紧扣 PG 18.1 在 src/backend/optimizer/path/joinpath.c、joinrels.c 乃至 equivclass.c 的源码，深度剖析动态规划建树过程。

1. 动态规划底盘：致敬 System R

PostgreSQL 骨子里流淌着 IBM System R 奠定的经典动态规划（DP）搜索思维。

1.1 从单打独斗到万物互联

动态规划秉持最优子结构思想：求 N 张表合并的最优解，就从底部的任意 2 个表搭伙算起，再算 3 张的……以此类推。负责发号施令的主函数为 make_rel_from_joinlist。

算法调用 standard_join_search 进入标准的 DP 轮回：

  • 第一层 (Level 1)：单表路径（来自 set_base_rel_pathlists 的成果）业已就绪，被投入到 join_rel_level[1]。
  • 状态跃迁 (Level 2~N)：外围庞大的循环从 Level = 2 开始爬坡。每一层的新组合都由之前计算出的最优子集拼合而来，执行核心递归计算枢纽是 join_search_one_level。

1.2 Left-deep 与 Bushy Tree

在连接树的探索过程中，有两种典型的生长结构：

  • 左深树 (Left-deep Tree)：老连接结果集直接和单个新基表去拼凑。这种操作占用的内存上下文极少，非常契合 NestLoop 带 Index Scan 这种流式循环。
  • 灌木丛树 (Bushy Tree)：允许两个中途拼出的临时大合集互相合并。当复杂查询两侧同时具备高过滤性时，它能打出断代级别的性能优势。

相比古典引擎死守左深树，PG 在 join_search_one_level 推进下不仅包容左深树路线，更敢于让老手互拼去探索狂放的 Bushy 魔树空间！

2. 推理大师的逆袭：等价类 (Equivalence Classes, EC)

在放飞动态规划之前，绝对绕不开一张最强底牌：位于 src/backend/optimizer/path/equivclass.c 的等价推理机制。

2.1 隐式约束的高维度拦截

如果有句 SQL 写着 WHERE A.id = B.id AND B.id = C.id AND A.id = 5。传统引擎往往忘了通知还没见面的 B 和 C！ 但 PG 凭借等价类结界 EquivalenceClass 实体桶，直接将 {A.id, B.id, C.id, 5} 当成同一根绳上的蚂蚱死死绑定。

随后祭出 generate_implied_equalities 神技：不管优化器当下在哪个角落想要扫描底层的 B 或者 C，它都会下意识查老底。一看 EC 里存在常量 5，立马如获至宝地凭空变出 B.id = 5 和 C.id = 5 的前置隐式拦截指令。 这就赶在多表庞大合并引发雪崩之前，通过直接调用单表的 Index Scan，硬生生把上亿行的烂摊子秒杀在前线阵地上。

3. 点兵列阵的终极派活：populate_joinrel_with_paths

当 DP 决定拿 RelA 和 RelB 拼作一个大联盟（Rel A+B）后，它会将打磨三大物理连接方式的脏活苦活悉数推给 populate_joinrel_with_paths() 熔炉函数。

3.1 狂野盲连与死磕：NestLoop / HashJoin

该函数一上来大喊 match_unsorted_outer，测试原生态无序对抗：

  • NestLoop Join：如果此时手头带有参数化借力武器（Parameterized Paths），外表便可将循环的每行值作为狙击参数下发给内表，从而使得内表直接唤醒沉睡中的索引发挥神效，绝境逢生翻盘！
  • Hash Join：要是碰到了等值交换，并且算盘先生评估内表塞进内存哈希桶（Hash Bucket）的开销尚能忍受，这就成了大批流灌数据相互撕咬最高效的无序神兵。

3.2 讲究排场的序列：Merge Join

随后的 sort_inner_and_outer 则要高雅得多。 Merge Join 极度依赖双方自带的序列。它如饥似渴地扒着双边结果集的 pathkeys（输出特征），看看它们原生的排序是否对上眼了联接条件。如果恰好都有命合，这招的发起成本便是一路雪崩跌到谷底；如果不巧没有，哪怕硬着头皮加上高昂内存 Sort 花费，PG 也会把它送上决策席比对。

3.3 Materialize 的缓存魔法

如果在多层 Nestloop 内部嵌套里发现某个内部小圈子被反复频繁读取而累得喘不上气，系统会自动呼唤 create_material_path 在途中加塞一个钉子般的缓存节点，保留救命结果集。

最终，各显神通的所有招式会被集体塞入无情的 add_path 刽子手门外。它像极其功利的督工一样排查评估谁跑得快谁带的好处多，把没用的统统扔进深渊，剩下的真金战神统统稳坐 RelOptInfo->pathlist 决胜席位。

遗传启发算法 (GEQO, Genetic Query Optimizer)

在前一章中，我们深度领略了 PostgreSQL 自底向上的动态规划（Dynamic Programming, DP）算法。这套精密至极的机制，誓要在错综复杂的表网中抠出一条理论上绝对完美的执行路线。

但这套机制并不是无底洞。即便是启用了启发式剪枝，DP 枚举排列的开销同样也会呈现不可阻挡的指数级大爆炸。倘若你在超大型数仓的复杂视图里牵连了上十几二十张表（超越 geqo_threshold 默认值 12 的警戒线），如果非要强摁着 DP 去穷举到底。那么可能到最后，光是把“计划算出来”所花死等的时间，就足以超过去硬表硬扫实际跑完总时间的好几百倍！

这时候，就轮到 GEQO (Genetic Query Optimizer 遗传式查询优化器) 挺身而出了。 它不指望算出绝对最佳组合，但它笃定向你保证，在仅需区区几十毫秒推演里，利用生物演化的力量，就能给你端上来一道**“未必天下第一，但也绝对堪用之上乘”的退求其次解法**。

1. 隐喻对账：SQL 乱炖化身旅行商难题 (TSP)

遗传演化的关键，就在于如何把问题重新包装成“染色体”结构变种。 在 GEQO 视角下，多张大表的连接顺序争杀其实就是在重演全球最火爆的 NP 级数学难题——旅行商问题 (Traveling Salesman Problem, TSP)。

  • 城池 = 待匹配的基础表（Base Relation）。
  • 车道 (Tour) = 表之间连接的先后序列组，例如 $A \to C \to B \to D$。
  • 盘缠开销 (Path Cost) = 顺着这条方案往下跑，底层执行器所需的代价（Total Cost）账单。

在 PG 的源码限定下，GEQO 系统被剥夺了探界繁杂无比的多路 Bushy 树魔阵的野心，只能踏实地在传统的 Left-deep 树（一条道走黑、线性联营序列）这方小天地内推演染色体排谱。

2. GEQO 的全局生物演化钟

这套基因游戏全赖 geqo_main() 发条手拨转。整个轮盘涵盖下面这五段宛若天神造物的创世纪流程：

2.1 鸿蒙初辟：种群大爆发 (Initialization)

造物之初唤起 random_init_pool()。 系统直接按掷骰子般的随机序列排列组合法，粗暴地先抓出一把“先头部队”路线放入初始池（Pool）！基表数量 N 越多，这繁衍大池子也会基于特定的公式膨大激增，产生数百只怪胎个体。

2.2 阎王判官：生存力测算 (Fitness Evaluation)

残酷的试金石主掌在 geqo_eval() 之手。 对池子里的每一个孤单小串串（比如排序集合 $A, B, C, D$），GEQO 自己压根都懒得推敲。它极其讨巧地跑去白嫖那个 DP 正规军手头的传统算账代价引擎（Cost Engine）！ 它硬逼着别人把这次序捏成完工的 Join Tree，塞给 cost_qual_eval 底座让它死命算。算完拿到一个生冷的代价值，倒转印刻为自己在这个大池里能不能存活的资本——适应度 (Fitness)。

而后它大手一挥，无情调用 gimme_pool_size() 将这一水箱的所有人马依据分数排高低搞大榜单 (Ranking)。占据榜单制高点那几个就是战力最猛的，在底端垫底的随时面临斩草除根之命。

2.3 天选提拔 (Selection)

基因界里越是彪悍，那必定得狂撒种子。为了不彻底垄断，它用上了坐落于 geqo_selection.c 大殿里的 基于排名的线性择优术 (Linear Ranking Selection)。 排名极其靠前的（消耗越少的）个体，其作为准双亲被钦点繁衍大业的基率极高；不过即便你是底端垫底的，依旧存留微小的配种率以防止大池基因僵死。

2.4 最核弹魔力交叉互融 (Crossover / ERX)

PG 在远古大作 geqo_erx.c 选定了一个最高绝阵法：边缘重组交叉算子 (Edge Recombination Crossover, ERX)。 这玄学法术能够极其贪心保留了那一代代遗传下来的“连襟死党邻座对”。比如当爹的那里有 A->C，当娘的恰好也有 C->A！说明这对死基友有着天生优良联系。到了小下一代，子程序就会疯狂向这些游离周边基因靠拢，死死保留住新子代的绝强捆绑落段。完美吻合联接树“相邻结合最好”的物理特征。

2.5 为了不死循环的叛逆突变 (Mutation)

要是成百代的人只死盯交配繁衍不加变化，全群早晚会在一个名为 局域极值耗死 (Local Minima) 的巨坑里憋死（大家长得一模一样，再也配不出更佳的优形种）。 在此 geqo_mutation() 给每个新下的瓜注入疯血乱性小药剂。通过扔极个别的乱骰数，强行将新生一代里的第三和第五位成员硬做一次对调置换叛乱，强行注入野生新息，推拉进化车轮脱离局限烂泥。

3. 玄学终局的苦果：因果命数

当历经了 geqo_effort 参数控制指令下的足足成百代疯狂更迭发酵之后，主神降临强硬鸣金喊停。淘沙剩下来的最终池子里，傲然屹立爬到全团顶尖排位座次第零位的最彪悍极品，会被如神选般抽将出来。 规划器：“天命难违，弟兄们下海吧，我们就干这票 Join 图形了！”

被副作用反噬：让人心脏骤停的善变计划

要是你真的在十来张超巨大核心表里放肆享受了大杂烩拉联操作，你势必迎来一场灵异戏剧： 既然这套遗传玄学打从始至终就带有极大运气伪随机脾性，这不就代表着即便是同样一模一样的惊天 SQL。早上拿出的跑位路线，和下午拿出来的最终底盘执行解析解释图表（Explain Plan），两者竟然很有可能是长得鸡飞狗跳、完全大相径庭的两棵截然不同狂树！ 查询响应时间会出现长短不一的玄学心跳图。

你要是深受这玄学波动困扰，想要绝对稳定执行期这稳坐钓鱼台的话：

  1. 请干脆把该连的 geqo_seed (所谓的演变乱斗随机种子主值) 定死为一个你熟悉永远不动的固化常数值去打保底复现重演锁定死。
  2. 要是你有着多的大运存时间也不计较前期图纸计算耗时，干脆强杀 SET geqo = off 去锁死其运作，也就是放极大化那个 geqo_threshold 去活活让老底儿动态计算老黄牛去强推平所有的动态阵列！

后置优化与物理算子拼接 (Post-Optimization & Upper Planner)

如果说查询规划的“上半场”（代码里叫 Lower Planner，即底层的 Join 树构建）解决的是“去哪找数据、按什么顺序拼最快”的问题，那么规划机制的“下半场”——Upper Planner (高层优化器)，解决的则是“拼完了数据怎么满足用户最终的语义控制”的问题。

当 FROM 和 WHERE 连接网络被 query_planner 函数结算为一颗拥有最优代价值的 RelOptInfo 返回时，优化器会回到其广袤的母调度函数：grouping_planner（位于 src/backend/optimizer/plan/planner.c）。

在 grouping_planner 中，系统会为这棵刚刚建成的基底数据树逐层裹上“包装”，处理包括但不限于：GROUP BY (聚合), WINDOW FUNC (窗口函数), ORDER BY (排序), DISTINCT (去重) 以及 LIMIT/OFFSET (截断) 等高级操作。

结合 PostgreSQL 18.1 的最新代码，我们将深入这条“物理算子包装流水线”。

1. 入口统筹：Upper Rel 与 grouping_planner

与底层基表或连接不同，PG 抽象出了一种名为 Upper Rel (上层关系) 的 RelOptInfo，通过 UPPERREL_GROUP_AGG、UPPERREL_WINDOW 等专有分级不断往顶层堆叠。

在 grouping_planner 中，标准的装配链是极其严苛有序的。每装配完一层，就会生成一个涵盖了前面所有计算结果的新 RelOptInfo：

  1. 底层扫描与联接 (query_planner 完成)。
  2. （如有）追加分组与聚合 UPPERREL_GROUP_AGG。
  3. （如有）追加窗口函数计算 UPPERREL_WINDOW。
  4. （如有）追加 SELECT DISTINCT 去重。
  5. （如有）追加 ORDER BY 排序 UPPERREL_ORDERD。
  6. （如有）追加 LIMIT / OFFSET 截断。

下面重点剖析这套外置神兵利器的核心模块。

2. 聚集规划 (Aggregation & Grouping)

在面对包含聚合函数（SUM(), AVG()）或声明了 GROUP BY 的查询时，系统会跳入 create_ordinary_grouping_paths() 去生成聚合方案。

PG 支持三种硬核的物理聚合算子：

2.1 基础聚合 (Plain Aggregate)

应对的是诸如 SELECT count(*) FROM table，没有 GROUP BY 的查询。 系统只需在顶端挂载一个单行的聚集器。输入流来了之后，它只会在内存中累加内部状态值，流结束后输出唯一的一条合并行记录。启动代价小，逻辑极其精简。

2.2 排序归并聚合 (GroupAggregate / Sorted Aggregation)

要求输入的数据流必须已经按照 GROUP BY 列排好序。

  • 机制：类似于流式游标。当读入的值和上一条的值在 Group 键上一样时，累加聚合状态；一旦读到新的值（越界），立即抛出刚刚结算完的那个聚合组结果，并重置计数器重新开始。
  • 优缺点：内存占用极致省钱。但代价是：如果你底层吐出来的数据压根没有顺序（比如 HashJoin 出来的），GroupAggregate 就会惨叫着要求在底下先强压一个昂贵的 Sort Node（排序算子）。

2.3 哈希聚合与落盘 (HashAggregate & Disk Spill)

不需要预先排序。

  • 机制：上来就在内存（work_mem）里修筑一个庞大的 Hash Table。扫入的数据通过计算 Hash 值强行扔入不同的 Bucket 并累加状态。扫完全表后，再将结果一口气倾倒出栈。
  • 重大进化：在 PG 13 之后加入了 HashAgg Disk Spill (落盘机制)。这意味着一旦发现分组数量超过内存，它不会简单粗暴抛弃哈希聚合，而是会引入多批次落盘合并的额外磁盘惩罚，极大地增强了稳定性。

3. 窗口函数 (Window Functions)

在聚合之后，就是窗口函数的战场 UPPERREL_WINDOW（位于 planner.c: create_window_paths）。

当你用到了 SUM(x) OVER (PARTITION BY a ORDER BY b) 这种强大的跨窗函数时，必须明白：

  1. 窗口函数实质上是一种变种的 GroupAgg，它极端渴求输入序列必须按 a 和 b 严格有序。
  2. 多窗口合并优化：如果一个 SQL 里有五六个窗口函数，有的 PARTITION BY a ORDER BY b，有的则是 PARTITION BY c，PG 会在 group_window_funcs 操作里尝试把 OVER 子句诉求完全一致的合并在一次节点轮询里执行掉。不同规则的必须被切割成独立栈。
  3. 优化器会重新评估底下这堆由聚合输出的路。假如下面的路正好自带按顺排好的 pathkeys，那就省去了二次排序。如果没有？那就硬着头皮加上显式的 Sort Node。

4. 排序神器：增量排序 (Incremental Sort)

这是近期 PostgreSQL 带来的真正惊艳业界的一大特性。 所有的排序需求都要通过 create_ordered_paths 往上塞。对于最棘手的 Sort 我们以前只有全表堆排序。

试想场景： 假如你需要 ORDER BY a, b。由于你在 a 上有一个出色的索引，底层 Index Scan 捞上来的数据已经按 a 排好序了，只是 b 是乱序的。也就是所谓的“提供了部分前缀排序（Prefix Sorted）”。 在没有 Incremental Sort 之前，PG 会长叹一声：“反正 pathkeys 对不上 (a,b) 组合，底下的排序全当废纸！全量装进内存强制 Sort！”。

Incremental Sort 机制： 现在，由相关逻辑捕获 Prefix Sorted 特征。优化器能够插入一条轻量级的 Incremental Sort 路径。 在执行时，只要探测到前一个读取的 a 的值没变，它就把 b 暂时吃进缓存；一旦 a 变了，这就立即针对前面那个小缓存栈里的区区几条数据对 b 进行极低代价的微型内存 Sort 并输出。 这个技术直接将长尾查询的延迟削减了数量级。

5. 截断 (Limit / Offset) 对底层引擎的倒逼

如果你以为 LIMIT 1 只是最后加一个截断算子，那就大错特错了。 LIMIT 具备极为强烈的计划倒置影响力 (Plan Inversion Power)！它存在于 create_limit_path() 之中。

当你的整棵查询树带着几十万总代价值，遇到 LIMIT 10 时： PG 的代价估算器不仅修改上层代价值，甚至会去翻阅之前在底端遗留的所有未决执行路径。

之前说过，优化器会保留两条路：cheapest_startup_path (启动极快但总体最慢) 和 cheapest_total_path (启动很慢但总体最快)。 如果是没有 LIMIT 的提取：Hash Join 由于建哈希表，扫全表 total_cost 极低，必定压倒 NestLoop。 但在有了 LIMIT 之后：优化器只需要拿到十行记录。Hash Join 因为需要先把两张表各自的几百万全缓存下来才能吐出第一行，其 startup_cost 令人发指。而看似非常笨拙的 NestLoop + Index Scan 却能在 0 缓冲的情况下瞬间拿到第一行、第十行的结果并直接终止循环。

在 LIMIT 的神来之笔下，优化器会给拥有最优 startup_cost 的路径重新打上极高的权重，成功颠覆由总价主导下的 HashJoin 王权，切换到那些极速启动的参数化嵌套循环和纯净索引路径。

并行执行计划生成 (Parallel Query Planning)

自从 PostgreSQL 9.6 划时代地引入并行查询架构以来，如今在 PostgreSQL 18.1 中已经发展成了一个极为庞大、健壮且无缝融入代价优化器（CBO）的深水区模块。

并行查询的魔力在于：当面对亿级别的海量数据集扫表或重型聚合时，规划器可以果断打破此前单进程单核运算的桎梏，派遣多个后台工作进程（Background Workers）分进合击，最后再由发起查询的主进程（Leader Process）进行数据汇总。

在这个章节中，我们将深深扎入源码底层的并行网格，搞清楚“谁在下派任务”、“怎么汇总结果”以及“惊艳的并行 Join 是怎么实现的”。

1. 并行的基石抽象：Partial Path 与 Gather

所有的并行计划，在优化器里的表现形式无非是两个全新出现的构件。

1.1 部分路径 (Partial Path)

如果你之前读过“单表扫描”章节必然记得：在 set_plain_rel_pathlist 中，系统不但会添加一条 SeqScan 路径，还会调用 create_plain_partial_paths 添加一条被称作 Partial Path 的分身。

什么是 Partial？ Partial 的物理意义是：“这条路只要扔给一个独立 Worker 去跑，它只会吐出现有总数据量的一个局部切片（Slice）”。 为了实现互不干涉的切片，底层执行器在扫表时会依赖动态共享内存（DSM）里维护的原子游标（Block Counter）。多个 Worker 拿原子操作去抢页，谁抢到了谁读，导致输出的数据是无序且切分的。 在规划阶段（Planner），它通过设置并行进程数大于零，并且将此路的预期代价值按比例折算入库。

1.2 并行安全 (Parallel Safe) 的铁律

不是所有的一切都可以拉起并行！优化器在任何一个关系被标记为 consider_parallel = true 前，都会启动一场极度严苛的 is_parallel_safe() 地毯式搜查。

  1. 函数检验：如果在 WHERE 或者 TargetList 里面引用了一个被标记为 PARALLEL UNSAFE 的用户自定义函数（比如直接写表的函数），那么整个脉络将永久拉黑并行。
  2. 写事务隔离：绝不可以在含有未提交缓冲数据的临时表（Temporary Tables）上生成并行作业，因为 Worker 进程不允许看到 Leader 的私有本地内存更改。

2. 收集节点：Gather 与 Gather Merge

光有各个 Worker 在底下苦哈哈地算出局部结果（Partial Path）是不够的。这些支流必须要被收束回主进程（Leader Process）形成唯一的流。 优化器会通过不断往上推进调用 generate_useful_gather_paths() 强行为底部的 Partial 树冠盖上一个叫 Gather Node 的盖子。

有了 Gathering，一条残缺的 Partial Path 在顶层看来就重新变成条完整的正常路参与总角逐。按照是否保序，PG 提供了两大收束神器：

2.1 粗犷提速的无序统合 (Gather)

最经典的 Gather 算子。 Leader 创建多个队列池并启动协程。各个 Worker 算完数据就往共享内存的消息队列里无脑塞。Leader 轮询拿到什么数据就立刻吐给客户端。这是对性能损耗最低的手段，但它把所有的底层排列完全打乱了。

2.2 优雅的保序收集 (Gather Merge)

这是一个极为考究代价取舍的东西。 假设你底下一个 Worker 在走并行 B-Tree 索引扫描（保证了单 Worker 自己读到的局部区块是有序的）。Leader 在收到这 N 个流时，如果不加思索地套用 Gather，出来的总集就是一堆交错乱序的数据。

但在某些场景下，上层非常需要这个 ORDER BY 排序。 此时 PG 会盖上一个 Gather Merge！ Gather Merge 节点自带一个 堆排优先队列 (Min-Heap / Priority Queue)。它死死盯住 N 个 Worker 发来的管线头，不断抓取极小值向上抛出，从而完美确保了 N 个已排序局部流在合成时，丝毫不破坏总体的有序性！

3. 并行连接 (Parallel Joins) 的底层黑科技

如果并行只能做全表扫，遇到 Join 依旧回退到串行那就如鸡肋。PostgreSQL 的王炸在于针对三种不同的物理 Join 设计了出神入化的并行执行版本。

3.1 Parallel Hash Join (并行哈希连接)

普通的 Hash Join，如果让 N 个进程并行，很可能会有 10 个进程在内存里各自建了一个一样的外表 Hash Table，完全浪费内存，甚至 OOM。

在 PG 中，通过了史诗级的同步：基于 DSM 的 Shared Hash Table（共享哈希表）。

  1. 建表 (Build Phase): Leader 在 DSM 中开辟好小表的坑位。所有的 Worker 一拥而上读取内表，然后共同把 Tuple 塞进这唯一的大 Hash 桶里！
  2. 锁步栅栏 (Synchronization Barrier): 用一个全局信号，强制等所有 Worker 把建表拼完。
  3. 嗅探 (Probe Phase): 放开栅栏！所有 Worker 扭头去对外表（Outer Table）开启无序分块扫。扫出一条就跑去大表碰哈希。彻底把 Join 计算跑满多核 CPU。

3.2 Parallel Nested Loop (并行嵌套循环)

这个看起来非常粗暴但往往有奇效（尤其当使用参数化路径 Parameterized Path）时。

  • 外表切片：Outer Table 必须是一个 Partial 数据流状态（多个 Worker 在抢外表数据）。
  • 内表广播：Inner Table 是串行的 Full 扫描。这意味着每一个 Worker 每次抢到一行外表数据后，都要拿着这行数据去对完整的内表进行一次（通常是 Index Scan 的）扫描。这类似于大数据的 Broadcast Join 思想。

4. 并行查询的代价模型打压 (Cost Punishments)

你或许会问：既然并行这么天下无敌，为什么 PG 大多数查询还是单机的？ 优化器里存在两个著名的估算打压：Setup Cost 与 Communication Cost。

  1. parallel_setup_cost (启动成本高昂) 拉起后台进程是一个结结实实的操作系统 fork()。优化器会在计划里硬砸下默认高达 1000.0 代价的初始启动惩罚，这意味着只有执行时间可能需要大几秒的沉重查询，才“配得上”并发。
  2. parallel_tuple_cost (通讯细微阻塞) 将数据从 Worker 写入环形缓冲，并在主进程抽出来。这是一笔不可忽略的 IPC 收发代价。若查询结果有千万级交由 Leader 吐出，这笔惩罚加起来极为震撼。系统可能会因此妥协：“还是干脆单核扫完，免得在内存里互相等锁阻塞”。

通过对上述所有精妙代数模型估算并在最后层交火比拼。在 grouping_planner 收尾时刻，如果是自带 Gather 的这棵庞然大物路径总体成本比串行确实大幅胜出，那么恭喜，这张 SQL 稿件，即将驱动整个服务器的 CPU 集群矩阵疯狂咆哮了。

standard_planner 函数入口

standard_planner 是 PostgreSQL 默认优化器的主要入口函数。

位置: src/backend/optimizer/plan/planner.c

整体流程

standard_planner 的执行流程大致如下：

PlannedStmt *
standard_planner(Query *parse, const char *query_string, int cursorOptions,
                 ParamListInfo boundParams)
{
    /* 1. 初始化 */
    root = makeNode(PlannerInfo);

    /* 2. 预处理 (Subquery Pull-up, Constant Folding 等) */
    /* 实际上这部分主要在 subquery_planner 中完成 */
    root->parse = parse;

    /* 3. 调用 subquery_planner */
    /* 这是递归优化的核心，处理子查询和当前层级的优化 */
    plan = subquery_planner(root->glob, parse, NULL, false, tuple_fraction);

    /* 4. 最终收尾 */
    /* 将 Plan 树转换为 PlannedStmt，准备交给 Executor */
    result = makeNode(PlannedStmt);
    result->planTree = plan;
    
    return result;
}

subquery_planner

subquery_planner 负责处理特定查询层级的逻辑。

  1. pull_up_subqueries: 提升子查询。
  2. preprocess_expression: 规范化表达式。
  3. reduce_outer_joins: 外连接消除。
  4. grouping_planner:
    • 这是真正进行路径搜索的地方。
    • 它首先调用 query_planner 来处理 Scan/Join 优化。
    • 然后自己处理 Group By, Agg, Sort, Limit 等非 Join 操作。

query_planner

query_planner 是核心中的核心 (The Core)：

  1. setup_simple_rel_arrays: 初始化 RelOptInfo 数组。
  2. add_base_rels_to_query: 识别所有基表。
  3. make_one_rel:
    • set_base_rel_pathlists: 生成单表访问路径。
    • make_rel_from_join_list: 生成多表连接路径 (DP / GEQO)。
    • 返回一个代表最终结果关系的 RelOptInfo。

总结

调用栈： standard_planner -> subquery_planner -> grouping_planner -> query_planner -> make_one_rel

核心数据结构 RelOptInfo

在 PostgreSQL 优化器中，一切皆 RelOptInfo。

定义: src/include/nodes/pathnodes.h

概念

RelOptInfo (Relation Optimizer Info) 用来表示一个“关系”。这个关系可以是：

  1. Base Relation: 一个真实的表（或子查询结果）。
  2. Join Relation: 两个或多个表连接后的结果。
  3. Upper Relation: 进行了聚合、排序后的结果。

在优化过程中，优化器会为每个参与的表创建一个 RelOptInfo，然后通过两两组合创建新的 Join RelOptInfo。

关键字段

typedef struct RelOptInfo
{
    NodeTag     type;
    RelOptKind  reloptkind;     /* RELOPT_BASEREL, RELOPT_JOINREL, etc. */
    Relids      relids;         /* 包含的基表 ID 集合 (Bitmap) */
    
    Rows        rows;           /* 估算的行数 (Cardinality) */
    int         width;          /* 估算的平均行宽 */
    
    List       *reltarget;      /* 关系的输出列 (TargetList) */
    List       *pathlist;       /* 这是一个 Path 列表！存放所有可能的访问路径 */
    List       *ppilist;        /* ParamPathInfo list */
    Path       *cheapest_startup_path; /* 启动代价最低的路径 */
    Path       *cheapest_total_path;   /* 总代价最低的路径 */
    
    /* ... 统计信息, 索引信息, 等价类信息 ... */
} RelOptInfo;

Path 与 RelOptInfo 的关系

  • 一个 RelOptInfo 代表逻辑上的一个关系（比如 A Join B）。
  • pathlist 存储了实现这个关系的多种物理方式（比如 NestLoop(A, B), HashJoin(A, B)）。
  • 优化器的目标就是为最终的 RelOptInfo 找到 cheapest_total_path。

Path 数据结构

typedef struct Path
{
    NodeTag     type;
    NodeTag     pathtype;       /* T_SeqScan, T_IndexScan, T_NestLoop ... */
    RelOptInfo *parent;         /* 所属的 RelOptInfo */
    
    Cost        startup_cost;   /* 启动代价 (返回第一行前的代价) */
    Cost        total_cost;     /* 总代价 (返回所有行) */
    
    List       *pathkeys;       /* 结果的排序属性 */
} Path;

每种具体的路径（如 IndexPath, NestPath）都继承自 Path 结构体，并增加特定的字段。

等价类 (Equivalence Classes, EC) 深度剖析

在 PostgreSQL 优化器的世界里，无论是对单表扫描条件的过滤，还是在复杂的 Join 规划中寻找匹配路径，亦或是处理 ORDER BY 时查找可用的排序索引，其最底层都离不开一个贯穿全局、至关重要且极其精妙的抽象概念——等价类 (Equivalence Classes, 简称 EC)。

由于关系型数据库的查询语句中经常会出现大量相互关联的过滤条件，例如 A = B AND B = C AND C = 5。如果优化器仅仅从语法层面对抽象语法树进行机械的解析，它就只能识别出 A 与 B 有关、B 与 C 有关。

然而，一旦系统引入了“等价类”这个基于离散数学中“传递闭包（Transitive Closure）”概念的抽象机制，优化器就仿佛开启了“上帝视角”。它能够在瞬间在这些相互关联的数据关系中，刻画出了一张全图连通的无向图。它能毫不费劲地推断出 A、B、C 这三个字段在逻辑上是完全等价的命运共同体，而且在查询执行时，它们实际的数值必定都等于常量 5！

本文结合 PostgreSQL 18.1 的最新源码（其核心逻辑实现在 src/backend/optimizer/path/equivclass.c 与 src/include/nodes/pathnodes.h 中），对这个经常被外围技术文章一笔带过，但在 PG 底层却发挥着中流砥柱作用的“幕后英雄”进行一次抽丝剥茧般的硬核拆解。

1. 源起：等价类的生命周期与无向图抽象

当一条 SQL 语句经过解析和重写，形成查询树到达优化器被展开处理后（例如在 initsplan.c 中的 distribute_qual_to_rels 阶段），系统在遇到 WHERE 或 JOIN/ON 子句中的各种二元布尔表达式时，便开始寻找等价关系的蛛丝马迹。

当规划器捕获到一个形如 X = Y 的表达式时（严格来说，这里的 = 操作符必须能够在 B-Tree 索引中支持排序，并且在 pg_amop 系统表中被声明具有哈希相容性和等价语义，通常作为 RestrictInfo 存在），系统并不会简简单单把它当作一个普通的过滤条件随手下发。相反，系统会立刻暂停通用处理流程，转而触发一个专属的核心构建函数：process_equivalence()。

1.1 从离散条件到传递闭包的野蛮生长

process_equivalence(PlannerInfo *root, RestrictInfo **p_restrictinfo, bool below_outer_join) 函数的核心职责就是将一个个零散的点，编织成一张互相连通的拓扑网：

  1. 新建与收容：假设当前优化器是一张白纸，系统第一次遇到了等式 X = Y。它会立刻分配内存，创建一个新的 EquivalenceClass (EC) 实例对象，并将 X 和 Y 这两个表达式各自包装成 EquivalenceMember (成员) 存放进该集合的链表中。
  2. 顺藤摸瓜的扩容：紧接着，系统读取到了下一个谓词 Y = Z。在处理这段逻辑时，引擎会先去检索当前已经存在的所有等价类集合。它会敏锐地发现表达式 Y 已经存在于刚刚建立的某个 EC 中了。此时，系统不会再去重复建构新的集合，而是利用等价的传递性，直接把新出现的成员 Z 加入到包含 X, Y 的已有等价类里。于是，这个集合扩充成了 {X, Y, Z}。
  3. 不同集合的合并：在复杂的查询中，这种情况非常普遍。假设系统在处理某一侧的表连接时，推断出了一个局部的等价类 {A, B}。而在处理查询的另一部分时，得到了另一个局部的等价类 {C, D}。此时，主查询抛出了一个跨越这两侧的连接条件 B = C。 见证优化的合并时刻上演了！优化器在检索时发现了这两个相互独立的集合片段，于是立刻执行底层的并查集 (Union Find Set) 操作，将这两组集合做物理级别上的融为一体。瞬间，它们合并成了一个跨越整颗查询大树的超级等价族：{A, B, C, D}！

通过这种不断进行自我融合与扩张的动态打法，原本在语法树上只是零散字符排列组合的 WHERE 条件语句，在 PostgreSQL 的核心内存池中，被升维成了四通八达、信息密度极高的数据依赖拓扑网络。

2. 探视深渊：核心数据结构剖析

为了彻底弄懂这个体系，我们需要查阅 pathnodes.h 这个统御着无数路径信息的头文件，去看看这些结构体在 C 语言层面真实的定义样貌。

2.1 统御母体：EquivalenceClass

它相当于等价操作的总容器，掌控并指引着优化器的判断大纲。

typedef struct EquivalenceClass
{
    NodeTag     type;
    List       *ec_opfamilies;   /* 极重要的地标：这个等价类所绑定的 B-Tree 操作符族。*/

    Oid         ec_collation;    /* 字符排序规则 Collation (用于解决法语排序和中文排序等号判定不通用的基础问题) */
    List       *ec_members;      /* EquivalenceMember 列表：里面存放着属于这个家族的所有表列、常量及子计算 */
    List       *ec_sources;      /* 回源指针：生成这个大图论集合最初的、最“原汁原味”的 RestrictInfo 谓词本身 */
    List       *ec_derives;      /* 这是由主引擎向外辐射衍生成的用于各级 Join 匹配的约束条件包 */
    Relids      ec_relids;       /* BitMap：这个集合内所有的属性涵盖跨越了哪些具体的表 (RTE) */
    bool        ec_has_const;    /* 魔神之眼：哪怕有上万个子表达式，只要这个集合里混入了一个常量值，它就变成了 true！ */
    bool        ec_has_volatile; /* 是否存在易失的 (Volatile) 不稳定函数成员？如果是，则拒绝下发相关合并逻辑 */
    /* ... operations info ... */
} EquivalenceClass;

2.2 下位个体：EquivalenceMember

包裹在这个大家庭里的就是一个个具体的运算因子。什么样的节点能成为成员？通常是普通的 Var (代表表里的某一列)，也可能是完整的一个函数运算式 FuncExpr，亦或是 RelabelType（类型强转映射操作封装等）。

typedef struct EquivalenceMember
{
    NodeTag     type;
    Expr       *em_expr;         /* 这个成员实际对应的可执行表达式结构树，比如 Var(T1.A) */
    Relids      em_relids;       /* 该独立表达式依赖着哪些底层级表 (这对于推断 Join 的上下级顺序异常关键) */
    Relids      em_nullable_relids;
    bool        em_is_const;     /* 核心标志位：这个个体不是普通的列，它是一个带着具体化常量的字面值吗？ */
    bool        em_is_child;     /* 对于分区表操作尤其重要，用来指代基于分区拆分后的细分子节点 */
    Oid         em_datatype;     /* 该表达式求出该值的底层数据类型，用于强制校验匹配 */
} EquivalenceMember;

2.3 ec_has_const 与常量下推的碾压式特权

所有介绍 SQL 重写技巧和优化的教科书中，最底层的能力源泉往往都依赖于对常量的处理机制。 在 PostgreSQL 的等价类体系里，如果某一个具体的 Member 被判定设置了 em_is_const = true（比方说，业务在一段冗长的 SQL 中写了 T1.A = 100，100 被解析为一个带有 const 恒态属性的成员）。那一瞬间，整个包裹它的巨大的 EquivalenceClass 就像是被真理光环彻底附魔了：顶层的 ec_has_const 标志位会被直接置为 true。

因为一旦优化引擎确认了集合 {A, B, C, D} 在底层算术流转中其实“毫无实际变量计算意义”，这群看似花里胡哨的变量名词，在物理世界全都被具象化且绝对等同于常量 100！ PG 在后序规划中无论走到哪一步的算子——无论是多路 Join 进行哈希探测、深层子查询往内圈下推、还是做非常底层的并发索引扫描。只要能读到这个等价类的引用指针，它都能将针对这些巨大字段之间的动态匹配消耗，瞬间点石成金般转换为“直接去磁盘上找寻常数值 100 的确切记录”。并强制给其他关联成员赋予这个常量参数。 这便是深究 Cost-Based Optimizer (代价优化器) 中为何总是引以为傲的最高维武器——基于关系映射传递的横向常量下推 (Constant Propagation & Pushdown) 技术中枢基座！

3. 隐式衍生约束的魔法 (Implied Equalities)

光是建立起一套等价类模型其实并没有用，真正让 PostgreSQL 从众多依靠死板规则化干预行事的传统关系型数据库引擎中脱颖而出大放异彩的，是深藏在这背后于 equivclass.c 里的魔法工坊： generate_implied_equalities() 等一系列生成衍生约束条件的函数调用。

3.1 跨表谓词下穿衍生机制

举一个非常具象的例子。假设有应用前端依靠 ORM 代码生成器胡乱拼凑出了这么一句常见的检索： SELECT * FROM T1, T2, T3 WHERE T1.id = T2.id AND T2.id = T3.id AND T1.id = 99;

试想一下，在执行计划刚刚起草、T2 和 T3 还处于独立构思如何单表扫描的早期阶段。如果是在传统的非闭包体系引擎中，优化器要去扫描拥有千万数量级的 T3 表时是非常痛苦无助的——因为它压根不知道 T3.id 到底要去搜寻谁。它除了默默去承受代价高昂的 SeqScan 顺序全表扫描硬盘以外毫无办法，然后必须把读出来的成百上千兆数据生硬地抛到上层参与后面缓慢如蜗牛的 Join 过滤匹配。

但在 PostgreSQL 强大的 generate_implied_equalities_for_column() 函数机制背书中。由于 T1.id, T2.id, T3.id 以及那个常量值 99 全部落网被收容进入了唯一的巨型全域等价类。当 query_planner 调度到底层需要评估扫描 T3 这张表的最佳访问路径 (Paths) 时，代码会拿着 T3 表所独有的 Relids 当作钥匙，去向这个巨型等价中心发问祈求： “请问在这里面，有没有哪怕仅能由我 T3 一张表就可以单独享受匹配、不需要往上走依靠关联运算的隐性过滤推演条件（Implied Restrict)？”

等价类模块通过遍历，敏锐发觉自身不仅有常量 99，并且还有请求者的 T3.id。于是系统这如同虚空捏造出了一把尚方宝剑——自动凭空变出了一条原本在文字层面并不存在、但是在数学上绝对站得住脚的确凿过滤条件：T3.id = 99！ 随后，这个崭新生成的过滤条件参数会被作为推导限制被即刻下发，绑定成为了 T3 自己的单表专属扫描谓词。原本 T3 即将遭遇的那一波漫长的百万行 SeqScan 以及在内存中同前面表死磕合并运算的恶劣命运随之烟消云散。这一动作直接将其成本评估定性为了超轻微代价量级：不仅可以走索引，而且只需要在极其短暂的时间内针孔定位完成 I/O 的顶级访问模式——Index Scan (T3.id = 99)。优化器提速狂飙的大幕在这个节骨眼上拉开。

3.2 强前置互验与阻止“笛卡尔”核爆

退一万步讲，哪怕在这个逻辑运算链条上我们完全没有看到常委级别的常量出现，仅仅只是关联起了跨表的字段集合 {T1.A, T2.B, T3.C}。当优化器因为表的数据量选择率等等复杂原因估量，觉得有必要先将 T1 和 T3 来一场连接大作战执行 T1 JOIN T3 时（例如系统估算出即使盲连，两边返回的表行数也非常极小）。哪怕你回头审视那份极端的业务传入的原始 SQL 文本，压根根本找不到那俩表哪怕有丝毫一星半点的字段关联语句。

但神圣而强大的 PostgreSQL EC 抽象集合模块仍会一视同仁。由于它坐拥此三子于一室，明知三方连通有着等同的强逻辑背书，系统便会极其从容地通过派发操作，生成出一条隐性联姻线缆：创造出 T1.A = T3.C，并将其强行施加这二者初次结合的连接判断标准（Join Qual）！

这套机制几乎完美且不可破防地从代码根本设计上截断化解了一场原本必然演变为笛卡尔全铺积惨剧的糟糕连接爆炸隐患。当我们从这个宏大的视角去凝视晦涩生僻的等价类 (EC)，它存在的真正意义从不光是进行简单语法替换，它更是为了阻滞底层检索灾难、铺设规划指引路径和用以空间降维的最坚实的守护神与路标塔。

4. 与序列流向 PathKeys 的极高依附羁绊

如果我们想要精通于理解 PostgreSQL 内核反应堆里最高深的代计算法则（Cost 和代价比拼），就必须得先理顺困扰着最多数新手研究员的那条最生硬的死结： 判断或者指代某一条执行记录长河是否在这个阶段内部已经恰巧满足了你要求的排序规则配置时，它的终极判断信标（Beacon），永远不是去简单僵冷地对比诸如所谓的类似 ORDER BY Table1.TName 等几个简单的字符串名，它的底层引信是远远地锚定、并且绑定包裹着一整颗等价类！(指向 EquivalenceClass 的母体内存地址)。

这段千丝万缕的核心羁绊将在专门用来剖析排序内核的《PathKeys》深度核心篇章中大规模展开，但鉴于它们二者生死的关联性我们必须要在本节作一浅尝揭开。

为什么在物理的数据结构设计抽象里头，PathKey 链表的这个核心必须紧紧夹杂挂钩 EC 的指针向标而不是直接对应上一个什么 Var(A) 的变量位呢？ 假设现在系统在堆排查探的过程构建了一个包含 {t1.A, t2.B} 的局部 EC 家族。 目前，底层引擎派出了一个得力干将老老实实地从磁盘上沿着建立在 t1 的 B 树索引走了一轮查找（IndexScan）。由于检索特性沿着那棵树一直扫出来，这串喷涌而出的结果集先天在列 t1.A 上就是顺畅递增且有条理的了。这时刻这个路径会对外吐露它带有光环的一份证明报告册：它的 pathkeys 数据包说明了：“注意我手头这堆数据可是完美臣服并且符合那个极其著名的 EquivalenceClass{t1.A, t2.B...} 流向哦” 。

执行计划层层往上汇报。然而在终点站查询的最外层顶端遇到了一个极高的过滤巨石：你的主逻辑命令强要求系统在这儿汇总做最终结算排序：ORDER BY t2.B。 这会子，ORDER BY 这个引擎管控模组可是非常贪婪也极度渴望地去向查询主决策模块打听索要：“从下面那些底层兄弟一路干活抛给我的结果流里面，有没有哪一个特别机灵的小伙伴，它那条数据线的供货排向老早以前就按我想要求看的 t2.B 顺顺利利的整理分配过给大伙儿了？”

工程代码在此展现出了神迹。底层规划器绝不会死板僵硬要求查核里面是否真就包含了 t2.B 这一串字母的完全一致打标（字符串硬比对在这个档口铁定会导致出现“我底下是用按照 A 来规整出来的，这上边死命令说要核实 B 的顺位，风牛流马不相及，统统当作废处理强行大乱再重新大把内存全部重排梳理一遍序列！”的惨烈悲剧和性能严重错判）。而是拿出从底下沿着流程传向自己的“这个状态集归档的内存地址引用指针”，去跟自己上游规划里的渴求期望目标硬顶比拼：比对上了那个同样身为这个族长神圣统御者也就是同等价阶位的 EquivalenceClass 实体指向指针！

一旦引擎探知下面底层那些数据特征关联着的 EC，竟然 及其巧妙合拍地等同于自己所需要拥有的那个实体指针！这两套体系简直是宛如雷击瞬间拍板互相对上眼：“那是因为在早前一开始 SQL 过筛阶段我们全伙计就了解了 t1.A = t2.B 等式的永恒不变关系。我此刻验收到下方提交上的这份文件其实已经完美踩着贴着 t1.A 是顺位排列整理的结果了。如此一来从物理计算普遍真理的原则上来谈，在当下这一瞬间这流体、这份顺序态，早已就绝对并且纯天然地履行等同遵从了我亟需最迫切想要见到的 t2.B 排好顺序的大前提啦！”

这就意味着，只要它在底下随手捡便宜借用了在一个小小列名 A 身上一条微不足道低消费的索引遍历跑法，竟然可以通过一路往上通过跨阶层的核心概念概念转移包裹魔法替换法，神乎其技的省跨越了大表和执行节点之间的巨大鸿沟。顺带来连在最尾端呈现全部表格输出的时候连那些最庞大要吃掉好几个 G 运存排序的超巨型的全排列重排内存杀手节点都能省得一去不回！这种在 RDBMS 执行领域宛如神出鬼没般的维度换挡置换的空间转移技巧。对现在那些随便做几个粗暴生硬匹配依赖玩具解析层去优化的粗制型引擎看来，近乎如同不可跨越、绝不明白的异星位面技术级打击。这代表着一代代 RDBMS 先驱设计大师几十年代码打磨结晶下来的算法思想极意巅峰——也是使得它这只大象至今立于不败之地无可替代的关键壁垒内核深渊所在。

路径序列与排序核心密码 (PathKeys)

几乎所有关于性能优化的面试题，最后都会绕回一个命题：“为什么加了索引，ORDER BY 或 GROUP BY 变快了？” 在执行引擎的表象逻辑看来似乎是常理，但如果要在拥有极大复杂组合的 PostgreSQL 代价优化内核（CBO）中完美统一地实现这一机制，就需要一种能够统一度量并且伴随路径推演任意流转的通用通货。

在 PG 18.1 中，这套跨越所有的关系代数操作、用来锚定和流转排序特征通货，名叫 PathKey (路径键)。相关代码的核心座落于 src/include/nodes/pathnodes.h 和 src/backend/optimizer/path/pathkeys.c。 不懂 PathKeys 的设计与传播机理，你就无法真正看懂优化器在 add_path 里做出的那些惊心动魄的剪枝取舍。为何某些代价极高的“破烂”执行计划，反而会在复杂查询的最后关头被 PG 奉为至宝？

1. 结构起底：什么是 PathKey？

优化器在评估物理路径（比如一次走索引的扫描、或者哈希连接）时，不仅看“花了多大 Cost”、“能吐多少 Rows”，还必须精确标注出这股数据流携带的元素排序特征。

这套特征在源码中由 List *pathkeys（链表）承载。其基础粒子组件 PathKey 如下：

typedef struct PathKey
{
    NodeTag     type;
    EquivalenceClass *pk_eclass;   /* 极其震撼：它不指向具体的列，而是指向一个覆盖全部域的等价类！ */
    Oid         pk_opfamily;       /* 排序操作符族 (如 B-tree opfamily OID)，指定排序规则 */
    int         pk_strategy;       /* 升序 ASC 还是 降序 DESC */
    bool        pk_nulls_first;    /* NULL 值是悬浮最前还是垫底 */
} PathKey;

1.1 为什么主体必须包裹 EquivalenceClass？

在上一个附录《等价类 (EC)》中埋下过伏笔：PG 极度高超的设计使得 PathKey 的最核心属性 pk_eclass 直接硬链接在了 EquivalenceClass 上。

试想场景：SELECT t2.info FROM t1 JOIN t2 ON t1.id = t2.uid ORDER BY t2.uid; 在优化器构建最底层时，系统发现 t1 表的 id 有 B-Tree 索引。于是探索出极速路径 IndexScan(t1)。这条底层路生成的初始 pathkeys 护照上骄傲地写着：“我送出的源头数据是严格基于 EquivalenceClass{t1.id, t2.uid} 排序的！”

当数据流抵达上层、需要执行全局决算的 ORDER BY t2.uid 时： 假若当年仅仅用字符串列名映射作为载体（如：pk_var = t1.id），那么顶层节点拿起客户要求的 t2.uid 一碰：“天啊，底层是 t1.id，客官要的是 t2.uid，给我退回！” 于是迫使几百万数据强行扔进 Sort 内存重排栈，引发算力海啸。

但是，正因为 PG 的 PathKey 绑死的是极度抽象化的等价类！ 顶端审判官只需拿 t2.uid 代表的 EC 句柄，和底层的那个 EC 指针一碰。刹那间同气连枝完美吻合 (那正是因为 t1.id = t2.uid，起初就被合二为一)。判定官放行，极其平滑地帮引擎省切掉了最昂贵的重排局！

2. 需求侧大一统：规划器的排序货币化

在嵌套复杂的 SQL 分析大河中，混杂了无数试图干预排序的操作符：

  • 终端发出了 ORDER BY a, b 要求顺位。
  • 聚集指令下了 GROUP BY c, d 在启动 GroupAgg 前要求聚类有序。
  • 过滤节点加了 SELECT DISTINCT e 企图通过排序来实现滑动去重。
  • 大名鼎鼎的 Merge Join 强迫两营输入流按等同键绝对序列接壤。

对于老手工作坊式引擎，可能需要为以上所有情况各自手编一套“排序对账单”。 但在 PG 那神级铺陈的 pathkeys.c 架构中，所有这种排序请求，无论初衷为何，都会在预检阶段被诸如 make_pathkeys_for_sortclauses 的函数接管，彻彻底底等价兑换为唯一标准流通物——PathKey List 链表。

自大优化网络全面铺开的那一顷刻，优化器这个中央智脑，再也听不懂所谓的 ORDER BY 或 GROUP BY。它在判断下层分支能否直接插接时，唯一遵循的就是查验 pathkeys 国书。

3. 供给端长河：PathKeys 的诞生、传承与覆灭

面对这条流转下的继承物 pathkeys，不同物理计划节点态度截然不同：

3.1 源氏的血脉：底层索引扫描

最纯粹的 pathkeys 孕育自 B-Tree。定位到 indxpath.c 内部的 build_index_pathkeys： 当检索刺入索引结构时，PG 产生出一个拼接的衔接 PathKey 队列，赋予在即将上抛的 IndexPath 头上。如果是走路线粗犷的 SeqScan，生成的归口登记簿对于 pathkeys 则是一个干脆的 NIL (完全空指针)，宣告这是一堆无序的散沙。

3.2 屠夫：Hash Join 的破壁

当你汇入双表关联： 性能王者 Hash Join 的作风极其粗野。它压根不顾及底层是不是辛苦排好序的。它的底盘是将小表倒进哈希池开桶，随即拿大数据集激化碰撞。碰撞出水瞬间，所有的历史排序规律全盘崩散。 因此任何基于 hash_inner_and_outer 衍生的路径，它的光环继承簿 path->pathkeys 必定被无情洗白清零成为 NIL。

3.3 恪礼的承传者：NestLoop Join

嵌套循环联合 (NestLoop) 恪守着外表驱动一再往复过滤匹配的准则。 外环本有的序列起伏节奏，在吸纳内表后竟然绝妙地没有丝毫偏差！因此优化器会温柔地调用 build_join_pathkeys，把外表（Outer Rel）持有的金字令箭原封不动、极其虔诚完整地誊抄接手下来！

4. 终界审判：修剪免死金牌 (Path Pruning)

这也是隐藏于代价运算机制底下最为反转的绝魂之跃。 前文提过：add_path() 是一座极度现实的达尔文“裁决法庭”。如果不带排序加成，一条起步慢、总价高昂的路，百分百会被当场遗弃清盘！

但是！ PathKeys 的设计存在，作为优化器世界的最高精神免死金牌，超降维介入干涉了丛林淘汰法则！

我们在极其严密的 Cost 较量之前，会看到如圣言般的预演过滤：pathkeys_contained_in(path1->pathkeys, path2->pathkeys)。

一出震撼的剧本模拟：

  • 路径 A 候选：采用了一套笨重到令人发指的 SeqScan 自己全强制做重排序，开机代价大、总跑算价高昂至上千万！但自带了飘扬顺气的 pathkeys = (t1.uid) 序列大旗。
  • 路径 B：一条能让算力尖叫绝世闪电的 Hash Join 超短通。起步快，全跑完不过区区大几千的消耗！但其身如恶兽全无秩序，pathkeys == NIL。

在没有神圣豁免的高维排序非闭包界中，这路 A 小丑绝必立刻暴死；新贵 B 将扫平一切！ 但！在 PG 那深不可测的总决策眼界里，如果裁断虽然感觉 A 烂如破泥，但是它顶头上那副全维的列旗却死超囊括了那光秃和残旧的 B 所全不配所有的法度红利。 哪怕 A 的纸面耗费远高千倍估算成本线，PG 的主引擎手此时都会赫然插进力保：活拉下这面破 A 强行封卷！并永作存档列留进那最重要的预备大底座。

最高中枢为何要苦保下流之列？ 因为中枢老辣深沉，在那远不见底的高顶组装关隘，它可能遭遇十重极度变态的 Merge Join 强令，更甚至撞见最冷酷杀神终极底符 —— LIMIT 1。

在极端灭顶的洪荒恶劣要求下。那个自持着武功第一极速猛飙全图却无次序的 Hash Join，为了强硬生拼全图结果以抗排期对局，甚至会被逼到去开启巨型外部破落磁盘暂存归排而令总价飙出百合千万计而炸穿。 反观那被上帝强保下这带好顺行秩序之破胎分支 A！在瞬满交接关斩停下 LIMIT 1 诉求那一刻。凭那极其超前精准拿对序列之完美契合并极速掐流神力直接砍停，瞬间封神！ 通过此一层维向特比的神来保护局体系法道，奠基于 PostgreSQL 历经数十载依然傲立于星空云端王者之无上统治地位！

选择率与代价引擎的数学基座 (Selectivity & Cost Model)

如果说执行器节点 (Executor Nodes) 是在战场上冲锋陷阵的铁血部队，那么 PostgreSQL 核心的 代价基调引擎 (Cost-Based Optimizer, CBO) 就是在这之前于作战统帅部里进行图上千百次沙盘推演的灵魂中枢。 它一切的生杀予夺衡量指北，全部依托于两个底层基石机制：选择率预估 (Selectivity) 猜测系统会返回多少行数据，以及 代价计量 (Cost Model) 折算出这批数据在磁盘和内存中将会耗费多少系统滴答。

在 PG 18.1 中，这套左右引擎命脉的公式算法与核心常数，沉睡于 src/backend/optimizer/path/costsize.c 与 src/backend/utils/adt/selfuncs.c。 本文将穿越表象，带你深入这个堪称统计学与经验法相互融合的魔幻中枢。

1. 统计学透视：选择率 (Selectivity) 的精准测写

所谓 Selectivity (简写为 sel)，是一个介于 0.0 到 1.0 之间的浮点数，它表示绝命的“卡口率”，允许全表百分之多少记录透射过去。 计算 sel 的一切前提，都基建于 ANALYZE 命令收集并写入到 pg_statistic 表中的系统元数据。

1.1 单列特征：MCV 与直方图的双峰分布

现实世界业务库的数据通常符合长尾和聚集极化（如某平台大多数用户年龄 25 左右，夹杂极少的 18 和 50）。 在 selfuncs.c 里处理操作符如 col = 'A' 时：

  • 系统首先去查目标列的 MCV (Most Common Values) 高频词表数组。如果查到 'A' 就在此处，会直接精准地抽出记录好的频率（可能占 32%）作为 sel。
  • 如果是 col > 100 这样的带域检索，MCV 就捉襟见肘了。系统转而启用 Histogram Bounds (直方图切面界限)。PG 将排序好的表数据强行切分为数百个均等记录数的“桶 (Bucket)”。通过线性插值法探测 100 横跨了几个桶，积分计算精确 sel。
  • 最后还要叠加一个 Null Fraction (空值率)，直接扣掉 NULL 的占比作为基础衰减。

1.2 维度的灾难与破局：扩展统计信息 (Extended Statistics)

在代码侧集中于 clauselist_selectivity() 的连乘会带来缺陷盲区。 比如用户写 WHERE city = 'Beijing' AND country = 'China'。传统的概率计算遵循特征独立假设，默认这两个条件不相关。 查表得知：city='Beijing' 的 sel = 0.05，而 country='China' 的 sel = 0.2。 如果引擎采用朴素的 sel_total = 0.05 * 0.2 = 0.01 瞎乘算法，就会得出极其扭曲的结果，以至于被诱导选出孱弱错误计划分支。殊不知北京必定属于中国这二者拥有强绑定聚集概率！

为了挽救这种维度塌缩，PostgreSQL 引入了跨列的 CREATE STATISTICS 模块救场。当你为 (city, country) 建立扩展高维相关性依赖 (Dependencies) 后。系统就会在连乘前判定联动关联，打破“相互独立”的连乘魔咒，挽救底层评估树的坍塌偏斜。

2. 代价模型 (Cost Model) 的双轨评议机制

如果你查看 EXPLAIN 输出，永远会面对类似 cost=10.00..1050.25 两个用下划线相连接的神奇数值。 这代表着优化器世界基于双轨维度的账面计算法：启动代价 (Startup Cost) 与 总代价 (Total Cost)。

2.1 平行维度的角力

由于不同特征压榨，PG 对任何一条候选 Path 都在这两个账本上同时算分。

  • Startup Cost (初始启动沉没代价)：吐出第一行目标数据前，算子需要憋气阻塞筹备花去的时钟值。 比如 SeqScan 启动代价为 0。但 HashJoin 发车极长！你外表得等内表慢吞吞把几百万记录翻完并在内存建好桶池完毕后，才能下发外表探针撞线，因此启动代价巨大。

  • Total Cost (总代价)：完完整整把所有数据发完、关闭管口所需要的总耗。 SeqScan 哪怕只要一条也是全盘扫去碰，总价极高。

双轨博弈在于，如果上级传进来了 LIMIT 1。代价函数在汇总结算最优解时，权柄会疯狂倾向于低 Startup Cost 的算子。即使它的 Total 极难看也会果断采纳，以此规避诸如大排序大哈希表建造的愚蠢行径。

2.2 操控大局的因子 (Magic Numbers)

一切 Cost 都是计算浮点值。放缩的源头，来自 costsize.c 中被写死并允许重置的 GUC 常量参数，这是所有高端 DBA 进行魔改调参的圣地。

  • seq_page_cost = 1.0 (默认)：顺序读取一块 8KB 磁盘耗时被定义为基本黄金单位 1.0。
  • random_page_cost = 4.0 (默认)：标志发生磁头跳步的随机磁盘检索，因寻道通常比顺读大 4 倍以上。这就是为什么明明有索引，系统却嫌弃随机读太分散而果断放弃走全表扫的终极阀门！在全 SSD 时代，高玩通常会将其砍至 1.1 逼迫系统疯狂走查索引。
  • cpu_tuple_cost = 0.01 与 cpu_operator_cost = 0.0025：标志 CPU 脑运算一条 WHERE 等内存操作相比于捞磁盘（耗费 1.0）有多么轻如鸿毛。

3. 代价决断前线：核心算子估算探秘

在知悉参数之后，去 costsize.c 看看具体算法究竟是如何拨账的。

3.1 扫表的代价：cost_seqscan vs cost_indexscan

这是每天被系统狂暴触发千万次的基石结算。

  • 对于全表扫 (cost_seqscan)： 公式直白粗暴：总代价 = (目标表总 Page 数 * seq_page_cost) + (行数 * cpu_tuple_cost) + ...

  • 对于索引扫 (cost_indexscan)： 这里极其反人类。索引树一旦跑起来去拿原据，是非常恐怖离散化的跳跃。 如果表物理存放长期因增删改导致碎片四散（Correlation 相关性 趋近 0），只管强走索引拿十万记录，那就是在掀起十万昂贵物理跳变。random_page_cost 能把算盘直接敲出几千万成本。 但此时代码绝不死板，cost_index 中潜藏着平滑预测公式曲线，去估测那些已存于缓存池内的块并猛力往下折压分数，以免走入纯全栈封禁绝路。

3.2 兵家之要的关联大决斗代价

  • Nested Loop Join (cost_nestloop)： 计算方式无情恐怖：让 左表循环总数据 去硬全额连乘积 右内层单次查代！ 外层十万条，里层只是绝底的极小花费 50。连加也是巨震之价。但这全赖其 Startup 近 0 的绝杀奇点令它能稳抗高压存活。

  • Hash Join (cost_hashjoin)： 深沉点聚焦在于最开端 Startup 构建期。 算法要计算内表有多聚盘、砸多大 CPU cpu_operator_cost 去装填至工作内存 work_mem。 要命的极其分差于此，一旦系统查获了 work_mem 给得小器容不下全漏。算力会在极其平纯原加里生添一只消耗命极高的罚价：disk spill penalty（刷盘巨大惩罚开销）！让多盘狂读拉起极高 Total 总耗。向最上层示警，从而逼向退避并触发它线连接路全保生天。

经过成百上千次精密节点评估积累，PG 优化的王牌引擎得以从数百纠缠网里，斩钉截铁不差分毫断那向绝通往之绝胜黄金大通道长路。
