优化器概览

本章基于 PostgreSQL 源码中的 src/backend/optimizer/README，简要介绍查询优化器（Query Optimizer）的整体工作流程和核心函数调用链。

优化器的主要任务是接收查询树（Query Tree），并生成最优的执行计划（Plan）。

核心调用流程

PostgreSQL 优化器的入口函数位于 src/backend/optimizer/plan/planner.c。

整个规划过程大致可以分为以下几个层级：

graph TD
    A[planner] --> B[standard_planner]
    B --> C[subquery_planner]
    C --> D[grouping_planner]
    D --> E[query_planner]
    E --> F[make_one_rel]
    F --> G[make_rel_from_joinlist]
    D --> H[create_plan]

1. planner()

这是优化器的全局入口点。它的主要职责非常简单：

  • 如果配置了钩子函数（planner_hook），则调用钩子。
  • 否则，调用默认的优化器实现 standard_planner()。

2. standard_planner()

这是标准的优化策略入口。它的工作包括：

  • 配置设置：设置由于 CURSOR 或其他上下文导致的相关标志。
  • 预处理：调用 subquery_planner 来处理顶层查询。
  • 计划清理：生成的计划树可能包含一些不需要的节点，在这里进行最终清理。

3. subquery_planner()

这个函数负责处理一个特定的查询层级（Query Level）。对于包含子查询的 SQL，这个函数会被递归调用。

  • 重写与预处理：处理 ANY/ALL 子链接，提升子查询（Pull-up sublinks/subqueries）。
  • 调用下层：一旦查询树被扁平化或重写，它会调用 grouping_planner 来进行实际的规划。

4. grouping_planner()

这个函数是“全能选手”，负责处理：

  • 非 Join 部分：处理 GROUP BY、HAVING、WINDOW functions、DISTINCT 和 ORDER BY。
  • 它首先调用 query_planner 来计算最基本的“扫描/连接”（Scan/Join）路径。
  • 然后，它在 query_planner 返回的路径基础上，添加聚合（Aggregation）、排序（Sorting）和分组节点。

5. query_planner()

这是核心中的核心，专注于 Scan/Join 优化。

  • 基本关系处理：为查询中的每个表（Base Relation）生成扫描路径（Seq Scan, Index Scan 等）。
  • 连接搜索：调用 make_one_rel() 来搜索最优的连接顺序。它使用动态规划（Dynamic Programming）或遗传算法（GEQO）来组合这些表，直到所有表都被连接成一个最终的关系。

主要阶段总结

根据 standard_planner 的执行逻辑，我们可以将优化过程概括为以下步骤：

  1. 预处理 (Preprocessing):

    • 重写子查询，简化表达式（eval_const_expressions）。
    • 消除冗余的外连接（Outer Join Elimination）。
    • 分发约束条件（Qualifiers）。

  2. 扫描/连接规划 (Scan/Join Planning):

    • 为基表生成访问路径。
    • 计算两表连接、三表连接…直到生成包含所有表的连接路径。
    • 这部分主要由 query_planner 和 make_one_rel 完成。

  3. 后置处理 (Post-processing):

    • 在连接结果之上，添加 Grouping、Aggregation、Windowing 和 Sorting 节点。
    • 将最优的 Path 树转换为最终的 Plan 树（create_plan）。

查询树的预处理

在 PostgreSQL 优化器（Planner）接收到解析和重写后的查询树（Query Tree）以后，第一步并非直接开始路径搜索，而是通过 subquery_planner() 函数进行一系列复杂的预处理。预处理的核心目标是消除查询中的冗余结构、展开嵌套，并对表达式进行规范化和简化，从而为后续的 grouping_planner() 和 query_planner() 减轻负担，提供尽可能扁平化和易于优化的结构。

以下内容基于 PostgreSQL 18.1 源码（src/backend/optimizer/plan/planner.c 中的 subquery_planner）详细解析查询树预处理的各个核心阶段。

1. CTE 与底层结构展开

在真正进入表达式级别的预处理之前，优化器必须先把从属的查询块或语法结构给“铺平”。

  • 处理 WITH 语句 (CTE): 调用 SS_process_ctes。如果 CTE 可以被内联（Inline），则将其展开为 RTE_SUBQUERY；否则构建出 InitPlan 级别的 SubPlan，以便在主查询执行前计算完毕。
  • 补全空 FROM 子句: 通过 replace_empty_jointree。如果查询没有 FROM（如 SELECT 1），优化器会人为挂载一个 Dummy 的 RTE_RESULT 节点，统一后续的 JoinTree 处理逻辑。
  • SubLink 的显式提升 (ANY/EXISTS): 调用 pull_up_sublinks。将 WHERE 或者 JOIN/ON 里面涉及 ANY / EXISTS 的相关子查询（SubLinks）尽早转化为半连接（Semi-Join）或反连接（Anti-Join）。
  • 子查询扁平化 (Subquery Pull-up): 调用 pull_up_subqueries。如果 FROM 子句中直接写了普通的简单子查询，系统会尝试将其合并到当前查询层级，消除不必要的嵌套。
  • UNION ALL 的扁平化: 遇到简单的 UNION ALL，调用 flatten_simple_union_all 将其重写为并列的追加关系 appendrel，避免复杂的集合操作开销。

2. RTE (RangeTableEntry) 预处理

RangeTable 维护了查询涉及的所有表、视图、函数。

  • 函数 RTE 内联与常量化: preprocess_function_rtes 针对 FROM 子句中的函数进行计算。如果是简单的常数化函数，直接得出结果；如果有机会，则像展开视图一样将函数展开（Inline）。
  • 虚拟生成列展开: expand_virtual_generated_columns 会扫描所有的表，将查询内涉及到 Generated Columns 的 Var 引用直接替换为其背后的计算表达式，方便一并在后续步骤被折叠。

3. 表达式预处理阶段 (preprocess_expression)

这是预处理代码块中占篇幅最大、也最为细腻的一环。系统不仅要处理 WHERE 里的条件，还要将 TargetList、HAVING、LIMIT 及 ON CONFLICT 等各处的表达式利用 preprocess_expression() 统统洗一遍。核心动作包括：

3.1 连接别名展开 (Join Alias Variables Flattening)

通过 flatten_join_alias_vars，优化器将由于 JOIN ... USING 或自然连接引发的连接别名变量，替换为来自底层基表的原始列（Var 引用）。这确保后续的优化只需要面对最底层的真实列名。

3.2 常量折叠 (Constant Folding)

调用核心函数 eval_const_expressions。 不仅会将 1 + 1 化简为 2，还能对各种内部函数的常量输入进行预计算（前提是函数为 Immutable）。重要的是，它会压平嵌套的 AND/OR（将 AND (A, AND(B, C)) 变成 AND(A, B, C)）以保持逻辑树平坦。

3.3 条件标准化 (Canonicalize Qual)

利用 canonicalize_qual 函数，如果表达式是一个布尔条件（如 WHERE 里的项），它会尝试做一定的范式转化（比如尽力将其拉平或者排查明显的互斥矛盾）。

3.4 剩余 SubLink 转换与外层变量绑定

  • 经过 pull_up_sublinks 还没解决的杂项子链接（如出现在 SELECT 目标列里的标量子查询），在此被包装成 SubPlan。
  • SS_replace_correlation_vars 会找出对上层查询列的引用，并使用 Param（参数）进行绑定，使得父子查询产生实质的数据关联参数。
  • 最后对于 WHERE / HAVING 的顶层条件，调用 make_ands_implicit 退化为 Implicit-AND 的 List 结构存放，不再采用显式的 BoolExpr。

4. 后期合并与结构修正

经过表达式洗刷后，subquery_planner 在交付给 grouping_planner 之前会做最后的清理。

  • HAVING 下推为 WHERE: 这是经典的优化手段。系统会扫描 HAVING 子句，寻找那些不包含聚合函数、不包含易变函数（Volatile Function）并且不需要等到空分组集产生结果的过滤条件（例如 HAVING col = 1）。这些被认为是退化的聚合后过滤，系统会将其克隆或移动到 WHERE 列表中，从而在进入 Aggregate 算子之前尽早淘汰无关数据。
  • 外连接消除 (Outer Join Elimination): 调用 reduce_outer_joins。因为上一步各种 WHERE 表达式已经被处理，这一步系统会检查如果存在 WHERE 强约束要求某空侧列非空（比如 LEFT JOIN B ON ... WHERE B.id > 10），此时原本的 LEFT JOIN 已经失去保留 null 侧的意义，该外连接会被安全地降级成 INNER JOIN。
  • 剔除无用的 Result RTE: 调用 remove_useless_result_rtes。前面补充的或中间产生的不再起实质作用的 Dummy 关联在此被扫地出门。

至此，一个极其干净、扁平化且逻辑清晰的等价 Query Tree，才会正式交给 grouping_planner。

子查询提升 (Subquery Pull-up)

子查询提升（Subquery Pull-up）是 PostgreSQL 优化器中最为核心的逻辑重写优化机制之一。它发生在规划器的预处理早期阶段。其根本目标是**“打破查询边界，扁平化查询树”**，使得父子查询能够融合在一起，从而大幅度扩展 Join Order（连接顺序）的搜索空间，并且减少执行层的嵌套开销。

1. 背景与目标：为何要提升？

如果不进行提升，嵌套在 FROM 或者是 WHERE 子句中的子查询，在执行树上就会变成单独的 SubqueryScan 节点或是 SubPlan 操作（InitPlan/SubPlan）。 这限制了优化器的能力：

  • 黑盒隔离：父查询的表无法与子查询内部的表混合进行 Join Order 规划。
  • 运行时开销：执行引擎在运算时需要频繁穿越执行计划的不同层级进行上下文切换或独立求解。

因此，PostgreSQL 规划器在 src/backend/optimizer/prep/prepjointree.c 中，专门设计了两大核心路径来拆解各类“提升”。

2. SubLink 提升：pull_up_sublinks

SubLink 通常指的是出现在 WHERE 或者 JOIN/ON 过滤条件中的子查询结构。 优化器在 subquery_planner 一开始就会调用 pull_up_sublinks()。

其核心递归函数 pull_up_sublinks_qual_recurse 会遍历条件，并根据 SubLink 的类型采取不同的提升策略。这依赖于几个关键转换函数：

2.1 ANY SubLink (convert_ANY_sublink_to_join)

当遇到类似 WHERE x.id = ANY (SELECT id FROM y)（即 IN (subquery)）时：

  • 优化器会将其从 WHERE 条件中剥离出来，转化成与父查询的一张表的 JOIN_SEMI (半连接)。
  • 对于常数数组 IN (1, 2, 3)，也会在此阶段通过 convert_VALUES_to_ANY 做尝试折叠化简。

2.2 EXISTS SubLink (convert_EXISTS_sublink_to_join)

当遇到 WHERE EXISTS (SELECT 1 FROM y WHERE x.id = y.id) 时：

  • 本质与 ANY 类似，这会被转化为一个 JOIN_SEMI (半连接)。
  • 但是它进一步处理了带有 NOT 修饰的情况：如果是 WHERE NOT EXISTS ( ... )，在 PG 中会被直接转化为 JOIN_ANTI (反连接)。这就允许优化器使用 Hash Anti-Join 等高效物理算子直接过滤数据。

3. Subquery 提升：pull_up_subqueries

区别于 SubLink，一般的 Subquery 主要指代的是出现在 FROM 子句里面的表源定义（如 SELECT * FROM a JOIN (SELECT * FROM b) c ON ...）。 当处理完 SubLink 之后，planner 接着调用 pull_up_subqueries()，尝试将 FROM 下拉取出来的子查询给融会贯通。

3.1 简单子查询 (is_simple_subquery / pull_up_simple_subquery)

怎样才叫一个“简单（Simple）”的子查询？

  • 它不能包含 GROUP BY、HAVING。
  • 不能包含 LIMIT / OFFSET。
  • 不能包含集合操作（UNION、INTERSECT、EXCEPT）。
  • 不能包含顶层聚合函数或是窗口函数。

如果是简单子查询，pull_up_simple_subquery() 将大显身手：

  1. 它将子查询的 FromExpr 的儿子全部并入到当前层级的 FromExpr 树里。
  2. 剥除该子查询外层包装。
  3. 利用 pullup_replace_vars 构建或替换那些需要保持上下文独立性的变量引用（如果在 Grouping Set 里，有时还要为其包装一层 PlaceHolderVar）。

3.2 简单 UNION ALL 的扁平化 (pull_up_simple_union_all)

对于 FROM (SELECT x FROM a UNION ALL SELECT x FROM b) 这样的集合操作： 如果没有任何排序 ORDER BY、没有 LIMIT，并且是 UNION ALL （不用去重），优化器会调用 is_simple_union_all。 它会把这颗 UNION ALL 子树彻底压平，转化为一系列的追加关系 (AppendRelInfo)。这就将集合操作融化成底层扫描层级的简单 Append，而不用执行繁重的 Subquery 拼接。

3.3 LATERAL 子查询的修补

在 PostgreSQL 支持了 LATERAL 之后，提取时还涉及 LATERAL 跨层依赖参数。 如果发生 pull_up_subquery 且原本有 LATERAL 引用，代码会通过 IncrementVarSublevelsUp 等复杂的方法调整 Vars 的层级深度（Sublevel），并借由 CombineRangeTables 重组合并底层的 RangeTable 中记录被依赖的状态。这一机制对关联子查询能够被最终压平至关重要。

外连接消除与无用连接移除 (Outer Join Elimination & Join Removal)

在经过了表达式级别的折叠与扁平化之后，PostgreSQL 优化器会尝试通过约束分析，尽可能精简查询树中的连接节点（Join Nodes）。精简的连接树不但执行效率高，同时也给了后续的动态规划/遗传算法更多排列组合的选择。

这一步主要包含了两个阶段的优化：

  1. 外连接降级/消除 (reduce_outer_joins)
  2. 无用连接移除 (remove_useless_joins)

1. 外连接降级的原理与意义 (reduce_outer_joins)

源码位置：src/backend/optimizer/prep/prepjointree.c 调用时机：它是由 subquery_planner 函数中调用的。

外连接（Outer Join）通常会强制优化器遵循特定的连接顺序，因为 Nullable 一侧必须在关联后才能保留。reduce_outer_joins 的目的就是检查是否存在某些强有力的 WHERE 条件，使得原本的 Outer Join 可以退化为 Inner Join 甚至是 Anti Join。

1.1 两遍扫描机制 (Pass 1 & Pass 2)

为了减少不必要的递归，reduce_outer_joins 使用两阶段处理：

  • Pass 1 (reduce_outer_joins_pass1): 收集查询树底层的 Relids（涉及了哪些基础表），并快速标记哪些子树中真正包含外连接（contains_outer）。如果没有外连接，直接跳过整个复杂的分析。
  • Pass 2 (reduce_outer_joins_pass2): 进入实际的检查和转换逻辑。

1.2 Outer 到 Inner 的降级

其核心依靠提取过滤条件中的严格约束 (Strict Quals)。 利用 find_nonnullable_rels 扫描 WHERE 子句。如果 WHERE 强制要求 Nullable Side 的某个引用列不能为空（比如 LEFT JOIN B ON ... WHERE B.id > 10，当 B 补充 NULL 时，NULL > 10 为 NULL/False 被淘汰），那么：

  • LEFT JOIN 可以安全地转换为 INNER JOIN。
  • RIGHT JOIN 可以安全地转换为 LEFT JOIN 然后内部翻转，或者直接转为 INNER JOIN。
  • FULL JOIN 可以根据两边约束的情况降级为 LEFT、RIGHT 或是 INNER JOIN。

1.3 Outer 到 Anti 的转化

外连接不仅可以被降阶为内连接，有时还可以变成反连接。 利用 find_forced_null_vars 检查 WHERE 是否有显式要求变元为 NULL 的约束（例如 LEFT JOIN B ON ... WHERE B.id IS NULL）。 如果是这样，那么这个 Join 明确就是用来找“匹配不上的行”，代码中会将该 JOIN_LEFT 转换为 JOIN_ANTI。Anti Join 在执行器阶段具有更早熔断的高效特性。

2. 无用连接移除 (remove_useless_joins)

源码位置：src/backend/optimizer/plan/analyzejoins.c 调用时机：它发生得晚一些，是在 grouping_planner -> query_planner 的起始阶段，即准备构建底层路径（Scan / Join Paths）时进行。

有时候我们连接了一张表，只是为了取个数据，但后来发现（由于视图展开、ORM 生成的过度复杂的 SQL 等原因）这张表输出的列根本没有任何用处，那我们能直接移除这个 JOIN 吗？ 如果这是一个 LEFT JOIN，并且满足某些严格的等价条件，就能做到。

2.1 移除的苛刻条件 (join_is_removable)

要移除一个左连接的右表，必须同时满足以下苛刻条件：

  1. 没有列输出：Nullable Side（被连接的右表）的任何字段（包括 PlaceholderVars) 都没有被上层的 TargetList、HAVING、ORDER BY 引用（检查 attr_needed 集合）。
  2. 不改变左表的基数 (Cardinality)：这个最为关键。不能因为移除右表，导致原本左表匹配出多行的结果减少，或者原本膨胀的行数变少。因此右表的关联必须是最多匹配一行。
    • 优化器会调用 rel_is_distinct_for。
    • 根据连接条件，判断右表在这些条件限制下是否呈现唯一性 (Unique)。
    • 这通常依赖于数据库中建立的唯一索引 (Unique Index) 或 主外键约束 (Foreign Key)，且连接符必须是可合并连接的等值运算符。

2.2 全局状态清理 (remove_leftjoinrel_from_query)

如果这个无用连接成功被认定为可以移除，这绝不仅仅是在查询树里删掉一个节点这么简单。 因为此时规划器已经构建了诸如 RestrictInfo、EquivalenceClass（等价类）甚至是各种 PlaceHolderVar。规划器必须倒查并清理由于移除了该表（及其 Relid，甚至其特有的 Outer Join Relid ojrelid）而受到关联的所有辅助结构，确保后续基于 DP（动态规划）产生的子集连接搜索不会引用到幽灵节点。

单表扫描路径生成 (Access Paths)

在 PostgreSQL 查询优化的漫长旅程中，一旦查询树（Query Tree）经过了逻辑重写、子查询提升、表达式扁平化以及外连接消除等预处理阶段，规划器（Planner）就正式跨入了核心的代价优化阶段（Cost-Based Optimization, CBO）。

代价优化的第一步，也是构建整棵物理执行计划树的基石，就是为查询中涉及的每一个基础表（Base Relation）计算出所有可行的单表扫描路径（Single Relation Access Paths）。在这个环节，无论是顺序全表扫描、各种变体的索引扫描，还是并行的扫描方式，都会被事无巨细地生成出来，供后续的路径打分和连接搜索使用。

本章节将结合 PostgreSQL 18.1 的源码（主要集中在 src/backend/optimizer/path/allpaths.c 和 indxpath.c），带你深度潜入单表扫描路径的生成机制与数据结构。

1. 核心数据结构与入口：make_one_rel

生成基础路径的整个过程由核心大管家函数 make_one_rel()（位于 allpaths.c）统筹。顾名思义，它的最终目标是把查询里的所有关系（Relations）“合成”为一个整体（One Rel），代表最终的查询结果。

在此之前，首先要做的就是武装每一个基表。

1.1 两个关键阶段的拆分

在 make_one_rel 中，对基表的处理被极其考究地拆分为了两大步：

  1. 尺寸推断 (set_base_rel_sizes)：先估算出行数（Rows）、宽度（Width）等统计学信息。
  2. 路径生成 (set_base_rel_pathlists)：基于前面算出的尺寸信息，真刀真枪地生成多种底层数据访问的物理路径（Path）。

为什么要严格分两步走？ 因为并行查询和参数化路径的生成极度依赖于表的大小估算。 举个例子，某个表的单表估算数据量如果太小，优化器可能就会直接将其标记为不需要并行（Cancel consider_parallel）。如果一边估算一边生成，可能会导致前面生成的路径错失获取准确统计信息或是其它表横向对比信息的机会。

1.2 RelOptInfo：关系信息的集大成者

在进入生成环节时，每个基表、连接结果甚至子查询，都会被包装成一个非常庞大且极为核心的结构体——RelOptInfo（Relation Optimization Info，定义于 nodes/pathnodes.h）。

在路径生成阶段，RelOptInfo 有几个极为关键的字段：

  • rows / reltarget: 代表估算的基表输出行数与列宽，决定了扫描的代价。
  • baserestrictinfo: 也就是大家最熟悉的 WHERE 过滤条件。在逻辑优化阶段，它们已经被分解成了隐含着 AND 语意的约束信息列表集（RestrictInfo 的 List）。
  • pathlist: 这是一个链表（List）。在这一阶段，由 set_base_rel_pathlists 各种手段疯狂塞入的扫描路径（Path）都会挂载在此。
  • cheapest_startup_path / cheapest_total_path: 在尝试了所有可能的 Path 后，优化器会选拔出启动代价最小和总体运行代价最小的两名“状元”，缓存于此，方便快速检索。

2. 顺序全表扫描 (Sequential Scan) 与并行化

顺序扫描可以说是任何一棵基表的“保底策略”。即使你没有任何索引，哪怕代价再高，数据库也一定能通过从外存中一页一页读取全表来得到正确结果。

在 allpaths.c 的 set_plain_rel_pathlist 里：

2.1 兜底路径：create_seqscan_path

优化器首先无条件地通过 add_path(rel, create_seqscan_path(root, rel, required_outer, 0)) 塞入一个 SeqScan（对于普通关系表）。 顺序扫描的物理含义是沿着堆表（Heap Table）的底层分页逻辑，从第一块（Block 0）老老实实地扫到最后一块。

  • 代价估算要素：顺序扫描的代价几乎只由两件事决定——这表有多少个块（rel->pages）决定了连续 I/O 时间；这表能匹配多少条记录（rel->rows / CPU filtering）决定了 CPU 处理条件判断的时间。

2.2 并行化扫描：create_plain_partial_paths

在 PostgreSQL 支持了查询并行化后，SeqScan 迎来了一个重量级的变体——Partial Sequential Scan。 紧接着生成单纯的 SeqScan 后，优化器会判断：

if (rel->consider_parallel && required_outer == NULL)
    create_plain_partial_paths(root, rel);

在 create_plain_partial_paths 的内部，系统调用 compute_parallel_worker() 会根据该表的大小，套用一系列数学对数阈值来计算**该表值得分配多少个后台自动工作进程（Background Workers）**来齐头并进地扫描。 算出的 worker 数量会被赋值，并且调用 add_partial_path 添加一个带 parallel_workers 属性的分支。 注意：这里的 partial (单方面/部分) 指的仅是单个 worker 只会扫描其中的一部分块（PG 的并行扫描机制通过在内部共享一个 Block 计数器来抢块，而不是静态切片），最后必须在顶层被 Gather 节点统一汇总。

3. 索引扫描 (Index Paths) 深度剖析

只有全表扫描当然是不够的。对于大多数 OLTP 事务型查询和具备高选择性（High Selectivity，指只能捞出表中极少数特定行）的查询，利用 B-Tree 等索引能将查询耗时呈指数级地降低。

接下来，set_plain_rel_pathlist 会调用 indxpath.c 里的终极重头戏——create_index_paths()。这个函数的工作深度极高，它会尝试穷尽所有的单索引甚至多索引组合（Bitmap Scan）方案。

3.1 Index Scan：最传统的单索引回表策略

当通过 build_index_paths 检索到 WHERE 里面的约束（比如 id = 5）能够利用某个存在于系统的 IndexOptInfo（索引优化信息）时，将产生一条标准的 Index Scan Path。

执行流约束： 对于一个普通索引扫描，即使它在聚簇树的叶节点上瞬间定位了 id = 5 的数据条目所在地，获得了 TID（Tuple ID），它依旧不得不产生一次回表（Heap Fetch）动作，跳回表的数据块把整条记录抽上来，然后再判断这条记录上的其它没有索引的非主键约束（例如 name = 'pg') 是否满足条件。

3.2 Index Only Scan：VM 护航的最佳拍档

这也是在 build_index_paths 中自动衍生探索的一种路径。 很多时候由于查询写得很棒，你要取出的列，在那个索引本身涵盖的结构里就已经全部存在了（比如 SELECT id FROM table WHERE id = 5）。在这种极端的完美情境下（又名 Covering Index），优化器其实不需要也不应该回堆表的数据块（Heap Page）。

致命局限与 Visibility Map： 然而由于 PostgreSQL 是基于经典的多版本并发控制（MVCC）并把系统表自身作为版本维护载体，仅仅提取索引键出来并不能断定这行数据在当前事务快照里是不是已经过期的（Dead Tuple），或者是不是还处于对当前事务不可见的在途事务中——因为行可见性（xmin、xmax）等藏在原始数据行的 Header 里，索引条目上没有！ 为了促成不用回表的 Index Only Scan，PG 引入了 VM（Visibility Map，可见性映射文件）。 在 create_index_paths 评估代价时，会参考目标表对应数据块在 VM 里的位标志：只有当该块的所有数据被标为 all-visible（该块没有在跑并发脏写事务，全局确定可见），优化器才会确认能避开 Heap Fetch。否则哪怕列都在索引里，也会被按回表比例加大代价，甚至直接回退到普通的 Index Scan。

3.3 Bitmap Index Scan 与 Bitmap Heap Scan

如果你的 WHERE 里不是一个等值，而是范围，甚至还有 OR 分离的多个字段索引用法怎么办（比如 WHERE age = 20 OR salary > 8000）？

依靠单一的 IndexScan 会因为疯狂读跳不同索引来提取而带来极度混乱的随机 I/O （Random I/O）。于是优化器会构建两阶段联合战法——位图扫描（Bitmap Scan）。 在 indxpath.c 里相关的函数 generate_bitmap_or_paths 处理。

  • 阶段 1: Bitmap Index Scan：先不对表数据做任何触碰，仅仅在不同的索引文件中分别扫荡。如果匹配了某一行，就在内存里建立的一个虚拟位图（通常是物理 Page 级别的）把对应该行的位（Bit）涂成全变位 1。
  • 如果针对两个索引的查询是 AND 或 OR 关系，由于得到的仅仅是纯碎的内存位图，在内存中直接进行**集合按位与、或操作（BitmapAnd / BitmapOr）**的速度是非常极致的，合并非常优雅。
  • 阶段 2: Bitmap Heap Scan：拿到最后融合好的一张硕大 Bitmap 后，拿着这张图去访问表数据。因为是依着物理块序列号线性向后推进读取 1 的地方，此时对磁盘带来的完全是优雅降级的近乎顺序连续 IO (Sequential IO)，从而消灭了直接回表带来的大量碎片化磁盘寻道。这是一项非常有智慧的做法。

劣势：Bitmap Heap Scan 的强行物理排序意味着其生成的 Record 流毫无“内在的键值排序特征”。假如上层还有一个 ORDER BY age，那它将无法复用索引原生的有序性，导致上一层强行追加显式 Sort 节点。

4. 路径组合的魔法：参数化路径 (Parameterized Paths)

普通扫描只需要评估自己的 baserestrictinfo。然而 PostgreSQL 在规划层面十分早熟地考虑到了 JOIN 的情境。特别是大名鼎鼎的嵌套循环连接（Nestloop Join）。

试想这样一个场景：A JOIN B ON A.val = B.id。 在这里，假如 B 是内表（被反复驱动扫描的那张表），如果只给 B 提前留一条无参扫描路径，那么无论 A 给什么，B 都得全扫，代价巨大！但如果能把 A.val 作为一个**传递在途的变量（Parameter）**硬塞给 B，并在 B 的底层进行索引前置判断呢？

  • 在 PG 中，这被称为参数化路径 (Parameterized Path)。
  • PG 优化器引入了专门的机制：ParamPathInfo（参数化路径信息）。在 create_index_paths 时，优化器不仅仅只生成针对 B 自己过滤的一套独立路径；它还会“未雨绸缪”地去扫一眼连接缓存区，看 B 跟别人连接时能不能借助从别人那传过来的等值条件利用 B 自己的索引！
  • 如果有，PG 会生成附带 lateral_relids（依赖哪些外部基表）标记的特殊索引扫描，它具有更微弱甚至为 0 的单独执行可行性，但这将会是外层 NestLoop 构建极低代价算法的绝杀神兵。LATERAL 语法的基石也是借由这一机制发扬光大的。

5. 路径的残忍角斗场：Pruning 与 add_path

经过上面一番狂轰滥炸式的推演探索，无论是谁，都会为哪怕一个简单的表构建出几十种奇形怪状的 Path（SeqScan, IndexScan使用索引 X, 使用索引 Y, Bitmap组合，并行…）。

为了防止随后引发指数级别的内存溢出和连接组合爆炸（Path Explosion），优化器必须随时修建多余的枝叶——路径剪枝（Path Pruning）。

所有的扫描路径都不会直接插入 RelOptInfo->pathlist 的尾巴，而是必须经过唯一的“角斗场保安”门卫函数： add_path(RelOptInfo *parent_rel, Path *new_path)。

add_path 内蕴含了一套非常精彩的淘汰机制，遵循类似于帕累托最优（Pareto Optimality）的法则。 当新生成的 Path 来试图登记入册时，它必须和 pathlist 现存的老家伙们逐一比较：

  1. 代数对比（Cost Fudge）：如果新路径的 Startup 代价（startup_cost，流式吐出第一行需要的初始代价，有利于 Limit 处理）和 Total 代价（total_cost）都一致地高于某个老路径，那从纯纸面实力来看，新路径就是个废物。（PG 会引入大约 1e-10 级别的 fuzz 模糊乘数来应对浮点数舍入不确定性误差）。
  2. 排序特征免死金牌 (Pathkeys) ：这是 PG Planner 非常精妙的一击。如果新老两套路老代价小，新代价巨高，但老的是杂乱无章出来的结果，而新路径（比如恰好使用了按键值 B-Tree 索引）附带上了与上一层节点所需的 ORDER BY 契合的输出天然排序特征（pathkeys），那么新路径就不能被单纯按价抛弃！因为它由于免去了上层做外置大数据量排序（明文的显式 Sort 节点）而变相立功，此时它会与老路径被全数并存保留入名录。

最终，淘汰了那些各个维度（代价无优势、排列无收益）都一无是处的方案后，留存在 rel->pathlist 上的就是浓缩了最优解潜力、千锤百炼下来的基础扫描规划路径了。它们摩拳擦掌，即将在接下来的 Join 阶段互相重叠连结。

多表连接顺序算法与等价类推导 (Join Ordering & Equivalence Classes)

当查询的复杂度从单表查询跃升为多表连接（Multiple Joins）时，优化器的最核心也是最耗时的终极任务就浮出了水面：决定表的连接顺序（Join Order）以及连接方式（Join Methods）。

在关系型数据库理论中，N 个表的连接方式理论上有 $N!$ （阶乘）种之多，甚至当考虑所有树结构（包括 Bushy Tree 灌木丛树组合）时，组合数会呈卡特兰数级别的恐怖爆炸。对于动辄数表、十数表的大型分析或者复杂业务查询，如果优化器（Planner）选错了连接顺序，由于连接数量膨胀导致的数据量级乘方，执行代价会以几何级数飙增，瞬间压垮系统内存或跑满 CPU。

在 PostgreSQL 中，只要参与连接的基表数量低于预设的 geqo_threshold 阈值（默认 12），规划器就会坚定地走入最硬核、最精准的自底向上动态规划（Bottom-up Dynamic Programming）深水区。而只有超过阈值时，才会回退到启发式的遗传算法（GEQO，后文详述）。

本章节将结合 PG 18.1 在 src/backend/optimizer/path/joinpath.c、joinrels.c 与 equivclass.c 等模块的源码，深度剥析动态规划建树过程。

1. 动态规划的基石：System R 风格的自底向上

PostgreSQL 骨子里继承了早年 IBM System R 数据库奠定的经典动态规划（DP）搜索骨架。

1.1 从单表到万物互联

动态规划的核心思想是利用最优子结构：为了求 N 个表的最优连接，我们先求 2 个表的、再求 3 个表的……直至求得全貌。在规划器中，负责推动这股齿轮的司令官是 make_rel_from_joinlist 函数（由 make_one_rel 呼叫入场）。

如果没有使用 GEQO，它将调用 standard_join_search 进入标准的 DP 检索流程：

  • 初始状态 (Level 1)： 其实就是前面 set_base_rel_pathlists 章节干出来的事。每一个单表（Base Relation）的最优访问路径早已躺在 RelOptInfo->pathlist 中整装待发。系统把所有基表放入一个层级数组 join_rel_level[1] 中。
  • 状态转移循环 (Level 2 到 Level N)： 外层一个巨大的循环从 level = 2 开始，一直运行到参与连接的总表数。在计算第 N 层（即 N 个表的连接组）时，算法会遍历切分方案。 例如计算 $Level_K$，可以是由 $Level_1$ 加上 $Level_{K-1}$ 组合而成，也可能由 $Level_2$ 加上 $Level_{K-2}$ 组合而成。 这个核心递推函数被称为 join_search_one_level。

1.2 Left-deep 还是 Bushy Tree？

如果你学过数据库理论一定知道连接树的方向：

  • 左深树 (Left-deep Tree)：永远是一个 Join 的结果去同一个基表（Base Table）做 Join。（如 ( (A JOIN B) JOIN C ) JOIN D）。这种树占用的内存上下文最少，特别符合 NestLoop 带 Index Scan 这种流式驱动的行为。
  • 灌木树/粗壮树 (Bushy Tree)：允许两个都是 Join 的临时结果集互相去 Join。（如 (A JOIN B) JOIN (C JOIN D)）。当 C和D 的连接率极速塌缩（比如返回0行或1行）时，Bushy Tree 会表现出惊人的前瞻性优势。

在早期的 System R 中，为了保证时间可控只考虑左深树。但现代 PostgreSQL 的 join_search_one_level 会通过配置参数与组合检查，同时探索左深树和 Bushy 树空间。它不仅尝试用 $Level_1$（单表）连接 $Level_{k-1}$ 来构造出 $Level_k$（左深树路线），只要启用了灵活连接选项，它还会尝试 $Level_2$ 和 $Level_{K-2}$ 碰撞！

2. 推理大师的底牌：等价类 (Equivalence Classes, EC)

在进入暴力 DP 枚举之前，必须提到 PG 优化器里的一张隐秘且极其强大的王牌：src/backend/optimizer/path/equivclass.c 里的等价类推理机制。

2.1 隐式约束下推

假设有这样的 SQL：

SELECT * FROM a JOIN b ON a.id = b.id JOIN c ON b.id = c.id WHERE a.id = 5

如果你只是个呆板的机器，你会先拿着 a.id = 5 去过滤表 A。结果你在试图单独连接 B 和 C 时，发现不仅 A、B、C 成了交叉联合，B 和 C 还依然带着庞大的数百万行去拼表，最后才被 A 捞走一条孤零零的数据。

PostgreSQL 是怎么解决这种信息壁垒的呢？靠建立等价类（EC）！

在预处理时，遇到 a.id = b.id 并且 b.id = c.id，PG 会拉起一个 EquivalenceClass，在这个桶里装入全体“平权成员”：{ a.id, b.id, c.id }。 更美妙的是，由于发现了常量约束 a.id = 5 也在里面，它会把 5 作为 em_is_const 的特权会员投入桶内。

连锁反应 (generate_implied_equalities)： 此后，在 DP join_search_one_level 搜索无论哪层路径、试图扫描哪怕是任意单表 b 或者 c 之前，优化器会去查它的等价桶，立刻推导出“哦！原来还有一个原 SQL 里没写的隐式条件：b.id = 5 以及 c.id = 5！”。 这就是 等价类约束下推（Implied Equalities）。在进入任何 Join 前，基表自身就已经通过被下推得到的隐式约束利用单表 Index Scan 把所有无关据消灭在了萌芽之中，给接下来的动态规划卸去了数百倍的负担。

3. 生成特定的连接途径：populate_joinrel_with_paths

当 DP 决定要撮合 RelA 和 RelB 组成一个新 Rel (Rel A+B) 层级后，它调用 make_join_rel。这里的关键就是 到底以何种算法、何种物理操作符（Operator）将这两个子结果拼在一起。

负责实施拼合的是大名鼎鼎的 populate_joinrel_with_paths() （位于 joinpath.c），它为这个新结合的关系打磨生成这三大门派的物理路径集合，并一股脑丢给 add_path 去进行适者生存淘汰：

3.1 预处理：无序匹配与巢状循环 (NestLoop / HashJoin)

通过调用 match_unsorted_outer，系统开始假定：如果不要求特殊的顺序，就原生态地把外表和内表塞进去。 这时候会衍生出两种主流路径：

  • NestLoop Join：朴实的双循环。但 PostgreSQL 在这里非常聪明！它会检查有没有我们上一章讲到的 参数化路径 (Parameterized Paths) 可供使用。如果有，它会让驱动表（外表）在流式遍历时，把它的行变量利用 ParamPathInfo 直接塞到被驱动表（内表）底层作为 Index Qual 发射！这种做法极大地盘活了 Nestloop。
  • Hash Join：如果探测到了连接条件有等号（hashjoinable = true），优化器也会塞入 Hash 路径。Hash Join 是无序流对接的最强利器，代价估算中它会考虑内表尺寸去建模内存里修建 Hash Bucket 的代价（cost_hashjoin）。

3.2 有序匹配：排序归并 (Merge Join)

随后调用 sort_inner_and_outer。 Merge Join 极其依赖于数据输入的预先排好序。优化器会检查两边输出集合如果自身带着 pathkeys（比如从 Index Scan 出来自带 B-Tree 排序，或上一层已经做了显式 Sort），是否正好命合了当前这个 Join Qual （连接条件）的排序诉求！ 如果两边天然有序，那构成 Merge Join 的启动成本将暴雪式骤降。如果不具备或者只有单边具备，PG 也会为它强行添加在内存中做 Sort Node 的成本（cost_sort），然后构筑起 Merge Join 路径进行公平比稿。

3.3 缓存魔法：Materialize 路径辅助

在各种嵌套内部中，如果被频繁重复读取内部关系时（尤其是没有命中参数化索引降维的情形），会调用辅助流程加上隐式缓存节点操作，即 create_material_path。这个节点能在内存（或临时文件盘）中将查询出的子查询结果物理钉住，从而挽回大量重算的性能流失。

4. 防御爆炸：启发式空间约束与剪枝

如果你允许对所有左深树、灌木树，搭配全排列的各种 Nested/Hash/Merge 连接全部敞开探索，DP 即使再优，时间复杂度本身依然是呈极高的多项式的。特别是遇到笛卡尔积（Cross Join，没写 ON 条件盲拼）乱入，空间状态更是翻倍。

为了保证查询不会为了节约两秒的执行时间，反而在 Planning Server 活活死等算了一分钟计划，PG 利用参数做出了强烈的启发式空间剪枝：

  • join_collapse_limit：强行限制由 FROM 罗列起来隐式交叉拼凑而来的列表展平数量。超过限制就不打乱你的拼装顺序。
  • from_collapse_limit：针对显式 JOIN 写的查询，同理，一旦查询中明明白白写的表堆叠超过这个阈值，就不再允许跨多边界把里面的嵌套展开来大乱炖，直接在原始代码结构小圈子里进行局部 DP。
  • 推迟笛卡尔积（Defer Cross Join）：在搜索某一 Level 时，任何缺乏 JoinClause 限制、必定导致乘方数据膨胀的关系匹配，如果不属于最终必须强制捏合的表尾，优化器都极度拒绝对其进行早期评估联合！它宁可在别的地方找有连接条件的表生成 Left-Deep 链，也不会轻易提早把没关联的结构连在一起从而拖死执行栈。

至此，在这个被精心裁剪过搜索树分支的 DP 主框架里，PostgreSQL 像编织藤网一般，最终凝聚出位于最高层级 join_rel_level[levels_needed] 里的一份终极 RelOptInfo。里面存下了一颗挂满算子、代价和排序策略最优解的物理查询计划树干。

遗传查询算法优化器 (GEQO, Genetic Query Optimizer)

在前一章中，我们详细解析了 PostgreSQL 的骄傲——自底向上的动态规划（Dynamic Programming, DP）机制。DP 能在极度复杂的关系里寻找出理论上的绝对最优执行序列。

但天下没有免费的午餐，即便是应用了 join_collapse_limit 等启发式参数进行搜索边界裁剪，DP 寻找最优路径的算法复杂度依旧是随基表数量呈 $O(3^N)$ 乃至更糟的指数级膨胀的。 当你的一个极其庞大的数仓或分析型报表视图涉及到数十个表（默认参数 geqo_threshold = 12 甚至更多）互相 Join 时，如果坚持让数据库去穷举动态规划排列组合，优化器规划一条 SQL 的时间可能会比直接全表扫一遍执行出来的总时间还长成百上千倍。

这就是 GEQO (Genetic Query Optimizer) 挺身而出的时刻。

PostgreSQL 在 src/backend/optimizer/geqo/ 目录下植入了一个完备的基因进化算法器。它的目的并不是给你最完美的解答，而是在你可以接受的几十毫秒时间内，抛弃遍历思想，直接给你一个**“虽然可能不是全局最顶尖，但绝对足够好”的近似优解**。

1. 概念隐喻：SQL 优化如何转变为旅行商问题 (TSP)

生物演化与遗传算法的核心在于将现实问题编码成“染色体结构”。

在 GEQO 视角下，寻找多个表的连接排序，实质上相当于组合优化领域最著名的 NP-Hard 难题——旅行商问题 (Traveling Salesman Problem, TSP)。

  • 城市 (City) = 基础表（Base Relation）。
  • 路线 (Tour) = 表之间拼合的连接顺序（Join Order），比如这是一条路径：$A \to C \to B \to D$。
  • 路费 (Path Cost) = 当前路线上跑出来的连接计划的总代价值（Total Cost），也就是执行器要付出多少资源来跑这条排好序的执行计划。

由于 GEQO 的本意是简化复杂度，目前 PG 源码里限制了 GEQO 只在 Left-deep 树空间内进行染色体编码搜索（这也是 TSP 问题的一个自然映射方式——从一个点出发一条路走到底），放弃了复杂的 Bushy 多路聚合基因组建模。

2. GEQO 的整体运转循环

遗传算法是一个基于达尔文“优胜劣汰、适者生存”理论的高度迭代模型。它的全局调用入口在 geqo_main() 这个函数里。

整个 GEQO 宇宙的创世与演进包含了以下五个神圣阶段：

2.1 种群初始化 (Initialization)

上帝说要有光，程序里调用了 random_init_pool()。 优化器会从所有可能的表排列组合中，直接按白噪音一般的随机排列，生成初始的若干个路线配置组合（由于基表数量决定了基因变异的跨度，此时的初始种群数量 Pool Size 会通过内置公式基于 N 自适应地扩张。N 越大，池子越大，默认最多会产生数百个个体）。

2.2 适应度评估 (Fitness Evaluation)

这是一个“照妖镜”环节，入口在 geqo_eval()。 对于 Pool 里的每一个个体（排好序的一串表名集合，例如 $A, B, C, D$），GEQO 不会自己凭空算盘，而是非常讨巧地去白嫖传统优化器的代价计算引擎！ 它会按这个顺序去强行构建一个完整的 Join Tree，交给 cost_qual_eval 等底座函数计算一遍，拿到诸如需要 3450.25 的硬性数字指标，然后把它映射为个体在这个种群中生存下去的适应度 (Fitness)。Fitness 越低越好（因为 Cost 越低越快）。

然后，所有染色体由于它们不同的“强壮度”会被打分，并调用 gimme_pool_size() 在数组里面按优劣直接排好队（Ranking）。最上面的一定是最强壮的，最底下的随时要被淘汰。

2.3 选择 (Selection) 权重的马太效应

遗传的法则是：越是强者，留下子嗣的概率越高。 当系统进入繁殖轮次时，会随机挑选两个个体作为父母 (Parents)。如何保证强者恒强但又保留穷人的翻盘几率？ 在源文件 geqo_selection.c 中，PG 采用了一种叫做 基于排名的线性选择法 (Linear Ranking Selection)。排名越靠前（Fitness 好的代价值低的个体），抽中作为父母的概率基数越高，但底下的哪怕是最次的依然有一丝微弱概率被点兵中选。

2.4 交叉/受孕 (Crossover / Breeding)

最核能的部分来了，由于 TSP 序列必须是表名的无重复混洗（不能出现 $A,B,A,C$ 这种畸形路线），PG 在几十年前选用了一种学术界公认最为精密巧妙的乱交算法：边缘重组交叉算子 (Edge Recombination Crossover, ERX)。 调用在 geqo_erx.c 中的 gimme_edge_table() 及其核心调度里。

这个奇妙算法能极大地保留父母代“连接对”的基因。例如，爸爸有 A->C，妈妈也有 C->A，既然 C 一直缠着 A 不放，那说明虽然你们俩路径别的不管，但 A 相邻着连接 C 必定是个极其有缘的优良基因！在产下全新子代（Child）时，子代码会主动向周边有交集的游离基因靠拢并生成出 $A \to C$。这是打破常规遗传算法断点交叉的最强防线，非常贴合 Joint 树“关联近邻更好”的特征。

2.5 变异 (Mutation) 为了防止内卷

如果在好几百个代的推演里，大家只生娃，整个群体的基因很容易陷入局部最优解 (Local Minima)（大家都变成了一种相近的排列，怎么交叉还是生出那个德性的路线），从而彻底停止了对更优路径的探索。 PG 设计了 geqo_mutation() 函数，给每一个新产下的个体植入一点点叛逆的随机数。它会以微小的概念发生染色体调换，比如让里面的第 3 个表和第 5 个表突然对调一下顺序。这就强行引入了野生的新特性，使得搜索图能够跳出局部坑洼。

3. 收敛与局限性：双刃剑的利与弊

经过由配置参数 geqo_effort（从 1 到 10 级）控制的上百代种群疯狂繁衍以后，系统强行叫停。整个上帝视角的演化场里，站在 Pool 存活最顶端的排在第 0 顺位的那个最强个体，就会被强行抽取出来。 PG 就拿着这个通过概率、玄学以及漫长时间线压缩计算得到的最后序列，拍板定夺：“我们下一跳就按照这个 Join 计划去运行吧！”

副作用与建议：每次看 Explain 计划都在跳舞？

如果你真的在超过 12 张表上尝试复杂的 BI 报表级 JOIN，你会面临一个非常神奇且让你 DB 图谱工程师血压升高的现象：

既然遗传算法本身是带有伪随机特性起手的，并且随机算子始终交织在繁衍树中，那么同一个复杂的查询，在第一次跑出和第二次跑出的时候，哪怕整个世界的数据都不曾变动，它最终选出来的解释计划（Explain Plan）竟然很有可能是大相径庭的！ 这也就意味着查询响应时间会出现偶尔长偶尔短的神奇心跳图。

为了解决调试麻烦，如果你确实希望在复杂图景里获得极其稳定的执行规划树：

  1. 可以考虑去修改当前会话里的 geqo_seed（遗传随机种子），把它定死为一个小数，使得同一个版本内的搜索具备唯一重演性。
  2. 如果不差几百毫秒准备时间或者用的是极高性能规划服务器，甚至可以通过设置 SET geqo = off 强行禁用掉遗传算法引擎，或者极度调大 geqo_threshold 强逼底层执行标准的 DP 爆破扫描。

在这个庞大犹如钟表机械般细致刻度的规划器深渊里，无论动态规划如同地毯式轰炸般的精准，抑或是在极端条件下遗传算法的退缩防御妥协，都展示出了 PostgreSQL 对数学之美在应用工程中调和最强悍的一面。

后置优化与物理算子拼接 (Post-Optimization & Upper Planner)

如果说查询规划的“上半场”（严格来说在代码里叫 Lower Planner，即底层的 Join 树构建）解决的是“数据去哪里找、按什么顺序拼接最快”的问题，那么规划机制的“下半场”——Upper Planner (高层优化器)，解决的则是“拼完了数据怎么满足用户最终的语义控制”的问题。

当你的 FROM 和 WHERE 连接网络被 query_planner 函数结算为一颗拥有最优代价值的 RelOptInfo（例如 join_rel_level[N]）返回时，优化器会从 query_planner 函数退出来，回到其广袤的母调度函数：grouping_planner（位于 src/backend/optimizer/plan/planner.c）。

在 grouping_planner 中，系统会为这棵刚刚建成的基底数据树逐层裹上“包装”，处理包括且不仅限于：GROUP BY (聚合), WINDOW FUNC (窗口函数), ORDER BY (排序), DISTINCT (去重) 以及 LIMIT/OFFSET (截断) 等高级操作。

结合 PostgreSQL 18.1 的最新代码，我们将深入这条“物理算子包装流水线”。

1. 入口统筹：Upper Rel 与 grouping_planner

与底层基表（Base Rel）或连接（Join Rel）不同，PG 抽象出了一种名为 Upper Rel (上层关系) 的 RelOptInfo，通过 UPPERREL_GROUP_AGG、UPPERREL_WINDOW、UPPERREL_ORDERD 这种专有的 UpperRelationKind 分级不断往顶层堆叠。

在 grouping_planner 中，标准的装配链（Pipeline）是极其严苛有序的。每当装配完一层，就会生成一个涵盖了前面所有计算结果的新 RelOptInfo：

  1. 底层扫描与联接 (query_planner 完成)。
  2. （如有）追加分组与聚合 UPPERREL_GROUP_AGG。
  3. （如有）追加窗口函数计算 UPPERREL_WINDOW。
  4. （如有）追加 SELECT DISTINCT 去重。
  5. （如有）追加 ORDER BY 排序 UPPERREL_ORDERD。
  6. （如有）追加 LIMIT / OFFSET 截断。

下面重点剖析这套外置神兵利器的核心模块。

2. 聚集规划 (Aggregation & Grouping)

在面对包含聚合函数（SUM(), AVG()）或声明了 GROUP BY 的查询时，系统会跳入 create_ordinary_grouping_paths() （普通分组路径构建）去生成聚合方案。

PG 支持三种硬核的物理聚合算子：

2.1 基础聚合 (Plain Aggregate)

应对的是诸如 SELECT count(*) FROM table，没有 GROUP BY 的查询。 此时系统只需在顶端挂载一个单行的聚集器。输入流来了之后，它只会在内存中累加内部状态值（Internal State），流结束后输出唯一的一条合并行记录。启动代价小，逻辑极其精简。

2.2 排序归并聚合 (GroupAggregate / Sorted Aggregation)

要求输入的数据流必须已经按照 GROUP BY 列强行排好序。

  • 机制：类似于一个流式游标。当读入的值和上一条的值在 Group 键上一样时，累加聚合状态；一旦读到新的 Group 键值不同的数据（越界），立即抛出刚刚结算完的那个聚合组结果，并重置内部计数器重新开始。
  • 优缺点：内存占用可谓是极致的省（无论表有几百 T，内存消耗也几乎不变）。但代价是：如果你底层吐出来的数据压根没有 pathkeys（比如是用 HashJoin 做首选计划出来的无序堆栈），GroupAggregate 就会惨叫着要求在底下先强压一个昂贵的 Sort Node（排序算子）再去 Agg。

2.3 哈希聚合与落盘 (HashAggregate & Disk Spill)

不需要预先排序。

  • 机制：上来就在内存（work_mem）里修筑一个庞大的 Hash Table。扫入的数据通过计算 Group 键的 Hash 值强行扔入不同的 Bucket 里并累加状态。扫完全表后，再将 Hash 表里的结果一口气倾倒（Egest）出栈。
  • 重大进化 (PG 13+)：在老版本的 PG 中，如果 Group 的类别数量异常庞大，Hash 表把内存活生生撑爆会导致直接 OOM 报错。后期的 PG 加入了 HashAgg Disk Spill (落盘机制)。这意味着优化器在调 cost_agg 进行代价估测时，一旦发现不同组数（Number of Groups，通过 ndistinct 估算）超过内存，它也不会简单粗暴抛弃哈希聚合，而是会引入多批次落盘合并的额外磁盘 I/O 代价折算惩罚。
  • Mixed Aggregate：如果 GROUP BY 有多组展开逻辑（如 GROUPING SETS, CUBE, ROLLUP），PG 会利用专有的 GroupingSetsPath 同时衍生混合 Hash 与 Sorted 的交叉策略。通常是对 ROLLUP 等能构成线性前缀依赖的使用 Sort 算子一次切出多个聚合层次（优美地降维），对平级的无序组合使用 Hash 并行构建。

3. 窗口函数 (Window Functions)

在聚合之后，就是窗口函数的战场 UPPERREL_WINDOW（位于 planner.c: create_window_paths）。

当你用到了 SUM(x) OVER (PARTITION BY a ORDER BY b) 这种强大的跨窗函数时，必须明白：

  1. 窗口函数实质上是一种变种的 GroupAgg，它极端渴求输入序列必须按 a 和 b 严格有序（pathkeys 的要求简直苛刻到了极致）。
  2. 多窗口合并优化：如果一个 SQL 里有五六个窗口函数，有的 PARTITION BY a ORDER BY b，有的则是 PARTITION BY c，PG 会在 group_window_funcs 操作里尝试把 OVER 子句诉求完全一致的合并在一次物理节点轮询里执行掉。不同规则的必须被切割成独立执行栈。
  3. 优化器会重新评估底下这堆由 GROUP BY 输出出来的路（Path）。假如下面的路正好自带按 a, b 排好的 pathkeys，那就省去了二次排序，直接加上窗口算子。如果没有？那就硬着头皮加上显式的 Sort Node。

4. 排序的神器进阶：增量排序 (Incremental Sort)

这是近期 PostgreSQL 带来的真正惊艳业界的一大特性。 所有的排序需求（无论源于 ORDER BY a, b 还是别的）归根结底都要通过 create_ordered_paths 往上塞。对于最棘手的 Sort 我们以前只有全表堆排序。

试想场景： 假如你的查询需要 ORDER BY a, b。由于你在 a 上有一个出色的 B-Tree 索引，底层的 Index Scan 替你捞上来的数据确实已经按 a 排好序了，只是 b 还是乱序的。也就是所谓的“提供了部分前缀排序（Prefix Sorted）”。 在没有 Incremental Sort 之前，PG 会长叹一声：“反正 pathkeys 对不上 (a,b) 组合的完美一致，底下的排序全当废纸！全量装进内存来强制 Sort！”。

Incremental Sort 机制： 现在，由 pathnode.c 中相关的逻辑负责捕获 Prefix Sorted 特征。当检测到已存在 a 的前缀有序性时，优化器能够插入一条轻量级的 Incremental Sort 路径。 在执行时，只要探测到前一个读取的 a 的值没变，它就把 b 暂时吃进缓存；一旦 a 变了，这意味着前一个组所有的 a 全部到齐了，就立即针对前面那个小缓存栈里的区区几条数据对 b 进行极低代价的微型内存 Sort 并输出。 这个技术直接将 ORDER BY a, b LIMIT 查询的延迟在很多长尾场景下削减了数量级倍。

5. 截断 (Limit / Offset) 对底层引擎计划的倒逼

如果你以为 LIMIT 1 只是最后在最外层加一个类似 head -n 1 的截断算子，那就大错特错了。 LIMIT 具备极为强烈的计划倒置影响力 (Plan Inversion Power)！它存在于 create_limit_path() 之中。

当你的整棵查询树带着几十万总代价值（Total Cost），经过了一路淘汰走到最顶的 grouping_planner 遇到了 LIMIT 10 时： PG 的代价估算器（cost_limit 函数）不仅修改上层代价值，甚至会去翻阅之前在 RelOptInfo->pathlist 遗留的所有未决执行路径。

之前说过，优化器会为一个组合保留两条路：cheapest_startup_path (启动极快但总体最慢) 和 cheapest_total_path (启动很慢需要等待加载但一旦动起来总体全扫完最快)。 如果是没有 LIMIT 的普通提取：Hash Join 由于扫全表建哈希表，total_cost 极低，必定压倒 NestLoop。 但在有了 LIMIT 之后：优化器只需要拿到十行记录 (tuples needed = 10，基于 rel->rows 做了乘积比估值降维)。Hash Join 因为需要先把两张表各自的几百万全缓存下来才能吐出第一行，其 startup_cost 令人发指。而看似非常笨拙的 NestLoop + Index Scan 却能在 0 缓冲的情况下瞬间拿到第一行、第十行的结果并直接终止循环。

在 LIMIT 的神来之笔下，优化器会给拥有最优 startup_cost 的路径重新打上高倍数的惩罚削减分数，成功颠覆由系统默认的总价计算主导下的 HashJoin 王权，切换到那些可能一辈子都不会被选用的 Fast-Start 参数化嵌套循环和纯净索引路径（Index Scan with order）。

整个 post_optimization 的 Upper 处理环节看似简单地串联收尾，实则是处处存在与底层状态联动拉扯的精妙算计！

并行执行计划生成 (Parallel Query Planning)

自从 PostgreSQL 9.6 划时代地引入并行查询架构以来，这个体系经历了数十个大版本的迭代，如今在 PostgreSQL 18.1 中已经发展成了一个极为庞大、健壮且无缝融入代价优化器（CBO）的深水区模块。

并行查询的魔力在于：当面对亿级别的海量数据集扫表或重型聚合时，规划器可以果断打破此前单进程单核运算的桎梏，派遣多个后台工作进程（Background Workers）分进合击，最后再由发起查询的主进程（Leader Process）进行数据汇总。

在这个章节中，我们将深深扎入源码底层的并行抽象网格，搞清楚“谁在下派任务”、“怎么汇总结果”以及“惊艳的并行 Join 是怎么实现的”。

1. 并行的基石抽象：Partial Path 与 Gather

所有的并行计划，在优化器里的表现形式无非是两个全新出现的构件。

1.1 部分路径 (Partial Path)

如果你之前读过我们的“单表扫描”章节必然记得：在 set_plain_rel_pathlist 中，系统不但会添加一条 SeqScan 路径，还会调用 create_plain_partial_paths 添加一条被称作 Partial Path 的分身。

什么是 Partial？ Partial 的物理意义是：“这条路只要你把它扔给一个独立 Worker 去跑，它只会吐出现有总数据量的一个局部切片（Slice）”。 为了实现这个互不干涉的切片，底层执行器在扫表时会依赖一块动态共享内存（DSM, Dynamic Shared Memory）里维护的原子游标（Block Counter）。多个 Worker 齐头并进地拿原子操作去抢页（Page），谁抢到了谁读，这导致输出的数据是绝对无序且切分的。 在规划阶段（Planner），它通过设置 path->parallel_workers 大于零，并且将此路的预期代价值 / workers 比例折算进行入库。

1.2 并行安全 (Parallel Safe) 的铁律

不是所有的一切都可以拉起并行！优化器在任何一个关系被标记为 consider_parallel = true 前，都会启动一场极度严苛的 is_parallel_safe() 地毯式搜查。

  1. 函数/操作符检验：如果你在 WHERE 或者 TargetList 里面引用了一个被标记为 PARALLEL UNSAFE 的用户自定义函数（UDF，比如带外发网络请求、或者直接写表的函数），那么整个这一脉络将永久拉黑并行探索。
  2. 写事务隔离：绝不可以在含有未提交的写缓冲数据的临时表（Temporary Tables）和游标结果集上生成并行作业，因为 Worker 进程是不允许看到 Leader 的私有本地内存更改的。

2. 收集节点：Gather 与 Gather Merge

光有各个 Worker 在底下苦哈哈地算出局部结果（Partial Path）是不够的。这些零散的支流必须要被收束回主进程（Leader Process）形成唯一的返回流。 优化器会通过不断往上推进调用 generate_useful_gather_paths() 强行为底部的 Partial 树冠盖上一个叫 Gather Node 的盖子。

有了 Gathering，一条原本残缺的 Partial Path 在顶层看来就重新变成了一条完整的正常 Path 参与总路线的角逐。按照是否保序，PG 提供了两大收束神器：

2.1 粗犷提速的无序统合 (Gather)

最经典的普通 Gather 算子。 Leader 创建多个物理队列池并启动协程。各个 Worker 算完数据就往共享内存的消息队列（Tuple Queue）里无脑塞。Leader 轮询拿到什么数据就立刻吐回给客户端。这是对性能损耗最低的收集手段，但它把所有的底层排列完全打乱了。

2.2 优雅归并的保序收集 (Gather Merge)

这是一个极为考究代价取舍的东西。 假设你底下一个 Worker 在走并行 B-Tree 索引扫描（其实在扫描中依然是按页分配，只是保证了单 Worker 自己读到的局部区块是有序的）。Leader 在收到这 N 个流时，如果不加思索地套用 Gather，出来的总记录集就是一堆交错穿插乱序的数据。

但在某些场景下，上层非常需要这个 ORDER BY 排序（比如正好遭遇了 Merge Join 或者外层有极耗时的全局大排序限制）。 此时 PG 会给它盖上一个 Gather Merge！ Gather Merge 节点自带一个 堆排优先队列 (Min-Heap / Priority Queue)。它同时死死盯住 N 个 Worker 发来的管线头部数据记录，不断抓取最小（排序最优）的一个向上抛出，从而完美确保了 N 个已排序局部流在合成 1 条全球流时，丝毫不破坏总体的有序性（Pathkeys 完美保存）！

3. 并行连接 (Parallel Joins) 的底层黑科技

如果并行只能做全表扫，遇到多表连接依旧回退到串行，那等于鸡肋。PostgreSQL 的王炸在于针对三种不同的物理 Join 设计了出神入化的并行执行版本。

3.1 Parallel Hash Join (并行哈希连接)

普通的 Hash Join，如果让 N 个进程并行，很可能发生 10 个进程在内存里各自都建了一个一样的外表 Hash Table，完全浪费内存，还可能爆 OOM。

在 PG 的实现中，通过了史诗级的动态底层同步：基于 DSM 的 Shared Hash Table（共享哈希表）。

  1. 建表阶段 (Build Phase): Leader 在 DSM 中开辟好内存坑位。所有的 Worker 一拥而上读取内部小表（Inner Table），每个人扫多少块算多少数据，然后用原子操作共同把 Tuple 塞进这唯一一个共享巨大的 Hash 桶里！这种利用多核 CPU 加速建立散列表的过程非常恐怖。
  2. 锁步栅栏 (Synchronization Barrier): 用一个全局信号栅栏，强制等所有 Worker 都把建表这一步拼完了。
  3. 嗅探阶段 (Probe Phase): 放开栅栏！所有 Worker 扭头去对外表（Outer Table）开启互相切片的无序分块扫。扫出一条就跑去 DSM 的中间大表里碰哈希，碰上就合成输出。彻底把 Join 计算在多核 CPU 跑满了 100% 的并行度。

3.2 Parallel Nested Loop (并行嵌套循环)

这个看起来非常粗暴但往往有奇效（尤其当使用前面提到的参数化路径 Parameterized Path）时。

  • 外表切片：Outer Table 必须是一个产生 Partial 数据流的状态（也就是多个 Worker 在各自切块抢这几百万条外表数据）。
  • 内表广播：Inner Table 是串行的 Full 扫描。这意味着每一个 Worker 每次抢到一行外表数据后，都要拿着这行数据作为上下文，去对完整的内部大表进行一次（通常是 Index Scan 的）完整扫描。各个 Worker 之间互不干扰。这类似于 MPP 大数据引擎里的 Broadcast Join 分发思想。

4. 并行查询的代价模型与现实打压 (Cost Punishments)

你或许会问：既然并行这么天下无敌，为什么 PG 大多数查询还是单进程执行的？ 优化器里存在两个非常著名的“反并行强力惩罚”估算：Setup Cost 与 Communication Cost。

  1. parallel_setup_cost (启动成本高昂) 拉起后台进程（Backend Process）并不是通过操作系统轻量级线程。PG 由于历史架构，这是一个结结实实的进程级 fork()（并且带有海量的锁库与共享内存地址映射重建）。优化器会在计划里硬生生砸下默认高达 1000.0 代价的初始启动惩罚，这意味着只有执行时间可能需要秒级的查询，才“配得上”去等待拉起并行的开销。
  2. parallel_tuple_cost (管线通讯极细微的阻塞) 将一行计算好的 Tuple 从 Worker 封包写入 DSM 的环形缓冲环，然后在主进程又把它解包抽出来。这是一笔显著不可忽略的 IPC 收发代价。若查询在 Join 给出了千万级海量的最终结果要交由 Leader 吐出，这笔通讯惩罚加起来极为震撼。系统甚至可能会因此倒逼出一个：“我们还是干脆用原版单核自己安安静静扫完，还不用在内存里互相等锁阻塞” 的最优妥协。

通过对上述所有的精妙代数模型估算并在最后层通过 add_path 交火比拼。在 grouping_planner 收尾时刻，如果是自带 Gather 的这棵庞然大物路径总体成本比串行确实依然大幅胜出， 那么，恭喜，你写下的这张 SQL 稿件，即将驱动整个服务器那密密麻麻的 CPU 集群矩阵疯狂咆哮了。

standard_planner 函数入口

standard_planner 是 PostgreSQL 默认优化器的主要入口函数。

位置: src/backend/optimizer/plan/planner.c

整体流程

standard_planner 的执行流程大致如下：

PlannedStmt *
standard_planner(Query *parse, const char *query_string, int cursorOptions,
                 ParamListInfo boundParams)
{
    /* 1. 初始化 */
    root = makeNode(PlannerInfo);

    /* 2. 预处理 (Subquery Pull-up, Constant Folding 等) */
    /* 实际上这部分主要在 subquery_planner 中完成 */
    root->parse = parse;

    /* 3. 调用 subquery_planner */
    /* 这是递归优化的核心，处理子查询和当前层级的优化 */
    plan = subquery_planner(root->glob, parse, NULL, false, tuple_fraction);

    /* 4. 最终收尾 */
    /* 将 Plan 树转换为 PlannedStmt，准备交给 Executor */
    result = makeNode(PlannedStmt);
    result->planTree = plan;
    
    return result;
}

subquery_planner

subquery_planner 负责处理特定查询层级的逻辑。

  1. pull_up_subqueries: 提升子查询。
  2. preprocess_expression: 规范化表达式。
  3. reduce_outer_joins: 外连接消除。
  4. grouping_planner:
    • 这是真正进行路径搜索的地方。
    • 它首先调用 query_planner 来处理 Scan/Join 优化。
    • 然后自己处理 Group By, Agg, Sort, Limit 等非 Join 操作。

query_planner

query_planner 是核心中的核心 (The Core)：

  1. setup_simple_rel_arrays: 初始化 RelOptInfo 数组。
  2. add_base_rels_to_query: 识别所有基表。
  3. make_one_rel:
    • set_base_rel_pathlists: 生成单表访问路径。
    • make_rel_from_join_list: 生成多表连接路径 (DP / GEQO)。
    • 返回一个代表最终结果关系的 RelOptInfo。

总结

调用栈： standard_planner -> subquery_planner -> grouping_planner -> query_planner -> make_one_rel

核心数据结构 RelOptInfo

在 PostgreSQL 优化器中，一切皆 RelOptInfo。

定义: src/include/nodes/pathnodes.h

概念

RelOptInfo (Relation Optimizer Info) 用来表示一个“关系”。这个关系可以是：

  1. Base Relation: 一个真实的表（或子查询结果）。
  2. Join Relation: 两个或多个表连接后的结果。
  3. Upper Relation: 进行了聚合、排序后的结果。

在优化过程中，优化器会为每个参与的表创建一个 RelOptInfo，然后通过两两组合创建新的 Join RelOptInfo。

关键字段

typedef struct RelOptInfo
{
    NodeTag     type;
    RelOptKind  reloptkind;     /* RELOPT_BASEREL, RELOPT_JOINREL, etc. */
    Relids      relids;         /* 包含的基表 ID 集合 (Bitmap) */
    
    Rows        rows;           /* 估算的行数 (Cardinality) */
    int         width;          /* 估算的平均行宽 */
    
    List       *reltarget;      /* 关系的输出列 (TargetList) */
    List       *pathlist;       /* 这是一个 Path 列表！存放所有可能的访问路径 */
    List       *ppilist;        /* ParamPathInfo list */
    Path       *cheapest_startup_path; /* 启动代价最低的路径 */
    Path       *cheapest_total_path;   /* 总代价最低的路径 */
    
    /* ... 统计信息, 索引信息, 等价类信息 ... */
} RelOptInfo;

Path 与 RelOptInfo 的关系

  • 一个 RelOptInfo 代表逻辑上的一个关系（比如 A Join B）。
  • pathlist 存储了实现这个关系的多种物理方式（比如 NestLoop(A, B), HashJoin(A, B)）。
  • 优化器的目标就是为最终的 RelOptInfo 找到 cheapest_total_path。

Path 数据结构

typedef struct Path
{
    NodeTag     type;
    NodeTag     pathtype;       /* T_SeqScan, T_IndexScan, T_NestLoop ... */
    RelOptInfo *parent;         /* 所属的 RelOptInfo */
    
    Cost        startup_cost;   /* 启动代价 (返回第一行前的代价) */
    Cost        total_cost;     /* 总代价 (返回所有行) */
    
    List       *pathkeys;       /* 结果的排序属性 */
} Path;

每种具体的路径（如 IndexPath, NestPath）都继承自 Path 结构体，并增加特定的字段。

等价类 (Equivalence Classes, EC) 深度剖析

在 PostgreSQL 优化器的世界里，无论是对单表扫描条件的过滤，还是在复杂的 Join 规划中寻找匹配路径，亦或是处理 ORDER BY 时查找可用的排序索引，其最底层都离不开一个贯穿全局、至关重要且极其精妙的抽象概念——等价类 (Equivalence Classes, 简称 EC)。

由于关系型数据库的查询语句中经常会出现大量相互关联的过滤条件，例如 A = B AND B = C AND C = 5。如果优化器仅仅从语法层面对抽象语法树进行机械的解析，它就只能识别出 A 与 B 有关、B 与 C 有关。

然而，一旦系统引入了“等价类”这个基于离散数学中“传递闭包（Transitive Closure）”概念的抽象机制，优化器就仿佛开启了“上帝视角”。它能够在瞬间在这些相互关联的数据关系中，刻画出了一张全图连通的无向图。它能毫不费劲地推断出 A、B、C 这三个字段在逻辑上是完全等价的命运共同体，而且在查询执行时，它们实际的数值必定都等于常量 5！

本文结合 PostgreSQL 18.1 的最新源码（其核心逻辑实现在 src/backend/optimizer/path/equivclass.c 与 src/include/nodes/pathnodes.h 中），对这个经常被外围技术文章一笔带过，但在 PG 底层却发挥着中流砥柱作用的“幕后英雄”进行一次抽丝剥茧般的硬核拆解。

1. 源起：等价类的生命周期与无向图抽象

当一条 SQL 语句经过解析和重写，形成查询树到达优化器被展开处理后（例如在 initsplan.c 中的 distribute_qual_to_rels 阶段），系统在遇到 WHERE 或 JOIN/ON 子句中的各种二元布尔表达式时，便开始寻找等价关系的蛛丝马迹。

当规划器捕获到一个形如 X = Y 的表达式时（严格来说，这里的 = 操作符必须能够在 B-Tree 索引中支持排序，并且在 pg_amop 系统表中被声明具有哈希相容性和等价语义，通常作为 RestrictInfo 存在），系统并不会简简单单把它当作一个普通的过滤条件随手下发。相反，系统会立刻暂停通用处理流程，转而触发一个专属的核心构建函数：process_equivalence()。

1.1 从离散条件到传递闭包的野蛮生长

process_equivalence(PlannerInfo *root, RestrictInfo **p_restrictinfo, bool below_outer_join) 函数的核心职责就是将一个个零散的点，编织成一张互相连通的拓扑网：

  1. 新建与收容：假设当前优化器是一张白纸，系统第一次遇到了等式 X = Y。它会立刻分配内存，创建一个新的 EquivalenceClass (EC) 实例对象，并将 X 和 Y 这两个表达式各自包装成 EquivalenceMember (成员) 存放进该集合的链表中。
  2. 顺藤摸瓜的扩容：紧接着，系统读取到了下一个谓词 Y = Z。在处理这段逻辑时，引擎会先去检索当前已经存在的所有等价类集合。它会敏锐地发现表达式 Y 已经存在于刚刚建立的某个 EC 中了。此时，系统不会再去重复建构新的集合，而是利用等价的传递性，直接把新出现的成员 Z 加入到包含 X, Y 的已有等价类里。于是，这个集合扩充成了 {X, Y, Z}。
  3. 不同集合的合并：在复杂的查询中，这种情况非常普遍。假设系统在处理某一侧的表连接时，推断出了一个局部的等价类 {A, B}。而在处理查询的另一部分时，得到了另一个局部的等价类 {C, D}。此时，主查询抛出了一个跨越这两侧的连接条件 B = C。 见证优化的合并时刻上演了！优化器在检索时发现了这两个相互独立的集合片段，于是立刻执行底层的并查集 (Union Find Set) 操作，将这两组集合做物理级别上的融为一体。瞬间，它们合并成了一个跨越整颗查询大树的超级等价族：{A, B, C, D}！

通过这种不断进行自我融合与扩张的动态打法，原本在语法树上只是零散字符排列组合的 WHERE 条件语句，在 PostgreSQL 的核心内存池中，被升维成了四通八达、信息密度极高的数据依赖拓扑网络。

2. 探视深渊：核心数据结构剖析

为了彻底弄懂这个体系，我们需要查阅 pathnodes.h 这个统御着无数路径信息的头文件，去看看这些结构体在 C 语言层面真实的定义样貌。

2.1 统御母体：EquivalenceClass

它相当于等价操作的总容器，掌控并指引着优化器的判断大纲。

typedef struct EquivalenceClass
{
    NodeTag     type;
    List       *ec_opfamilies;   /* 极重要的地标：这个等价类所绑定的 B-Tree 操作符族。*/

    Oid         ec_collation;    /* 字符排序规则 Collation (用于解决法语排序和中文排序等号判定不通用的基础问题) */
    List       *ec_members;      /* EquivalenceMember 列表：里面存放着属于这个家族的所有表列、常量及子计算 */
    List       *ec_sources;      /* 回源指针：生成这个大图论集合最初的、最“原汁原味”的 RestrictInfo 谓词本身 */
    List       *ec_derives;      /* 这是由主引擎向外辐射衍生成的用于各级 Join 匹配的约束条件包 */
    Relids      ec_relids;       /* BitMap：这个集合内所有的属性涵盖跨越了哪些具体的表 (RTE) */
    bool        ec_has_const;    /* 魔神之眼：哪怕有上万个子表达式，只要这个集合里混入了一个常量值，它就变成了 true！ */
    bool        ec_has_volatile; /* 是否存在易失的 (Volatile) 不稳定函数成员？如果是，则拒绝下发相关合并逻辑 */
    /* ... operations info ... */
} EquivalenceClass;

2.2 下位个体：EquivalenceMember

包裹在这个大家庭里的就是一个个具体的运算因子。什么样的节点能成为成员？通常是普通的 Var (代表表里的某一列)，也可能是完整的一个函数运算式 FuncExpr，亦或是 RelabelType（类型强转映射操作封装等）。

typedef struct EquivalenceMember
{
    NodeTag     type;
    Expr       *em_expr;         /* 这个成员实际对应的可执行表达式结构树，比如 Var(T1.A) */
    Relids      em_relids;       /* 该独立表达式依赖着哪些底层级表 (这对于推断 Join 的上下级顺序异常关键) */
    Relids      em_nullable_relids;
    bool        em_is_const;     /* 核心标志位：这个个体不是普通的列，它是一个带着具体化常量的字面值吗？ */
    bool        em_is_child;     /* 对于分区表操作尤其重要，用来指代基于分区拆分后的细分子节点 */
    Oid         em_datatype;     /* 该表达式求出该值的底层数据类型，用于强制校验匹配 */
} EquivalenceMember;

2.3 ec_has_const 与常量下推的碾压式特权

所有介绍 SQL 重写技巧和优化的教科书中，最底层的能力源泉往往都依赖于对常量的处理机制。 在 PostgreSQL 的等价类体系里，如果某一个具体的 Member 被判定设置了 em_is_const = true（比方说，业务在一段冗长的 SQL 中写了 T1.A = 100，100 被解析为一个带有 const 恒态属性的成员）。那一瞬间，整个包裹它的巨大的 EquivalenceClass 就像是被真理光环彻底附魔了：顶层的 ec_has_const 标志位会被直接置为 true。

因为一旦优化引擎确认了集合 {A, B, C, D} 在底层算术流转中其实“毫无实际变量计算意义”，这群看似花里胡哨的变量名词，在物理世界全都被具象化且绝对等同于常量 100！ PG 在后序规划中无论走到哪一步的算子——无论是多路 Join 进行哈希探测、深层子查询往内圈下推、还是做非常底层的并发索引扫描。只要能读到这个等价类的引用指针，它都能将针对这些巨大字段之间的动态匹配消耗，瞬间点石成金般转换为“直接去磁盘上找寻常数值 100 的确切记录”。并强制给其他关联成员赋予这个常量参数。 这便是深究 Cost-Based Optimizer (代价优化器) 中为何总是引以为傲的最高维武器——基于关系映射传递的横向常量下推 (Constant Propagation & Pushdown) 技术中枢基座！

3. 隐式衍生约束的魔法 (Implied Equalities)

光是建立起一套等价类模型其实并没有用，真正让 PostgreSQL 从众多依靠死板规则化干预行事的传统关系型数据库引擎中脱颖而出大放异彩的，是深藏在这背后于 equivclass.c 里的魔法工坊： generate_implied_equalities() 等一系列生成衍生约束条件的函数调用。

3.1 跨表谓词下穿衍生机制

举一个非常具象的例子。假设有应用前端依靠 ORM 代码生成器胡乱拼凑出了这么一句常见的检索： SELECT * FROM T1, T2, T3 WHERE T1.id = T2.id AND T2.id = T3.id AND T1.id = 99;

试想一下，在执行计划刚刚起草、T2 和 T3 还处于独立构思如何单表扫描的早期阶段。如果是在传统的非闭包体系引擎中，优化器要去扫描拥有千万数量级的 T3 表时是非常痛苦无助的——因为它压根不知道 T3.id 到底要去搜寻谁。它除了默默去承受代价高昂的 SeqScan 顺序全表扫描硬盘以外毫无办法，然后必须把读出来的成百上千兆数据生硬地抛到上层参与后面缓慢如蜗牛的 Join 过滤匹配。

但在 PostgreSQL 强大的 generate_implied_equalities_for_column() 函数机制背书中。由于 T1.id, T2.id, T3.id 以及那个常量值 99 全部落网被收容进入了唯一的巨型全域等价类。当 query_planner 调度到底层需要评估扫描 T3 这张表的最佳访问路径 (Paths) 时，代码会拿着 T3 表所独有的 Relids 当作钥匙，去向这个巨型等价中心发问祈求： “请问在这里面，有没有哪怕仅能由我 T3 一张表就可以单独享受匹配、不需要往上走依靠关联运算的隐性过滤推演条件（Implied Restrict)？”

等价类模块通过遍历，敏锐发觉自身不仅有常量 99，并且还有请求者的 T3.id。于是系统这如同虚空捏造出了一把尚方宝剑——自动凭空变出了一条原本在文字层面并不存在、但是在数学上绝对站得住脚的确凿过滤条件：T3.id = 99！ 随后，这个崭新生成的过滤条件参数会被作为推导限制被即刻下发，绑定成为了 T3 自己的单表专属扫描谓词。原本 T3 即将遭遇的那一波漫长的百万行 SeqScan 以及在内存中同前面表死磕合并运算的恶劣命运随之烟消云散。这一动作直接将其成本评估定性为了超轻微代价量级：不仅可以走索引，而且只需要在极其短暂的时间内针孔定位完成 I/O 的顶级访问模式——Index Scan (T3.id = 99)。优化器提速狂飙的大幕在这个节骨眼上拉开。

3.2 强前置互验与阻止“笛卡尔”核爆

退一万步讲，哪怕在这个逻辑运算链条上我们完全没有看到常委级别的常量出现，仅仅只是关联起了跨表的字段集合 {T1.A, T2.B, T3.C}。当优化器因为表的数据量选择率等等复杂原因估量，觉得有必要先将 T1 和 T3 来一场连接大作战执行 T1 JOIN T3 时（例如系统估算出即使盲连，两边返回的表行数也非常极小）。哪怕你回头审视那份极端的业务传入的原始 SQL 文本，压根根本找不到那俩表哪怕有丝毫一星半点的字段关联语句。

但神圣而强大的 PostgreSQL EC 抽象集合模块仍会一视同仁。由于它坐拥此三子于一室，明知三方连通有着等同的强逻辑背书，系统便会极其从容地通过派发操作，生成出一条隐性联姻线缆：创造出 T1.A = T3.C，并将其强行施加这二者初次结合的连接判断标准（Join Qual）！

这套机制几乎完美且不可破防地从代码根本设计上截断化解了一场原本必然演变为笛卡尔全铺积惨剧的糟糕连接爆炸隐患。当我们从这个宏大的视角去凝视晦涩生僻的等价类 (EC)，它存在的真正意义从不光是进行简单语法替换，它更是为了阻滞底层检索灾难、铺设规划指引路径和用以空间降维的最坚实的守护神与路标塔。

4. 与序列流向 PathKeys 的极高依附羁绊

如果我们想要精通于理解 PostgreSQL 内核反应堆里最高深的代计算法则（Cost 和代价比拼），就必须得先理顺困扰着最多数新手研究员的那条最生硬的死结： 判断或者指代某一条执行记录长河是否在这个阶段内部已经恰巧满足了你要求的排序规则配置时，它的终极判断信标（Beacon），永远不是去简单僵冷地对比诸如所谓的类似 ORDER BY Table1.TName 等几个简单的字符串名，它的底层引信是远远地锚定、并且绑定包裹着一整颗等价类！(指向 EquivalenceClass 的母体内存地址)。

这段千丝万缕的核心羁绊将在专门用来剖析排序内核的《PathKeys》深度核心篇章中大规模展开，但鉴于它们二者生死的关联性我们必须要在本节作一浅尝揭开。

为什么在物理的数据结构设计抽象里头，PathKey 链表的这个核心必须紧紧夹杂挂钩 EC 的指针向标而不是直接对应上一个什么 Var(A) 的变量位呢？ 假设现在系统在堆排查探的过程构建了一个包含 {t1.A, t2.B} 的局部 EC 家族。 目前，底层引擎派出了一个得力干将老老实实地从磁盘上沿着建立在 t1 的 B 树索引走了一轮查找（IndexScan）。由于检索特性沿着那棵树一直扫出来，这串喷涌而出的结果集先天在列 t1.A 上就是顺畅递增且有条理的了。这时刻这个路径会对外吐露它带有光环的一份证明报告册：它的 pathkeys 数据包说明了：“注意我手头这堆数据可是完美臣服并且符合那个极其著名的 EquivalenceClass{t1.A, t2.B...} 流向哦” 。

执行计划层层往上汇报。然而在终点站查询的最外层顶端遇到了一个极高的过滤巨石：你的主逻辑命令强要求系统在这儿汇总做最终结算排序：ORDER BY t2.B。 这会子，ORDER BY 这个引擎管控模组可是非常贪婪也极度渴望地去向查询主决策模块打听索要：“从下面那些底层兄弟一路干活抛给我的结果流里面，有没有哪一个特别机灵的小伙伴，它那条数据线的供货排向老早以前就按我想要求看的 t2.B 顺顺利利的整理分配过给大伙儿了？”

工程代码在此展现出了神迹。底层规划器绝不会死板僵硬要求查核里面是否真就包含了 t2.B 这一串字母的完全一致打标（字符串硬比对在这个档口铁定会导致出现“我底下是用按照 A 来规整出来的，这上边死命令说要核实 B 的顺位，风牛流马不相及，统统当作废处理强行大乱再重新大把内存全部重排梳理一遍序列！”的惨烈悲剧和性能严重错判）。而是拿出从底下沿着流程传向自己的“这个状态集归档的内存地址引用指针”，去跟自己上游规划里的渴求期望目标硬顶比拼：比对上了那个同样身为这个族长神圣统御者也就是同等价阶位的 EquivalenceClass 实体指向指针！

一旦引擎探知下面底层那些数据特征关联着的 EC，竟然 及其巧妙合拍地等同于自己所需要拥有的那个实体指针！这两套体系简直是宛如雷击瞬间拍板互相对上眼：“那是因为在早前一开始 SQL 过筛阶段我们全伙计就了解了 t1.A = t2.B 等式的永恒不变关系。我此刻验收到下方提交上的这份文件其实已经完美踩着贴着 t1.A 是顺位排列整理的结果了。如此一来从物理计算普遍真理的原则上来谈，在当下这一瞬间这流体、这份顺序态，早已就绝对并且纯天然地履行等同遵从了我亟需最迫切想要见到的 t2.B 排好顺序的大前提啦！”

这就意味着，只要它在底下随手捡便宜借用了在一个小小列名 A 身上一条微不足道低消费的索引遍历跑法，竟然可以通过一路往上通过跨阶层的核心概念概念转移包裹魔法替换法，神乎其技的省跨越了大表和执行节点之间的巨大鸿沟。顺带来连在最尾端呈现全部表格输出的时候连那些最庞大要吃掉好几个 G 运存排序的超巨型的全排列重排内存杀手节点都能省得一去不回！这种在 RDBMS 执行领域宛如神出鬼没般的维度换挡置换的空间转移技巧。对现在那些随便做几个粗暴生硬匹配依赖玩具解析层去优化的粗制型引擎看来，近乎如同不可跨越、绝不明白的异星位面技术级打击。这代表着一代代 RDBMS 先驱设计大师几十年代码打磨结晶下来的算法思想极意巅峰——也是使得它这只大象至今立于不败之地无可替代的关键壁垒内核深渊所在。

路径序列与排序核心密码 (PathKeys)

几乎所有关于数据库性能优化的面试题，最后都会绕回一个经典命题：“为什么加了索引，ORDER BY 或者 GROUP BY 变快了？” 在执行引擎的表象逻辑看来这似乎是顺水推舟的常理，但如果要在拥有极大复杂组合的 PostgreSQL 代价优化内核（CBO）中完美且统一地实现这一抽象机制，就需要一种能够统一度量并且伴随路径推演任意流转的通用通货。

在 PostgreSQL 18.1 中，这套跨越所有关系代数操作、用来锚定和流转排序特征的通货，名叫 PathKey (路径键)。相关代码的核心座落于 src/include/nodes/pathnodes.h 和 src/backend/optimizer/path/pathkeys.c。 如果说不懂“系统状态模型”就没做过微内核，那么不懂 PathKeys 的设计与传播机理，你就无法真正看懂优化器在 add_path 里做出的那些惊心动魄的剪枝取舍，也绝对无法理解为什么某些由于整体代价极高的看似“破烂”执行计划，反而会在复杂查询的最后关头被 PG 奉为至宝。

1. 结构大起底：什么是 PathKey？

优化器在多表迷宫中评估一条物理路径（比如一次走索引的大型扫描、或者某个并行的哈希连接）时，不仅需要看它“花了多少 Cost”、“预计吐出多少 Rows”，还必须在这条路径的户口本上，精确且持久地标注出这股数据流携带的元素排序特征。

这套特征在源码中由 List *pathkeys（一条链表）来承载。而它的基础粒子组件，即为结构体 PathKey：

typedef struct PathKey
{
    NodeTag     type;
    EquivalenceClass *pk_eclass;   /* 极度震撼：它不指向具体的列 Var，而是指向一个覆盖全部域的等价类！ */
    Oid         pk_opfamily;       /* 排序操作符族 (B-tree opfamily OID)，比如告知系统这是按数值大小排还是按字符串拼音排 */
    int         pk_strategy;       /* BTLessStrategyNumber (升序 ASC) 还是 BTGreaterStrategyNumber (降序 DESC) */
    bool        pk_nulls_first;    /* 终极细节考量：NULL 值处理在引擎级到底是悬浮排最前面还是沉底垫后 */
} PathKey;

1.1 为什么主体心智必然要包裹 EquivalenceClass？

在上一个深度探索附录《等价类 (EC)》中我们埋下了一个伏笔：身为 RDBMS 处理领域的巅峰瑰宝，PG 极为高超的设计使得 PathKey 的最核心属性 pk_eclass 直接硬链接在了 EquivalenceClass 集合框架上。

试想这样一个极度频繁的工业级数据提取场景：SELECT t2.info FROM t1 JOIN t2 ON t1.id = t2.uid ORDER BY t2.uid; 在优化器构建最底层的规划草图时，系统发现 t1 表有一根在 id 字段上的巨无霸 B-Tree 索引聚集簇。于是，优化器贪婪地探索出了一条顺着索引摸出的极速路径 IndexScan(t1) 返回单表局部数据。 这条底层路生成的初始 pathkeys 护照上骄傲地写着：“注意，我送出的这批源头数据是严格基于 EquivalenceClass{t1.id, t2.uid} 而井然排序的哦！”

当数据流经过某一种能原封不动保留顺序的连接算法（如普通模式下的 NestLoop），浩浩荡荡向上游输送，最终抵达了引擎最上层、需要执行全局决算的 ORDER BY t2.uid 节点站。

假若当年烂俗一些，让 PG 设计出的 PathKey 傻乎乎地仅用一个字符串列名映射来作为载体（例如：pk_var = t1.id），那么顶层节点在验收这批流出来的水柱时，拿起自己客户要求的判决词 order_var = t2.uid 拿去字符串匹配碰死角：“天啊，底层上来的是 t1.id 的账，客人要的是 t2.uid，完全风马牛不相及，给我统统退回！” 于是它悲愤地迫使整条管线停更，强行将几百万数据拦停并扔进巨大的 Sort 内存重排栈，引发恐怖算力海啸。

但是，正因为 PG 架构中的 PathKey 绑死的是极度抽象化的等价类集合！ 顶端审判官只需拿客户需求 ORDER BY t2.uid 背后代表的 EC 内存句柄，去和底层千里迢迢报送拿来的那个 EC 指针一碰。刹那间同气连枝完美吻合 (那正是因为 t1.id = t2.uid 的存在，让它们在优化器建立图基之初就被合二为一吸纳同源)。判定官惊呼一声放行，极其平滑且不增兵刃地直接帮引擎省切掉了最为昂贵的内存排序重排局！

这种高度内聚且从根本上解耦掉了具体表名字段耦合度的方法论模型，是 PG 在无数变幻莫测的关系运算域中得以制霸的极道法宝。

2. 需求侧的大一统：规划器的通用排序货币化

我们身处于一条千回百转、嵌套复杂的 SQL 分析大河中，可能横向上混杂了无数个试图干预宏观排序的业务操作符：

  • 终端发出了 ORDER BY a, b 要求最终强规整顺位。
  • 聚集指令下了 GROUP BY c, d 在启动极速 GroupAgg 之前强制要求聚类有序。
  • 过滤节点加了 SELECT DISTINCT e 企图通过底层排序来实现极速跳跃滑动去重。
  • 大名鼎鼎的 Merge Join (t1.f = t2.f) 则强迫左右大营的输入流必须按双边关联的等同键绝对一致序列接壤。

对于老手工作坊式、缺乏远见的其它劣质 SQL 引擎核心，很有可能需要为了迎合这上面所有的操作符，各自手编出一套又一套冗余的“排序核查对账单”。 但在 PG 那神级铺陈的 pathkeys.c 架构中，所有这种来自天边的局部排序请求，无论初衷为何，都会在预检门神阶段被强制接引调用如 make_pathkeys_for_sortclauses 的翻译官解析函数，彻底被碾平、收容并直接等价兑换为唯一标准流通物的 PathKey List 链表串序列。

在这之后，自大优化网络全面铺开的那一顷刻。优化器这个中央智脑，仿佛完全丧失并且再也听不懂那些所谓的 ORDER BY 或者 GROUP BY。它在判断下层分支与上游组装口是不是符合插接时，唯一遵循并只认这套被称之为 pathkeys 的内通国书机制。

3. 供给端命运的长河：PathKeys 的诞生、传承与覆灭

如果说顶端需要的是标准化的规章，那么底下奔跑着的一个个供节点就是这本 pathkeys 血脉的衍生与承接之所。 优化器在创造不同的物理生成计划（Paths）时，对前人遗留财产 pathkeys 所表达出来的接纳态度，真可谓是人性百态截然不同：

3.1 源氏的血脉：底层索扫描管线的原始挂载

最纯粹的 pathkeys 发源地毫无疑问孕育生长自 B-Tree 的原生家族。定位到核武模块 indxpath.c 内部的 build_index_pathkeys： 当检索针刺入一个定义横跨聚合列 (X, Y) 上的索引扫描簇结构时，PG 引擎核心会小心核查其构建游标轨迹。并且产生出一个将 X 和 Y 的成员紧密拼接的衔接 PathKey 双向队列 List，极度慎密且富有骄傲地赋予在即将上抛提交的一条叫 IndexPath 的执行路径表头上。 若另一边是个走大老粗路线通过盘面推的 SeqScan，那不好意思，该线路上生成的归口登记簿对于 pathkeys 会留出一个干脆利落的 NIL (完全空针)，宣告着它向外派出的仅仅是一堆不受管教、杂乱无章的散沙数据团。

3.2 冷血的屠夫：Hash Join 的破壁行径

当水流汇入双表交战关联大区，各种 Join 节点的算计本质展现得淋漓尽致： 身为性能王者也是内存大户的 Hash Join, 它的作风极其粗犷奔放。它压根不去鸟底下左臂右膀的外表输入流是不是费尽心机按某某神圣宗庙序列编排过的。由于它的底层机巧是一旦将小表直接倒进入哈希池重组开桶，随即拿大数据集展开猛烈的散列无序激化碰撞验证。这所有关于时间向度的历史积淀会在碰撞出水的一瞬间完全崩散。 因此代码法则极其残忍果断：任何基于核心构建法 hash_inner_and_outer 生成派生的路径结晶，它的光环继承簿 path->pathkeys 在合并的那一刹那刻，必定被无情地洗白清零成为 NIL。

3.3 恪礼的承传者：NestLoop Join 嵌套轮回

嵌套循环联合 (NestLoop) 的打法则极其恪守封建式的伦理教条。 由那最外环提供一条数据，内部则完整恭顺地随转跟进圈环一再往复过滤匹配。这暗示出什么宏图天机？ 外环原本提供过来的序列起伏节奏，在吸纳内表拼接组合出新的大型全息行实体后，其整体的推流首要序列阵仗竟然神奇绝妙地没有丝毫偏差！因此，在这个联合构型缔结算子路线时，优化器会温柔地调用 build_join_pathkeys 把外表（Outer Rel，左子表）持有着的一条条 pathkeys 金字令箭原封不动、极其虔诚完整地誊抄接手沿承下来，去向后续的更猛烈巨浪搏杀冲刺！

选择率与代价引擎的数学基座 (Selectivity & Cost Model)

如果说执行器节点 (Executor Nodes) 是在战场上冲锋陷阵的铁血部队，那么 PostgreSQL 核心的 代价基调引擎 (Cost-Based Optimizer, CBO) 就是在这之前于作战统帅部里进行图上千百次沙盘推演的灵魂中枢。 它一切的评估、一切生杀予夺的最终衡量指北，全部依托于两个底层基石机制：选择率预估 (Selectivity) 猜测系统会返回多少行数据，以及 代价计量 (Cost Model) 折算出这批数据在磁盘和内存中将会耗费多少系统滴答。

在 PG 18.1 中，这套左右引擎命脉的公式算法与核心常数，沉睡于 src/backend/optimizer/path/costsize.c 与 src/backend/utils/adt/selfuncs.c 等文件中。 本文将穿越表象，带领你深入这个堪称统计学与经验法相互融合的魔幻数学枢纽域。

1. 统计学透视：选择率 (Selectivity) 的精准测写

所谓 Selectivity (简写为 sel)，是一个介于 0.0 到 1.0 之间的浮点数，它表示某一个过滤条件 (Qual) 有多大的“卡口率”，能够允许整张表百分之多少的记录透射过去。 计算 sel 的一切前提，都基建于 ANALYZE 命令收集并写入到 pg_statistic 表中的系统元数据。

1.1 单列特征：MCV 与直方图的双峰分布

如果数据都是均匀分布的，那数据库根本不需要复杂的收集机制。但现实世界中，业务库里的数据通常符合长尾和聚集极化（如某平台大多数用户注册年龄是25岁左右，但夹杂了一堆 18 和 50岁的稀缺词点）。 在代码 selfuncs.c 里处理操作符如 col = 'A' 时：

  • 首先系统会查验目标列的 MCV (Most Common Values) 高频词表数组。如果查到 'A' 就在高频数组里，它会直接精准地抽出记录好的频率（可能占 32%）作为 sel。
  • 如果是 col > 100 这样的带域检索，MCV 就捉襟见肘了。这时系统转而启用 Histogram Bounds (直方图切面界限)。PG 将排序好的表数据强行切分为数百个均等记录数的“桶 (Bucket)”。通过线性插值法探测 100 横跨了几个桶，计算其面积积分来得出精确 sel 预估值。
  • 最后还要叠加一个 Null Fraction (空值率)，直接把所有 NULL 的占比扣掉作为基础衰退系数。

1.2 维度的灾难与破局：扩展统计信息 (Extended Statistics)

这是许多老旧数据库最遭人诟病的盲区，在代码侧集中于 clauselist_selectivity() 的连乘缺陷。 如果用户写了 WHERE city = 'Beijing' AND country = 'China'。传统的概率计算遵循特征独立假设定理，也就是默认这两个条件的出现是不相关的。 假设系统查表得知：city='Beijing' 的 sel = 0.05，而 country='China' 的 sel = 0.2。 如果引擎采用朴素的 sel_total = 0.05 * 0.2 = 0.01 的瞎乘算法，就会得出极其扭曲的结果，认为双重匹配的数据极小，以至于被诱导去选出 NestLoop Join 这种极其错误孱弱的计划分支。殊不知，现实中凡是北京的由于地域从属必定全都是中国领地这二者拥有强绑定聚集概率！

为了挽救这种维度塌缩，PostgreSQL 引入了跨列的 CREATE STATISTICS 模块进行救场。当你为 (city, country) 建立扩展高维相关性依赖 (Dependencies) 或多维 MCV 组后。系统就会在多条件连乘之前判定这俩词的联动特征相关系数值；打破“相互独立”的连相乘魔咒，挽救整个底层评估树的坍塌偏斜。

2. 代价模型 (Cost Model) 的双轨评议机制

如果你查看过 EXPLAIN 输出的括号第一手信息，永远会面对类似 cost=10.00..1050.25 这两个用下划线相连接的神奇数值。 这代表着优化器世界里一套极其严苛细分的基于双轨维度的总账计算法：启动代价 (Startup Cost) 与 总代价 (Total Cost)。

2.1 平行维度的角力

由于数据库被不同特征的长短命连接或者上层指令要求压榨，PG 对任何一条候选 Path 都在这两个账本上同时算分。

  • Startup Cost (初始启动沉没代价)：吐出前列第一行 (The First Tuple) 目标匹配数据前，当前算子需要憋气阻塞筹备花去的时钟值。 比如 SeqScan 它的启动代价为 0 （随时随地硬盘打来就开始往外流）。 而如果用 HashJoin，它的发车时间极长！你外表得等我内表慢吞吞把全量记录翻完并在内存建好几百万 Hash 数组的桶池完毕后，我才能开放外表探针去撞线吐出第一条结果。因此它启动代价巨大。

  • Total Cost (穷尽扫尾总代价)：完完整整把包含这符合条件的最后一万行数据全发送完毕、关闭管口所需要的总和分。 比如 SeqScan 这里就非常要命了，哪怕你只要其中一条也是要扫描全盘去碰的，总价奇高。

这双轨的博弈在于，如果上级传进来了诸如 LIMIT 1。代价函数在汇总结算最优解时，权柄会疯狂倾向于带有极低 Startup Cost 的算子。即使它的 Total 极其难看，系统也会果断采纳，以此规避诸如大排序大哈希表建造的愚蠢行径。

2.2 操控大局的魔力数字因子 (Magic Numbers)

一切 Cost 都是虚构的浮点打分单位。而这一切数字放缩的源头，来自 costsize.c 中被写死的并允许重置的 GUC 常量参数，这是所有高端 DBA 进行参数魔改调参的圣碑之地。

  • seq_page_cost = 1.0 (默认)：顺序读取一块 8KB 磁盘页的参考耗时被定义为基本单位 1.0 的黄金标尺。
  • random_page_cost = 4.0 (默认)：标志着发生磁头跳步的随机磁盘检索，因其寻道时间通常被认定要比顺读惩罚大 4 倍以上。这就是为什么明明有索引，然而系统却嫌弃随机读太分散而果断放弃走索引、转投入全表顺序扫面的终极解释阀门！在全 SSD 高速时代，高玩通常会将其砍杀降至 1.1 来逼迫系统疯狂走查寻索引分支。
  • cpu_tuple_cost = 0.01 与 cpu_operator_cost = 0.0025：标志了主板上 CPU 大脑去计算一条 WHERE a=1 这样内存处理操作相比读取磁盘（耗费 1.0）到底有多么的“轻如鸿毛”。

3. 代价决断前线：核心算子估算探秘

在知悉了底层宏观参数之后，让我们潜入 costsize.c 去看看具体算法节点是到底怎么花起这笔账面的。

3.1 扫表的代价：cost_seqscan vs cost_indexscan

这是整个系统内每天被调用最多数以千万次的基础算盘。

  • 对于全表扫 (cost_seqscan)： 公式直白粗暴，毫无虚空魔法： 总代价 = (目标表格所占的总 Page 数目 * seq_page_cost) + (表格行数 * cpu_tuple_cost) + ...

  • 对于索引扫 (cost_indexscan)： 这里极其反人类且充满了经验曲线法则。因为索引树一旦跑起来，你去找硬盘上的数据通常是非常恐怖离散化的。 如果你的主表物理存放与这颗 B 树由于长期增删改查导致碎片化四散（即 Correlation 特征聚簇相关性 趋近于 0），如果强走索引拿十万条记录，你可能要发起非常昂贵的十万次跨磁头大随机跳跃，random_page_cost 能把计分算盘直接干穿爆炸报废去到几千万成本。 但此时代码也绝不死板，cost_index 中潜藏了一套名为 Mackert-Lohman 的平滑预估公式曲线，去猜测这堆离散请求究竟会命中有多少页可能由于已经在 LRU shared_buffers (内存共享缓存池) 里面而省去了回源物理盘，从而打个猛烈的折扣把成本压制下来防止系统完全不走它。

3.2 兵家大关大争之地：关联的代价估量

  • Nested Loop Join (cost_nestloop)： 被称为无脑穷魔的梦魇。其计算方式为：让 左表外循环数据总规模 (outer_rows) 全额连乘 右表被当作单条被压榨的内循环单趟代价 (inner_cost)！ 哪怕最外层只读 1 万行，你的内部索引扫单次只是极小成本 50。在指数级复利连乘叠加下也是非常刺目的分量。但这恰由于其具备 Startup 为接近 0 的绝杀特性以及几乎没占用额外独立运存空间的优点而让它常立于高并发不败之地。

  • Hash Join (cost_hashjoin)： 其计算的深沉点全部聚焦在一开始的 Startup_Cost 构建蓄水阶段。 系统会精密计算内表 inner_rows 的总体积有多大，需要使用多大的 CPU 指令资源 cpu_operator_cost 去遍历、计算每一个数据单元的哈希根并放入工作内存 work_mem 内预装配。 最要命的分叉评估发生在这，一旦优化器察觉到给拨的 work_mem 太抠门小气存不下全部内表哈希链列。代价算口会在原本仅仅只吃 CPU 的计算里，凭空掺入一大笔极其恶心和高居不下的 disk spill penalty（刷盘大惩罚机制量） 开销，并且成倍地因为需要在探针碰撞期间于磁盘多批次重置读取而拉高 Total Cost。从而向顶层告警，规避或强制触发要求走别派系统连接路线以保平安顺遂。

经过如此成百上千次精密到苛刻毫厘偏差的小分支节点代价评估积累并合卷对账对比。PG 代价优化的王牌引擎得以从最少几百上千条乃至天文数字交错般混乱的查询可能路径网里，在短短不足几毫秒之内，犹如雷达盲区索敌指引核爆一般，斩钉截铁不差分毫地敲定并拿出了那条绝对通向最优解的绝杀黄金制胜长路。
