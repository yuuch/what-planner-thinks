# 等价类 (Equivalence Classes, EC) 深潜

在 PostgreSQL 优化器的宏伟蓝图中，无论是进行单表扫描条件过滤，还是复杂的 Join 规划寻找匹配路径，亦或者是 Order By 寻找可用排序索引，底层的心跳都离不开一个至高无上、贯穿全局且极其精妙的抽象概念——**等价类 (Equivalence Classes, 简称 EC)**。

由于关系型数据库中存在着大量形如 `A = B AND B = C AND C = 5` 这样的连环连接与过滤谓词。如果优化器只做字面（Syntax）意义上的表象树解析，它就只能机械地做 A 与 B 关联、B 与 C 关联。
而一旦系统引入了“等价类”这一基于离散数学“传递闭包（Transitive Closure）”的等价集合抽象机制，优化器就能犹如开启了上帝视角一般，瞬间在内聚的数据世界中刻画出一张全连通的无向图。它将毫不费力地推断出 `A、B、C` 同属一个命运共同体，且它们运算时的真值必定全为常数 `5`！

本文结合 PostgreSQL 18.1 最新源码（核心代码落座于 `src/backend/optimizer/path/equivclass.c` 与 `src/include/nodes/pathnodes.h`），对这个常被外围技术博文一笔带过、却在 PG 底层翻云覆雨的“无名架构英雄”进行庖丁解牛般的硬核拆解。

---

## 1. 原点：EquivalenceClass 的生命周期与无向图抽象

在 SQL 到达优化器被摊平拆解后（例如在 `initsplan.c` 中的 `distribute_qual_to_rels` 路由节点中），系统遇到 `WHERE` 或 `ON` 条件中的各种二元布尔表达式时，便开始了它的猎杀判断。

当规划网络中捕获到一个诸如 `X = Y` 的表达式时（严格地说，是该操作符使用了能够支持 B-Tree 可排序能力、且在 `pg_amop` 目录元数据中被声明了具有哈希相容与等价语义的 `=` 操作符的 `RestrictInfo`），它并不会简简单单把这个连词作为普通的下推过滤处理方案，而是立刻中断通用流程，转而触发一个专属的神圣构建函数：**`process_equivalence()`**。

### 1.1 从离散条件到传递闭包的疯长
`process_equivalence(PlannerInfo *root, RestrictInfo **p_restrictinfo, bool below_outer_join)` 的核心职责是由点到面的进行拓扑网编织：

1. **新建与收容**：假如目前是白纸状态，系统见到了首个等式 `X = Y`。它将立刻调用对象分配器，新建一个 `EquivalenceClass` (EC) 实例对象，并将 `X` 和 `Y` 这两个实体表达式各自包装为 `EquivalenceMember` 成员塞入集合链表。
2. **顺藤摸瓜的扩容**：系统读到了下一个紧随的谓词 `Y = Z`。进入代码流时，逻辑引擎开始检视手中已有的所有 EC 集合，它骇然发现表达式 `Y` 已经栖身于之前建立的某一个 EC 之中了。此时系统不再愚蠢地去新建集合对象，而是利用传递律边界直接将新词条 `Z` 丢进这个包含 `X, Y` 的已有等价类里，形成包含 `X, Y, Z` 的更广袤集合。
3. **两个平行宇宙的大碰撞大合并**：复杂查询中极为常见的名场面。假设系统在处理某颗子查询子树时推断产生了一个局部 EC `{A, B}`。而在处理外围结构时得到了另一套 EC `{C, D}`。此时，主查询里突然暴露了一条跨域联通管线 `B = C`。
   见证奇迹的合并时刻上演！优化器在搜寻中直接撞见了这两个独立的集装箱，立刻拉起极其底层的操作，将两者做物理级别的并查集糅合（Union Find Set），这二者瞬间归化为一个横跨超大子树的超级等价族：`{A, B, C, D}`！

通过这种永无休止的自我动态融合扩张逻辑，原本在文本解析树层面仅只是一条条零散冰冷字符串的 Where 条件，在 PG 的中心内存池内被高度升维化成了四通八达、极具信息密度的“强连通分量”。

## 2. 探视深渊：核心数据结构剖析

为了彻底明晓这个体系，我们需要直视 `pathnodes.h` 这个统御千层路径的头文件，看穿这些隐秘结构体真实的 C 语言身躯。

### 2.1 The `EquivalenceClass` (统御类母体)
它是整个等号抽象域的总容器，掌控并指引着优化器的动作大纲。
```c
typedef struct EquivalenceClass
{
    NodeTag     type;
    List       *ec_opfamilies;   /* 极重要的地标：这个等价类所绑定的 B-Tree opfamilies 集。*/
                                 /* 若没有底层声明等价传递性的 OpFamily 支持，整个推导直接非法。如 int4_ops */
    Oid         ec_collation;    /* 字符排序规则 Collation (解决比如法语词典比较和本地词典比较等号非通货的基础问题) */
    List       *ec_members;      /* EquivalenceMember 列表：里面躺着属于这个家庭的所有表列、常量及子计算 */
    List       *ec_sources;      /* 回源指针：生成这个巨无霸图论集合的最初的所有“原罪” RestrictInfo 谓词子句本身 */
    List       *ec_derives;      /* 这是由主引擎向外辐射衍生成的用于各级 Join 匹配的约束条件包 */
    Relids      ec_relids;       /* 这个集合内的属性总共横跨涉及到了哪些基本计算表(RTE 的集合 Bitmap) */
    bool        ec_has_const;    /* 魔神之眼：哪怕有上万个子表达式，只要这个集合里混入了一个常量值，它就变成了 true！ */
    bool        ec_has_volatile; /* 是否存在易失态不稳定函数成员，如果是将拒绝下发合并... */
    /* ... operations info ... */
} EquivalenceClass;
```

### 2.2 The `EquivalenceMember` (下位个体细胞)
包裹在这个大家庭里的一个个具体运算因子。谁能上位成为成员？它常常是普通的 `Var`(表某列的提取节点)、也可以是一整个 `FuncExpr`，或者是 `RelabelType`（类型强转映射包装等）。
```c
typedef struct EquivalenceMember
{
    NodeTag     type;
    Expr       *em_expr;         /* 这个成员真正的可执行表达式结构树，比如 Var(T1.A) */
    Relids      em_relids;       /* 该独立表达式依赖着哪些底层级表(对于推断 Join 的上下级顺序异常关键) */
    Relids      em_nullable_relids;
    bool        em_is_const;     /* 核心标志位：这个个体不是普通列运算，它是一个带着字面具体常量实体的值吗？ */
    bool        em_is_child;     /* 对于分区表操作尤其重要，用来指代基于分区拆分后的细分子节点 */
    Oid         em_datatype;     /* 该表达式求出该值的底层类型，用于强校验匹配 */
} EquivalenceMember;
```

### 2.3 `ec_has_const` 与常量下沉宇宙的碾压特权
所有的 SQL 重构教科书中对于条件极简和优化的最底层源泉便依赖于常量的处理能力。
在 PostgreSQL EC 体系里，如果某一个具体的 Member 被判定设置了 `em_is_const = true`（比方说，业务在海量 SQL 某句不起眼的位置写了 `T1.A = 100`，100 被解析为一个带有 const 恒态属性的成员），整个包裹它的巨大 `EquivalenceClass` 就像是被真理光环彻底镀上了金：`ec_has_const` 指示器被瞬间置为 `true`。

因为一旦优化引擎洞穿了集合 `{A, B, C, D}` 在底层算术流转中其实“毫无计算意义”，这群花里胡哨的名词全部都在物理世界里具象且**绝对等同于常量 `100`**，PG 后来无论行至计划图谱的哪个偏远深海角落——甚至挖到诸如多路 Join、子查询下探内圈、或者是极为关联的底层并发索引扫描处，由于 EC 引用指针的存在，它都能将针对大字段比值的过滤消耗，瞬间降魔般替换或叠加为“径直奔向硬盘找寻 `100` 的确切记录”以及对其它成员强行赋常参数！
这即是传说中令 Cost-Based Optimizer 代价优化器引以为傲、威力无比的 **基于关系映射传递的横向常量下推 (Constant Propagation & Pushdown)** 技术中枢基座！

## 3. 隐式横跨限制项的衍生魔法 (Implied Equalities)

空手持有一把神器 EC 其实毫无建树，真正让 PG 从其它那些老旧、依靠呆滞规则运作的古董级 RDBMS 优化器引擎中脱颖而出大放异彩的，是位于 `equivclass.c` 里的魔法工坊： **`generate_implied_equalities()`** 函数群落族。

### 3.1 跨表谓词下穿衍生机制
举一个非常具象的示例。假设有前端 ORM 机器胡乱拼凑、生成了这么一句呆傻但普遍存在的检索：
`SELECT * FROM T1, T2, T3 WHERE T1.id = T2.id AND T2.id = T3.id AND T1.id = 99;`

试想在执行计划构建的破晓之初，T2 和 T3 仍然处于独立扫描状态构想下。如果在传统的非闭包体系里，优化器在扫描 T3 这张拥有千万级数据的表时是异常痛苦无助的——因为它不知道 T3.id 到底要搜谁。系统除了默默承受一场代价几百万的恐怖全表堆数据 SeqScan 扫描以外别无选择，然后把读完了全部硬盘后的数据传送到上层参与那虚空慢如蜗牛的 Join 过滤去。

但在 PG 有了强有力的 `generate_implied_equalities_for_column()` 函数机制背书下。由于 `T1.id`, `T2.id`, `T3.id`, 以及神祇般的那个字面常量值 `99` 全部落网被收容进入了唯一的超级大联盟等价类里。当 `query_planner` 调度进入评估处理最基础最底层、属于扫描 `T3` 这张表的最佳路径 (最佳 Path) 时。代码会拿着 `T3` 表所独有的的 Relids 前去向这个巨大中央等价类发问寻求祈祷：
“请问在这里面，有没有只由我 T3 一人就可以单独享用、不用外包给上层运算的隐式提成条件（Implied Restrict)？”

EC 模块看到自身有着真理化身常量 `99` ，并且该寻宝者成员库里带有 `T3.id`。系统这如同虚空捏造出了一把惊雷破魔巨刃——变出了一个物理上不存在但在数学上绝对确凿的过滤条件：`T3.id = 99`！
随后，这个崭新衍生出的参数被当作为限制下发，立刻强行绑死作为了 T3 自己的单表扫描专属谓词。原本 T3 即将遭遇的那一波漫长的百万行 SeqScan 后再用 HashJoin 与前世诸表纠缠的恶劣命运烟消云散；直接被优化器评估认定降级为了一个超轻微代价、甚至只需要在毫秒级纳秒级内利用磁盘 IO 完成针孔定位的极强操作：`Index Scan (T3.id = 99)`。引擎的加速狂啸在这一刻便拉开了序幕。

### 3.2 强前置互鉴与阻止星系爆炸
退一万步，哪怕在这条闭包链上我们完全找寻不出常委常量的存在，而只是普通跨域的集合 `{T1.A, T2.B, T3.C}`。当优化器由于基于选择率决定尝试去赌一把“先将 `T1` 和 `T3` 大战连表做 `T1 JOIN T3` （因为 T1与 T3 表极小）”，尽管你回头审视那极为脑残的业务级原生传入的 SQL 脚本根本找不到写出了二者的丝毫正交连通性。但神圣强大的 EC 抽象集合模块会一视同仁。由于它坐拥此三子，且知晓相互依仗等价，通过同样的衍生派发操作，它极度从容地给出了一条隐含联姻：生成了新过滤子句 `T1.A = T3.C`，并将它强压成这二者结合的连接条件（Join Qual）！

这几乎完美不可破地从根源上截断并化解了一场原本必然演变为笛卡尔全铺积惨剧的查询银河星光爆炸。在这个层面去宏观凝视，庞大晦涩的所谓 等价类 (EC)，其存在的意义与其说是提供语法变换，**不如说是为了阻止底层搜索决策大厦崩塌而矗立起用以来指引降维、铺设最优计算捷径的最高神圣路标塔。**

## 4. 与 PathKeys 的极度伴生依附羁绊

如果我们想要精通 PostgreSQL 核反应堆里最高深的代价机制（Cost 与 Sort 比拼），就必须弄懂最令人初级研究员烧脑反胃的死结：
**判断或指代某一条执行记录长河是否在局部已经满足排序规则，其终极判定信标（Beacon），从来不是系统死死挂着类似 `ORDER BY Table1.TName` 具体名字字母的烂铁片，而是将引信遥遥指引并锚定着一整个 `EquivalenceClass` 母盘内存地址针！！**

这段孽缘将在下面专门用来剖析排序机制的《PathKeys》深度附录篇章中大规模延散，但在此我们只能浅尝揭开一角羁绊。

为什么在物理抽象上，`PathKey` 里核心紧紧裹夹的非要是 EC 的指针，而不是直白的类似 `Var(A)` 的节点标识？
假设现在系统在堆排处理的过程中得到了一个 EC，集合里密密麻麻打包好了 `{t1.A, t2.B}` 等变量族群。
现在，底层引擎用执行器去老老实实地从磁盘扫过某一个建立在表 `t1` 表的某一列上的出色的 B 树索引（`IndexScan`）。由于它是走的左子树遍历扫出来，这串冒出来的流结果先天就是在自身 `t1.A` 列上严格物理递增排好了序的。此时代表该次执行结果产生的相关报告：其 `pathkeys` 的数据位标志出，“这份数据的流向已经契合并且臣服于这个涵盖着 `t1.A` 与 `t2.B` 等的专属巨大集合(EC)” 。

执行层层上抛。突然，系统撞墙上了一块极高的拦路巨石：查询顶层需要处理一场非常浩大排面的重聚合或渲染限制，例如需要：`ORDER BY t2.B`。
这时候，`ORDER BY` 引擎模块非常贪婪也极度渴望地去向主规划器打听：“目前流到我手里的全部执行底层数据流途径规划中，有没有哪个老好人、哪条线路是老早就按我想看的 `t2.B` 精准地分配排列过了呢？”

工程代码层面的神迹在此呈现：这底层的优化器它不去刻板傻乎乎地核对是否含有 `t2.B` 这几个僵化的列名 ASCII 缩语匹配（字符串硬比对机制如果这么干肯定得出 “你底下是用 A 排序的，我不管它上面要用 B 排的，统统推翻作废进内存去强制重新洗牌”的惨烈错判结论）。它会拿出从下而上承接流转所指向来的“内存地址”，去和自身渴望目标去碰：比对起这个神圣统御者的 `EquivalenceClass` 引用指针！
引擎一旦探知那底层带有的特征指向关联的 EC **极其巧合与自己期望需要拥有的**竟是截然相同的一块实体指针空间！
这两者系统宛如闪电一拍即合大彻大悟：“因为早在 SQL 初筛阶段我就得知了 `t1.A = t2.B`，且我确认了此份源数据已经是绝对死死踩在 `t1.A` 顺序上了。那在广义物理真理学上讲，它在这一刻、它所承载的排列状态就绝对等同于天然的顺次完美的遵从了我所亟需渴望的 `t2.B` 的升序要求了！！！！”。

这意味着，底层简简单单在小列 A 借道的一条便宜到极点的索引扫法，竟通过层层向上在更高维度的概念转嫁洗包，奇迹般地跨越表之鸿沟省掉了另一端为了展现大表顶层全表必须历经巨大内存空间的大排序算子的屠宰重压。这种神出鬼没般的维基换挡式空间优化对那些依赖所谓“傻瓜硬绑基于字符串对比映射表” 的常规玩具级别的 SQL 解析器来说，几乎不啻于降维空间核武打击；亦或者这就是人类设计 RDBMS 代码几十年来所结晶的算法巅峰——那些根本让试图通过几本快餐优化书藉浅尝辄止的用户，去望而生畏无法攀越的暗星孤峰罢了。
