# 遗传启发算法 (GEQO, Genetic Query Optimizer)

在前一章中，我们深度领略了 PostgreSQL 自底向上的动态规划（Dynamic Programming, DP）算法。这套精密至极的机制，誓要在错综复杂的表网中抠出一条理论上绝对完美的执行路线。

但这套机制并不是无底洞。即便是启用了启发式剪枝，DP 枚举排列的开销同样也会呈现不可阻挡的指数级大爆炸。倘若你在超大型数仓的复杂视图里牵连了上十几二十张表（超越 `geqo_threshold` 默认值 12 的警戒线），如果非要强摁着 DP 去穷举到底。那么可能到最后，**光是把“计划算出来”所花死等的时间，就足以超过去硬表硬扫实际跑完总时间的好几百倍！**

这时候，就轮到 **GEQO (Genetic Query Optimizer 遗传式查询优化器)** 挺身而出了。
它不指望算出绝对最佳组合，但它笃定向你保证，在仅需区区几十毫秒推演里，利用生物演化的力量，就能给你端上来一道**“未必天下第一，但也绝对堪用之上乘”的退求其次解法**。

---

## 1. 隐喻对账：SQL 乱炖化身旅行商难题 (TSP)

遗传演化的关键，就在于如何把问题重新包装成“染色体”结构变种。
在 GEQO 视角下，多张大表的连接顺序争杀其实就是在重演全球最火爆的 NP 级数学难题——**旅行商问题 (Traveling Salesman Problem, TSP)**。

* **城池** = 待匹配的基础表（Base Relation）。
* **车道 (Tour)** = 表之间连接的先后序列组，例如 $A \to C \to B \to D$。
* **盘缠开销 (Path Cost)** = 顺着这条方案往下跑，底层执行器所需的代价（Total Cost）账单。

在 PG 的源码限定下，GEQO 系统被剥夺了探界繁杂无比的多路 Bushy 树魔阵的野心，只能踏实地在传统的 Left-deep 树（一条道走黑、线性联营序列）这方小天地内推演染色体排谱。

## 2. GEQO 的全局生物演化钟

这套基因游戏全赖 `geqo_main()` 发条手拨转。整个轮盘涵盖下面这五段宛若天神造物的创世纪流程：

### 2.1 鸿蒙初辟：种群大爆发 (Initialization)
造物之初唤起 `random_init_pool()`。
系统直接按掷骰子般的随机序列排列组合法，粗暴地先抓出一把“先头部队”路线放入初始池（Pool）！基表数量 N 越多，这繁衍大池子也会基于特定的公式膨大激增，产生数百只怪胎个体。

### 2.2 阎王判官：生存力测算 (Fitness Evaluation)
残酷的试金石主掌在 `geqo_eval()` 之手。
对池子里的每一个孤单小串串（比如排序集合 $A, B, C, D$），GEQO 自己压根都懒得推敲。它极其讨巧地跑去**白嫖那个 DP 正规军手头的传统算账代价引擎（Cost Engine）！**
它硬逼着别人把这次序捏成完工的 Join Tree，塞给 `cost_qual_eval` 底座让它死命算。算完拿到一个生冷的代价值，倒转印刻为自己在这个大池里能不能存活的资本——**适应度 (Fitness)**。

而后它大手一挥，无情调用 `gimme_pool_size()` 将这一水箱的所有人马依据分数**排高低搞大榜单 (Ranking)**。占据榜单制高点那几个就是战力最猛的，在底端垫底的随时面临斩草除根之命。

### 2.3 天选提拔 (Selection)
基因界里越是彪悍，那必定得狂撒种子。为了不彻底垄断，它用上了坐落于 `geqo_selection.c` 大殿里的 **基于排名的线性择优术 (Linear Ranking Selection)**。
排名极其靠前的（消耗越少的）个体，其作为准双亲被钦点繁衍大业的基率极高；不过即便你是底端垫底的，依旧存留微小的配种率以防止大池基因僵死。

### 2.4 最核弹魔力交叉互融 (Crossover / ERX)
PG 在远古大作 `geqo_erx.c` 选定了一个最高绝阵法：**边缘重组交叉算子 (Edge Recombination Crossover, ERX)**。
这玄学法术能够极其贪心保留了那一代代遗传下来的“连襟死党邻座对”。比如当爹的那里有 `A->C`，当娘的恰好也有 `C->A`！说明这对死基友有着天生优良联系。到了小下一代，子程序就会疯狂向这些游离周边基因靠拢，死死保留住新子代的绝强捆绑落段。完美吻合联接树“相邻结合最好”的物理特征。

### 2.5 为了不死循环的叛逆突变 (Mutation)
要是成百代的人只死盯交配繁衍不加变化，全群早晚会在一个名为 **局域极值耗死 (Local Minima)** 的巨坑里憋死（大家长得一模一样，再也配不出更佳的优形种）。
在此 `geqo_mutation()` 给每个新下的瓜注入疯血乱性小药剂。通过扔极个别的乱骰数，强行将新生一代里的第三和第五位成员硬做一次对调置换叛乱，强行注入野生新息，推拉进化车轮脱离局限烂泥。

## 3. 玄学终局的苦果：因果命数

当历经了 `geqo_effort` 参数控制指令下的足足成百代疯狂更迭发酵之后，主神降临强硬鸣金喊停。淘沙剩下来的最终池子里，傲然屹立爬到全团顶尖排位座次第零位的最彪悍极品，会被如神选般抽将出来。
规划器：“天命难违，弟兄们下海吧，我们就干这票 Join 图形了！”

### 被副作用反噬：让人心脏骤停的善变计划
要是你真的在十来张超巨大核心表里放肆享受了大杂烩拉联操作，你势必迎来一场灵异戏剧：
既然这套遗传玄学打从始至终就带有极大运气伪随机脾性，这不就代表着即便是同样一模一样的惊天 SQL。早上拿出的跑位路线，和下午拿出来的最终底盘执行解析解释图表（Explain Plan），**两者竟然很有可能是长得鸡飞狗跳、完全大相径庭的两棵截然不同狂树！** 查询响应时间会出现长短不一的玄学心跳图。

你要是深受这玄学波动困扰，想要绝对稳定执行期这稳坐钓鱼台的话：
1. 请干脆把该连的 `geqo_seed` (所谓的演变乱斗随机种子主值) 定死为一个你熟悉永远不动的固化常数值去打保底复现重演锁定死。
2. 要是你有着多的大运存时间也不计较前期图纸计算耗时，干脆强杀 `SET geqo = off` 去锁死其运作，也就是放极大化那个 `geqo_threshold` 去活活让老底儿动态计算老黄牛去强推平所有的动态阵列！
