那个叫 Planner 的家伙在想什么？

选择率与代价引擎的数学基座 (Selectivity & Cost Model)

如果说执行器节点 (Executor Nodes) 是在战场上冲锋陷阵的铁血部队，那么 PostgreSQL 核心的 代价基调引擎 (Cost-Based Optimizer, CBO) 就是在这之前于作战统帅部里进行图上千百次沙盘推演的灵魂中枢。 它一切的生杀予夺衡量指北，全部依托于两个底层基石机制：选择率预估 (Selectivity) 猜测系统会返回多少行数据，以及 代价计量 (Cost Model) 折算出这批数据在磁盘和内存中将会耗费多少系统滴答。

在 PG 18.1 中，这套左右引擎命脉的公式算法与核心常数，沉睡于 src/backend/optimizer/path/costsize.c 与 src/backend/utils/adt/selfuncs.c。 本文将穿越表象，带你深入这个堪称统计学与经验法相互融合的魔幻中枢。

1. 统计学透视：选择率 (Selectivity) 的精准测写

所谓 Selectivity (简写为 sel)，是一个介于 0.0 到 1.0 之间的浮点数，它表示绝命的“卡口率”，允许全表百分之多少记录透射过去。 计算 sel 的一切前提，都基建于 ANALYZE 命令收集并写入到 pg_statistic 表中的系统元数据。

1.1 单列特征：MCV 与直方图的双峰分布

现实世界业务库的数据通常符合长尾和聚集极化（如某平台大多数用户年龄 25 左右，夹杂极少的 18 和 50）。 在 selfuncs.c 里处理操作符如 col = 'A' 时：

  • 系统首先去查目标列的 MCV (Most Common Values) 高频词表数组。如果查到 'A' 就在此处，会直接精准地抽出记录好的频率（可能占 32%）作为 sel。
  • 如果是 col > 100 这样的带域检索，MCV 就捉襟见肘了。系统转而启用 Histogram Bounds (直方图切面界限)。PG 将排序好的表数据强行切分为数百个均等记录数的“桶 (Bucket)”。通过线性插值法探测 100 横跨了几个桶，积分计算精确 sel。
  • 最后还要叠加一个 Null Fraction (空值率)，直接扣掉 NULL 的占比作为基础衰减。

1.2 维度的灾难与破局：扩展统计信息 (Extended Statistics)

在代码侧集中于 clauselist_selectivity() 的连乘会带来缺陷盲区。 比如用户写 WHERE city = 'Beijing' AND country = 'China'。传统的概率计算遵循特征独立假设，默认这两个条件不相关。 查表得知：city='Beijing' 的 sel = 0.05，而 country='China' 的 sel = 0.2。 如果引擎采用朴素的 sel_total = 0.05 * 0.2 = 0.01 瞎乘算法，就会得出极其扭曲的结果，以至于被诱导选出孱弱错误计划分支。殊不知北京必定属于中国这二者拥有强绑定聚集概率！

为了挽救这种维度塌缩，PostgreSQL 引入了跨列的 CREATE STATISTICS 模块救场。当你为 (city, country) 建立扩展高维相关性依赖 (Dependencies) 后。系统就会在连乘前判定联动关联，打破“相互独立”的连乘魔咒，挽救底层评估树的坍塌偏斜。

2. 代价模型 (Cost Model) 的双轨评议机制

如果你查看 EXPLAIN 输出，永远会面对类似 cost=10.00..1050.25 两个用下划线相连接的神奇数值。 这代表着优化器世界基于双轨维度的账面计算法：启动代价 (Startup Cost) 与 总代价 (Total Cost)。

2.1 平行维度的角力

由于不同特征压榨，PG 对任何一条候选 Path 都在这两个账本上同时算分。

  • Startup Cost (初始启动沉没代价)：吐出第一行目标数据前，算子需要憋气阻塞筹备花去的时钟值。 比如 SeqScan 启动代价为 0。但 HashJoin 发车极长！你外表得等内表慢吞吞把几百万记录翻完并在内存建好桶池完毕后，才能下发外表探针撞线，因此启动代价巨大。

  • Total Cost (总代价)：完完整整把所有数据发完、关闭管口所需要的总耗。 SeqScan 哪怕只要一条也是全盘扫去碰，总价极高。

双轨博弈在于，如果上级传进来了 LIMIT 1。代价函数在汇总结算最优解时，权柄会疯狂倾向于低 Startup Cost 的算子。即使它的 Total 极难看也会果断采纳，以此规避诸如大排序大哈希表建造的愚蠢行径。

2.2 操控大局的因子 (Magic Numbers)

一切 Cost 都是计算浮点值。放缩的源头，来自 costsize.c 中被写死并允许重置的 GUC 常量参数，这是所有高端 DBA 进行魔改调参的圣地。

  • seq_page_cost = 1.0 (默认)：顺序读取一块 8KB 磁盘耗时被定义为基本黄金单位 1.0。
  • random_page_cost = 4.0 (默认)：标志发生磁头跳步的随机磁盘检索，因寻道通常比顺读大 4 倍以上。这就是为什么明明有索引，系统却嫌弃随机读太分散而果断放弃走全表扫的终极阀门！在全 SSD 时代，高玩通常会将其砍至 1.1 逼迫系统疯狂走查索引。
  • cpu_tuple_cost = 0.01 与 cpu_operator_cost = 0.0025：标志 CPU 脑运算一条 WHERE 等内存操作相比于捞磁盘（耗费 1.0）有多么轻如鸿毛。

3. 代价决断前线：核心算子估算探秘

在知悉参数之后，去 costsize.c 看看具体算法究竟是如何拨账的。

3.1 扫表的代价：cost_seqscan vs cost_indexscan

这是每天被系统狂暴触发千万次的基石结算。

  • 对于全表扫 (cost_seqscan)： 公式直白粗暴：总代价 = (目标表总 Page 数 * seq_page_cost) + (行数 * cpu_tuple_cost) + ...

  • 对于索引扫 (cost_indexscan)： 这里极其反人类。索引树一旦跑起来去拿原据，是非常恐怖离散化的跳跃。 如果表物理存放长期因增删改导致碎片四散（Correlation 相关性 趋近 0），只管强走索引拿十万记录，那就是在掀起十万昂贵物理跳变。random_page_cost 能把算盘直接敲出几千万成本。 但此时代码绝不死板，cost_index 中潜藏着平滑预测公式曲线，去估测那些已存于缓存池内的块并猛力往下折压分数，以免走入纯全栈封禁绝路。

3.2 兵家之要的关联大决斗代价

  • Nested Loop Join (cost_nestloop)： 计算方式无情恐怖：让 左表循环总数据 去硬全额连乘积 右内层单次查代！ 外层十万条，里层只是绝底的极小花费 50。连加也是巨震之价。但这全赖其 Startup 近 0 的绝杀奇点令它能稳抗高压存活。

  • Hash Join (cost_hashjoin)： 深沉点聚焦在于最开端 Startup 构建期。 算法要计算内表有多聚盘、砸多大 CPU cpu_operator_cost 去装填至工作内存 work_mem。 要命的极其分差于此，一旦系统查获了 work_mem 给得小器容不下全漏。算力会在极其平纯原加里生添一只消耗命极高的罚价：disk spill penalty（刷盘巨大惩罚开销）！让多盘狂读拉起极高 Total 总耗。向最上层示警，从而逼向退避并触发它线连接路全保生天。

经过成百上千次精密节点评估积累，PG 优化的王牌引擎得以从数百纠缠网里，斩钉截铁不差分毫断那向绝通往之绝胜黄金大通道长路。
