那个叫 Planner 的家伙在想什么？

选择率与代价引擎的数学基座 (Selectivity & Cost Model)

如果说执行器节点 (Executor Nodes) 是在战场上冲锋陷阵的铁血部队，那么 PostgreSQL 核心的 代价基调引擎 (Cost-Based Optimizer, CBO) 就是在这之前于作战统帅部里进行图上千百次沙盘推演的灵魂中枢。 它一切的评估、一切生杀予夺的最终衡量指北，全部依托于两个底层基石机制：选择率预估 (Selectivity) 猜测系统会返回多少行数据，以及 代价计量 (Cost Model) 折算出这批数据在磁盘和内存中将会耗费多少系统滴答。

在 PG 18.1 中，这套左右引擎命脉的公式算法与核心常数，沉睡于 src/backend/optimizer/path/costsize.c 与 src/backend/utils/adt/selfuncs.c 等文件中。 本文将穿越表象，带领你深入这个堪称统计学与经验法相互融合的魔幻数学枢纽域。

1. 统计学透视：选择率 (Selectivity) 的精准测写

所谓 Selectivity (简写为 sel)，是一个介于 0.0 到 1.0 之间的浮点数，它表示某一个过滤条件 (Qual) 有多大的“卡口率”，能够允许整张表百分之多少的记录透射过去。 计算 sel 的一切前提，都基建于 ANALYZE 命令收集并写入到 pg_statistic 表中的系统元数据。

1.1 单列特征：MCV 与直方图的双峰分布

如果数据都是均匀分布的，那数据库根本不需要复杂的收集机制。但现实世界中，业务库里的数据通常符合长尾和聚集极化（如某平台大多数用户注册年龄是25岁左右，但夹杂了一堆 18 和 50岁的稀缺词点）。 在代码 selfuncs.c 里处理操作符如 col = 'A' 时：

  • 首先系统会查验目标列的 MCV (Most Common Values) 高频词表数组。如果查到 'A' 就在高频数组里，它会直接精准地抽出记录好的频率（可能占 32%）作为 sel。
  • 如果是 col > 100 这样的带域检索，MCV 就捉襟见肘了。这时系统转而启用 Histogram Bounds (直方图切面界限)。PG 将排序好的表数据强行切分为数百个均等记录数的“桶 (Bucket)”。通过线性插值法探测 100 横跨了几个桶，计算其面积积分来得出精确 sel 预估值。
  • 最后还要叠加一个 Null Fraction (空值率)，直接把所有 NULL 的占比扣掉作为基础衰退系数。

1.2 维度的灾难与破局：扩展统计信息 (Extended Statistics)

这是许多老旧数据库最遭人诟病的盲区，在代码侧集中于 clauselist_selectivity() 的连乘缺陷。 如果用户写了 WHERE city = 'Beijing' AND country = 'China'。传统的概率计算遵循特征独立假设定理，也就是默认这两个条件的出现是不相关的。 假设系统查表得知：city='Beijing' 的 sel = 0.05，而 country='China' 的 sel = 0.2。 如果引擎采用朴素的 sel_total = 0.05 * 0.2 = 0.01 的瞎乘算法，就会得出极其扭曲的结果，认为双重匹配的数据极小，以至于被诱导去选出 NestLoop Join 这种极其错误孱弱的计划分支。殊不知，现实中凡是北京的由于地域从属必定全都是中国领地这二者拥有强绑定聚集概率！

为了挽救这种维度塌缩，PostgreSQL 引入了跨列的 CREATE STATISTICS 模块进行救场。当你为 (city, country) 建立扩展高维相关性依赖 (Dependencies) 或多维 MCV 组后。系统就会在多条件连乘之前判定这俩词的联动特征相关系数值；打破“相互独立”的连相乘魔咒，挽救整个底层评估树的坍塌偏斜。

2. 代价模型 (Cost Model) 的双轨评议机制

如果你查看过 EXPLAIN 输出的括号第一手信息，永远会面对类似 cost=10.00..1050.25 这两个用下划线相连接的神奇数值。 这代表着优化器世界里一套极其严苛细分的基于双轨维度的总账计算法：启动代价 (Startup Cost) 与 总代价 (Total Cost)。

2.1 平行维度的角力

由于数据库被不同特征的长短命连接或者上层指令要求压榨，PG 对任何一条候选 Path 都在这两个账本上同时算分。

  • Startup Cost (初始启动沉没代价)：吐出前列第一行 (The First Tuple) 目标匹配数据前，当前算子需要憋气阻塞筹备花去的时钟值。 比如 SeqScan 它的启动代价为 0 （随时随地硬盘打来就开始往外流）。 而如果用 HashJoin，它的发车时间极长！你外表得等我内表慢吞吞把全量记录翻完并在内存建好几百万 Hash 数组的桶池完毕后，我才能开放外表探针去撞线吐出第一条结果。因此它启动代价巨大。

  • Total Cost (穷尽扫尾总代价)：完完整整把包含这符合条件的最后一万行数据全发送完毕、关闭管口所需要的总和分。 比如 SeqScan 这里就非常要命了，哪怕你只要其中一条也是要扫描全盘去碰的，总价奇高。

这双轨的博弈在于，如果上级传进来了诸如 LIMIT 1。代价函数在汇总结算最优解时，权柄会疯狂倾向于带有极低 Startup Cost 的算子。即使它的 Total 极其难看，系统也会果断采纳，以此规避诸如大排序大哈希表建造的愚蠢行径。

2.2 操控大局的魔力数字因子 (Magic Numbers)

一切 Cost 都是虚构的浮点打分单位。而这一切数字放缩的源头，来自 costsize.c 中被写死的并允许重置的 GUC 常量参数，这是所有高端 DBA 进行参数魔改调参的圣碑之地。

  • seq_page_cost = 1.0 (默认)：顺序读取一块 8KB 磁盘页的参考耗时被定义为基本单位 1.0 的黄金标尺。
  • random_page_cost = 4.0 (默认)：标志着发生磁头跳步的随机磁盘检索，因其寻道时间通常被认定要比顺读惩罚大 4 倍以上。这就是为什么明明有索引，然而系统却嫌弃随机读太分散而果断放弃走索引、转投入全表顺序扫面的终极解释阀门！在全 SSD 高速时代，高玩通常会将其砍杀降至 1.1 来逼迫系统疯狂走查寻索引分支。
  • cpu_tuple_cost = 0.01 与 cpu_operator_cost = 0.0025：标志了主板上 CPU 大脑去计算一条 WHERE a=1 这样内存处理操作相比读取磁盘（耗费 1.0）到底有多么的“轻如鸿毛”。

3. 代价决断前线：核心算子估算探秘

在知悉了底层宏观参数之后，让我们潜入 costsize.c 去看看具体算法节点是到底怎么花起这笔账面的。

3.1 扫表的代价：cost_seqscan vs cost_indexscan

这是整个系统内每天被调用最多数以千万次的基础算盘。

  • 对于全表扫 (cost_seqscan)： 公式直白粗暴，毫无虚空魔法： 总代价 = (目标表格所占的总 Page 数目 * seq_page_cost) + (表格行数 * cpu_tuple_cost) + ...

  • 对于索引扫 (cost_indexscan)： 这里极其反人类且充满了经验曲线法则。因为索引树一旦跑起来，你去找硬盘上的数据通常是非常恐怖离散化的。 如果你的主表物理存放与这颗 B 树由于长期增删改查导致碎片化四散（即 Correlation 特征聚簇相关性 趋近于 0），如果强走索引拿十万条记录，你可能要发起非常昂贵的十万次跨磁头大随机跳跃，random_page_cost 能把计分算盘直接干穿爆炸报废去到几千万成本。 但此时代码也绝不死板，cost_index 中潜藏了一套名为 Mackert-Lohman 的平滑预估公式曲线，去猜测这堆离散请求究竟会命中有多少页可能由于已经在 LRU shared_buffers (内存共享缓存池) 里面而省去了回源物理盘，从而打个猛烈的折扣把成本压制下来防止系统完全不走它。

3.2 兵家大关大争之地：关联的代价估量

  • Nested Loop Join (cost_nestloop)： 被称为无脑穷魔的梦魇。其计算方式为：让 左表外循环数据总规模 (outer_rows) 全额连乘 右表被当作单条被压榨的内循环单趟代价 (inner_cost)！ 哪怕最外层只读 1 万行，你的内部索引扫单次只是极小成本 50。在指数级复利连乘叠加下也是非常刺目的分量。但这恰由于其具备 Startup 为接近 0 的绝杀特性以及几乎没占用额外独立运存空间的优点而让它常立于高并发不败之地。

  • Hash Join (cost_hashjoin)： 其计算的深沉点全部聚焦在一开始的 Startup_Cost 构建蓄水阶段。 系统会精密计算内表 inner_rows 的总体积有多大，需要使用多大的 CPU 指令资源 cpu_operator_cost 去遍历、计算每一个数据单元的哈希根并放入工作内存 work_mem 内预装配。 最要命的分叉评估发生在这，一旦优化器察觉到给拨的 work_mem 太抠门小气存不下全部内表哈希链列。代价算口会在原本仅仅只吃 CPU 的计算里，凭空掺入一大笔极其恶心和高居不下的 disk spill penalty（刷盘大惩罚机制量） 开销，并且成倍地因为需要在探针碰撞期间于磁盘多批次重置读取而拉高 Total Cost。从而向顶层告警，规避或强制触发要求走别派系统连接路线以保平安顺遂。

经过如此成百上千次精密到苛刻毫厘偏差的小分支节点代价评估积累并合卷对账对比。PG 代价优化的王牌引擎得以从最少几百上千条乃至天文数字交错般混乱的查询可能路径网里，在短短不足几毫秒之内，犹如雷达盲区索敌指引核爆一般，斩钉截铁不差分毫地敲定并拿出了那条绝对通向最优解的绝杀黄金制胜长路。
